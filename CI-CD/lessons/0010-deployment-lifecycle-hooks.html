Deployment Lifecycle Hooks

How deploy, postRouteTraffic, and on.failure turn a deployment into a pipeline

Beyond just "run the deploy"

In Lesson 6 you saw a basic deployment job with a single deploy: section. The Applications.Apply pipeline goes further: after the Octopus deploy runs, it automatically runs Playwright smoke tests against the live environment — and if anything fails, it fires a Slack alert. All of this is wired through the three lifecycle hooks of strategy: runOnce.

The three hooks, in order

deploy: always runs

The main deployment steps. In Apply: sets the Octopus channel, runs deploy-release-steps.yml. Uses checkout: none — source not needed, Octopus takes care of the actual file deployment.

postRouteTraffic: only if deploy: succeeded

Runs after traffic is routed to the new version. In Apply: runs Playwright automated tests against the live Dev or QA URL. Uses checkout: self because automated-tests.yml needs the test project source to build and run.

on.failure: only if job failed

Runs when any earlier hook failed (deploy or postRouteTraffic). In Apply: sends a Slack block-kit message to #dit-apply with a link to the failing build. Uses checkout: none.

The exact YAML from Applications.Apply

# From azure-pipelines.yml — Development stage

- deployment: DeployOctopusRelease
  displayName: 'Deploy on Dev'
  environment: Development
  strategy:
    runOnce:

      deploy:                                         # ← hook 1: the actual deploy
        steps:
          - checkout: none

          - powershell: |
              $branch = "$(Build.SourceBranch)"
              $channel = "Development"
              if ($branch.StartsWith("refs/heads/feature/")) {
                  $channel = "Testing-Feature"
              }
              Write-Host "##vso[task.setvariable variable=channel]$channel"
            displayName: 'Set Octopus channel'

          - template: CD/octopus/deploy-release-steps.yml@templates
            parameters:
              octopusProjectName: '$(ditManageIISProjectName)'
              octopusProjectChannel: '$(channel)'
              octopusProjectEnvironment: 'Development'

      postRouteTraffic:                               # ← hook 2: smoke tests
        steps:
          - checkout: self                            # needs source for test project!

          - template: automated-tests.yml
            parameters:
              baseUrl: '$(baseUrlAcmeDev)'
              deviceName: '$(deviceName)'
              browserName: '$(browserName)'
              testRunTitle: 'Automated tests - Dev'

      on:
        failure:                                      # ← hook 3: alert on failure
          steps:
            - checkout: none

            - template: slack-failed-deployment-notification.yml
              parameters:
                notificationWebHook: '$(notificationWebHook)'
                notificationChannel: '$(notificationChannel)'
                environment: 'Dev'

Why checkout: self in postRouteTraffic? The automated-tests.yml template needs to build and run the Equisoft.Apply.AutomationTests .NET project. That project lives in the source repo. Without checkout: self, the source directory is empty and dotnet build fails. Every other hook uses checkout: none because Octopus and Slack just need variables, not files.

Hook reference

HookWhen it runsApply uses it forCheckout
deploy: Always — the main hook Octopus Deploy release checkout: none
postRouteTraffic: Only if deploy: succeeded Playwright smoke tests against live URL checkout: self (needs test source)
on.failure: Only if the job failed (either hook) Slack alert with build link checkout: none
on.success: Only if all hooks succeeded (not used in Apply) —

There is also preDeploy: which runs before deploy: — useful for pre-flight checks (verify the environment is available, warm up a cache). The Apply pipeline doesn't use it, but it exists. Full hook order: preDeploy → deploy → routeTraffic → postRouteTraffic → on.success / on.failure.

What counts as a failure for on.failure:

on.failure: triggers when either the deploy: hook or the postRouteTraffic: hook fails. This means in the Apply pipeline:

  • Octopus deploy failed → Slack fires
  • Octopus deploy succeeded but Playwright tests failed → Slack also fires

The Slack message links to the pipeline run, so the team can click through to see which step failed — the deploy itself or the post-deploy smoke tests.

# The Slack notification template builds this payload:
# - Header: "⚠ Failed deployment on Dev"
# - Link: $(System.TeamFoundationCollectionUri)$(System.TeamProject)/_build/results?buildId=$(Build.BuildId)
# - Branch: $(Build.SourceBranch)
# - Commit: $(Build.SourceVersion)

The same template, eight ways at once

The automated-tests.yml template referenced in postRouteTraffic is also used in the nightly test pipeline (azure-pipelines-tests.yml), which runs it across 8 device/browser combinations in parallel using a matrix strategy:

# azure-pipelines-tests.yml (simplified)
strategy:
  matrix:
    Desktop_Chrome:
      deviceName: "Desktop Chrome"
      browserName: "chromium"
    iPhone_15_Pro_Max:
      deviceName: "iPhone 15 Pro Max"
      browserName: "webkit"
    Desktop_Firefox:
      deviceName: "Desktop Firefox"
      browserName: "firefox"
    # ... 5 more device/browser combos
  maxParallel: 8

One template, two usages: the deployment pipeline runs it once (smoke test with the team's default browser) and the nightly pipeline fans it out across all 8 browsers. This is the value of a well-designed reusable template.
