Azure Pipelines YAML Anatomy

The four-level hierarchy every pipeline is built from — and how to read it at a glance.

Why this lesson

Every Azure Pipeline — no matter how complex — is just a nested structure of four things. Once that hierarchy is automatic for you, any YAML file someone drops in your lap becomes readable in seconds. This is the mental model everything else hangs off.

The four-level hierarchy

Pipeline
One YAML file = one pipeline. Controls triggers and top-level variables.
↳
Stage
A named phase (e.g. Build, Test, Deploy). Stages run sequentially by default. You can add approvals between them.
↳
Job
A group of steps that runs on one agent. Jobs within a stage run in parallel by default. Each job gets a fresh agent.
↳
Step
A single task or script. Steps run sequentially within a job. This is where actual work happens.
Key pattern
Stages → sequential  ·  Jobs → parallel  ·  Steps → sequential. Parallel = fast. Sequential = safe ordering. Knowing which level you're at tells you the default behaviour.

Triggers — what starts the pipeline

Triggers live at the top of the pipeline file. The most common ones:

  • trigger — CI trigger: runs on push to specified branches
  • pr — PR trigger: runs when a pull request targets specified branches
  • schedules — cron-style scheduled runs

Setting trigger: none disables the CI trigger — useful for pipelines you only want to run manually or on PR.

A real pipeline, annotated

Here is a minimal Node.js build-and-test pipeline. Every line maps to the hierarchy above. 

# ① Trigger
trigger:
  branches:
    include:
      - main
      - develop

# ② Pipeline-level variables
variables:
  nodeVersion: '20.x'

# ③ Stage: Build
stages:
  - stage: Build
    displayName: 'Build & Test'

    # ④ Job
    jobs:
      - job: BuildJob

        # ⑤ Agent pool
        pool:
          vmImage: 'ubuntu-latest'

        # ⑥ Steps
        steps:
          - task: UseNode@1
            inputs:
              version: '$(nodeVersion)'

          - script: npm ci
            displayName: 'Install dependencies'

          - script: npm test
            displayName: 'Run tests'
  • ① trigger — this pipeline runs automatically on pushes to main or develop.
  • ② variables at pipeline scope — reusable across all stages. Referenced with $(variableName).
  • ③ stages list — just one here ("Build"). The displayName is what you see in the Azure DevOps UI.
  • ④ job — one unit of work on one agent. Can have many jobs per stage; they'd run in parallel.
  • ⑤ pool.vmImage — which Microsoft-hosted agent to use. ubuntu-latest is the standard Linux agent for Node.
  • ⑥ steps — the actual work. First a task (UseNode@1 selects the Node version), then two scripts.

Tasks vs scripts

Steps come in two flavours:

  • script: — a raw shell command (bash on Linux/Mac, cmd on Windows). Use this for anything you'd type in a terminal: npm ci, npm run build.
  • task: — a named, versioned, prebuilt action (like a GitHub Action). Tasks handle things scripts can't easily do: publishing artifacts, deploying to Azure, setting up a Node version. Find them in the task reference.
Common mistake
Task names include a version number: UseNode@1, AzureWebApp@1. Using a task without @version is an error. Always pin the version.

The shorthand: skipping stages

If your pipeline only has one stage, you can skip the stages: key entirely and just write jobs: at the top level. Azure injects an implicit single stage. You'll see both forms in the wild — they're equivalent for simple pipelines. 

# Shorthand (no explicit stages:)
trigger: [main]

pool:
  vmImage: 'ubuntu-latest'

steps:
  - script: npm ci && npm test

This works fine. But as soon as you need a Deploy stage separate from Build, you need explicit stages. Start simple; add structure when the pipeline actually needs it.
