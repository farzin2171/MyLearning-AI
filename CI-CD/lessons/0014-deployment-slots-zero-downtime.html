Deployment Slots: Zero-Downtime Deploys

Extend your Lesson 13 pipeline with the deploy → smoke test → swap pattern. Your users see the new version only after it has warmed up and passed a health check — no 502s, no cold starts in production.

The problem with a direct deploy

When AzureWebApp@1 deploys directly to the production slot, it does this:

  1. Stop the running app
  2. Copy the new files over
  3. Restart the app

During steps 1–3, any incoming request gets a 502 Bad Gateway. On .NET 8 with a SQL Server warm-up, that window can be 15–30 seconds. On a busy service, that is real user-facing downtime.

Deployment slots solve this by warming the new version up before routing traffic to it.

What a deployment slot is

A slot is a second live instance of your App Service running in the same App Service Plan. Azure gives you one free slot on Standard tier and above — conventionally named staging.

production
my-service-prod.azurewebsites.net
v1.2.0 — live
⇄
staging
my-service-prod-staging.azurewebsites.net
v1.3.0 — warming up

You deploy v1.3.0 to staging. It starts up, runs migrations, warms its caches, passes your health check. Then Azure executes a slot swap — it atomically redirects the production URL to the new instance. v1.3.0 is now live. The old v1.2.0 instance moves to staging, where it can serve as an instant rollback.

Rollback in seconds: The previous version sits in the staging slot. If you discover a bug in production after the swap, you swap back — no new build required.

The three-step pattern

1

Deploy to staging slot

New version lands in staging, not production. Users are unaffected.

2

Smoke test staging

Hit the staging URL's health endpoint. Fail the pipeline if HTTP is not 200.

3

Swap staging → production

Azure atomically routes production traffic to the warmed-up new version.

Updating your pipeline (the diff from Lesson 13)

Add one variable to your variable group, then replace the DeployProd job steps. Everything else in your Lesson 13 pipeline stays the same.

Variable group addition

In Pipelines → Library → my-service-vars, add: 

resourceGroupName   = my-resource-group    ← the RG containing your App Service
Why? The AzureAppServiceManage@0 swap task requires the resource group name — the AzureWebApp@1 task does not, which is why you didn't need it in Lesson 13.

Replace the DeployProd job steps

Find the DeployProd stage in your azure-pipelines.yml and replace the steps block under deploy:: 

      - deployment: DeployToProd
        displayName: 'Azure App Service: Prod'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'prod'
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none

                # ── Step 1: Deploy to STAGING slot (not production) ──────
                - task: AzureWebApp@1
                  displayName: '① Deploy to staging slot'
                  inputs:
                    azureSubscription: '$(azureServiceConnection)'
                    appType: 'webAppLinux'
                    appName: '$(appServiceNameProd)'
                    deployToSlotOrASE: true              # ← enables slot targeting
                    resourceGroupName: '$(resourceGroupName)'
                    slotName: 'staging'
                    package: '$(Pipeline.Workspace)/drop/**/*.zip'
                    appSettings: |
                      -ASPNETCORE_ENVIRONMENT Production
                      -ASPNETCORE_URLS "http://+:80"

                # ── Step 2: Smoke test the staging slot ──────────────────
                - script: |
                    echo "Waiting 10s for app to warm up..."
                    sleep 10
                    url="https://$(appServiceNameProd)-staging.azurewebsites.net/hc"
                    echo "Testing: $url"
                    status=$(curl -s -o /dev/null -w "%{http_code}" "$url")
                    if [ "$status" != "200" ]; then
                      echo "##[error]Smoke test FAILED — HTTP $status from staging slot"
                      exit 1
                    fi
                    echo "Smoke test passed (HTTP $status) — safe to swap"
                  displayName: '② Smoke test staging slot'

                # ── Step 3: Swap staging → production ────────────────────
                - task: AzureAppServiceManage@0
                  displayName: '③ Swap staging → production'
                  inputs:
                    azureSubscription: '$(azureServiceConnection)'
                    Action: 'Swap Slots'
                    WebAppName: '$(appServiceNameProd)'
                    ResourceGroupName: '$(resourceGroupName)'
                    SourceSlot: 'staging'
                    # TargetSlot defaults to 'production' when omitted
Create the staging slot first. In the Azure portal: App Service → Deployment slots → + Add Slot → name it staging, clone settings from production. The pipeline will fail with "slot not found" if it does not exist yet.

Sticky settings — the one gotcha everyone hits

By default, all app settings travel with the swap. If you set ASPNETCORE_ENVIRONMENT=Production in staging to warm it up, after the swap that setting is now in production — which is fine, that's what you want.

But some settings should not travel. They should stay locked to their slot. These are called sticky (or "slot setting") settings.

Setting Should be sticky? Why
ASPNETCORE_ENVIRONMENT No — travel with swap Staging should warm up with Production settings, then swap those settings too
Production DB connection string Yes — sticky to prod slot You do not want staging accidentally pointing at your live database
Staging DB connection string Yes — sticky to staging slot After the swap, the "old prod" instance that lands in staging should use the staging DB
Feature flags / API keys Depends on intent If environment-specific: sticky. If they travel with the release: non-sticky.

Mark a setting as sticky in the Azure portal: App Service → Configuration → Application settings → tick "Deployment slot setting" next to the key. You can also set sticky settings from the pipeline using the appSettings block in a separate AzureAppServiceSettings@1 task (advanced use case).

Manual rollback in an emergency

After a swap, v1.2.0 (the previous version) is sitting in the staging slot. To roll back immediately, swap again — either from the portal or from a one-off pipeline run with: 

# Emergency rollback: swap staging (old prod) back to production
- task: AzureAppServiceManage@0
  inputs:
    azureSubscription: '$(azureServiceConnection)'
    Action: 'Swap Slots'
    WebAppName: '$(appServiceNameProd)'
    ResourceGroupName: '$(resourceGroupName)'
    SourceSlot: 'staging'
Rollback time: ~15 seconds — the same as the original swap. No new build, no re-deploy. This is the primary reason the pattern exists.

Cost implications

A staging slot runs on the same App Service Plan as production. You pay for the compute already (the plan, not the instances). Slots are free on Standard tier and above (S1 / B1 do not support slots). The number of slots depends on tier:

  • Standard: 5 slots
  • Premium: 20 slots
If your App Service is on the Free or Shared tier, slots are not available. You will need to upgrade before this pattern works.
