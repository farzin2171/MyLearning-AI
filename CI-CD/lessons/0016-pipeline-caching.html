Pipeline Caching: Faster Builds

Every time your pipeline runs, it re-downloads npm or NuGet packages from the internet. Caching eliminates that. A 3-minute build becomes a 45-second build with one task.

Why builds are slow without caching

Azure Pipelines runs every job on a fresh agent VM. There is no package cache from previous runs. On every build, your pipeline:

Without cache

  • Downloads all packages from the internet
  • npm ci on a medium project: 60–120 s
  • dotnet restore on a large solution: 40–90 s
  • Slow on every run, even with no dependency changes

With Cache@2

  • Restores packages from Azure Pipelines cache storage
  • Cache hit: npm packages restored in 5–15 s
  • Cache hit: NuGet packages restored in 3–10 s
  • Cache miss only when lock file changes

How Cache@2 works

Step 1 — Azure computes a cache key

You define a key expression. Azure hashes the values you provide (typically including the content of your lock file). The result is a fingerprint for that exact set of dependencies.

Step 2 — Cache hit or miss

If a cache entry with that fingerprint exists from a previous run, Azure restores the path directory from it. The variable CACHE_RESTORED is set to true. If there is no match, the path starts empty — a cache miss.

Step 3 — Post-run save

After the job completes, Azure automatically saves the path directory back to the cache (only on a miss). You do not write a "save" step — it happens automatically at the end of the job.

Anatomy of a cache key

key: 'npm | "$(Agent.OS)" | package-lock.json'
npm — human label (arbitrary string) $(Agent.OS) — Linux vs Windows caches are separate package-lock.json — its content is hashed; when it changes, cache is busted
The lock file is the cache key signal. When you add or upgrade a package, the lock file changes, the hash changes, the key changes — you get a cache miss, fresh packages download, and the new cache is saved. No stale dependencies.

Implementation

Add Cache@2 before npm ci. The npm cache directory is controlled by the npm_config_cache environment variable. 

variables:
  npm_config_cache: $(Pipeline.Workspace)/.npm   # ← store cache here

steps:
  - checkout: self
    fetchDepth: 0

  - task: NodeTool@0
    displayName: 'Install Node.js 20.x'
    inputs:
      versionSpec: '20.x'

  - task: Cache@2
    displayName: 'Cache npm packages'
    inputs:
      key: 'npm | "$(Agent.OS)" | package-lock.json'
      restoreKeys: |
        npm | "$(Agent.OS)"
      path: $(npm_config_cache)

  - task: Npm@1
    displayName: 'npm ci'
    inputs:
      command: 'ci'
      # npm ci reads from the cache path automatically when npm_config_cache is set
Why npm ci and not npm install? npm ci requires a lock file and installs exactly what it contains — no version range resolution, no lock file mutations. It is faster and deterministic. Use npm ci in all CI pipelines.
restoreKeys is a fallback. If the exact key misses (e.g. a new package was added), Azure tries the broader npm | "$(Agent.OS)" key as a partial match. You get a stale-but-partial cache, then npm fills in the delta. Better than a full re-download. Always include a restoreKeys block.

Add Cache@2 before dotnet restore. Tell the .NET SDK where to store packages by setting NUGET_PACKAGES. 

variables:
  NUGET_PACKAGES: $(Pipeline.Workspace)/.nuget/packages   # ← store cache here

steps:
  - checkout: self
    fetchDepth: 0

  - task: UseDotNet@2
    displayName: 'Install .NET 8.0.x'
    inputs:
      version: '8.0.x'

  - task: Cache@2
    displayName: 'Cache NuGet packages'
    inputs:
      key: 'nuget | "$(Agent.OS)" | **/packages.lock.json,!**/bin/**,!**/obj/**'
      restoreKeys: |
        nuget | "$(Agent.OS)"
      path: $(NUGET_PACKAGES)

  - task: DotNetCoreCLI@2
    displayName: 'Restore'
    inputs:
      command: 'restore'
      projects: '**/*.csproj'
      # dotnet restore reads NUGET_PACKAGES automatically
NuGet lock files are opt-in. The glob **/packages.lock.json only works if your projects have package lock files enabled. Add this to your .csproj: 
<PropertyGroup>
  <RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>
</PropertyGroup>
Without this, fall back to hashing **/*.csproj files instead: key: 'nuget | "$(Agent.OS)" | **/*.csproj'

Skipping install on a cache hit

If you want to skip npm ci entirely on a cache hit (not just speed it up), use the CACHE_RESTORED variable that Azure sets after the Cache task runs: 

- task: Cache@2
  displayName: 'Cache npm packages'
  inputs:
    key: 'npm | "$(Agent.OS)" | package-lock.json'
    path: $(npm_config_cache)
  # After this step: CACHE_RESTORED is 'true' or 'false'

- task: Npm@1
  displayName: 'npm ci'
  inputs:
    command: 'ci'
  condition: ne(variables.CACHE_RESTORED, 'true')   # skip if cache hit
When to use this: If your path includes node_modules directly (not just the npm cache), you can skip install completely. But if you're caching the npm download cache and restoring into node_modules separately, you still need npm ci to unpack. Most teams cache the npm cache directory and let npm ci run (it's fast when the cache is warm).

Placement in your Lesson 13 pipeline

Insert the Cache task in the Build stage, after checkout and Node/SDK install, but before the package restore/install step. 

steps:
  - checkout: self
    fetchDepth: 0

  - task: gitversion/setup@3.2.1
    inputs:
      versionSpec: '5.x'
  - task: gitversion/execute@3.2.1
    name: GitVersion

  - task: NodeTool@0
    inputs:
      versionSpec: '20.x'

  ← INSERT Cache@2 HERE ←

  - task: Npm@1
    displayName: 'npm ci'
    inputs:
      command: 'ci'

  - task: Npm@1
    displayName: 'npm build'
    inputs:
      command: 'custom'
      customCommand: 'run build'
The cache task does not change the rest of the pipeline. The npm ci step still runs — it just finds all packages already in the cache directory and skips downloading them.
