Lesson 6 of Azure CI/CD with Azure Pipelines

Environments and Approvals

Gating deployments with required reviewers and branch controls

What an Environment actually is

An Environment in Azure Pipelines is not a compute resource — it's a named deployment target that Azure DevOps tracks and lets you gate. It does three things:

  1. Tracks history — records which pipeline run deployed what version, when, and who approved it
  2. Enforces checks — can pause the pipeline until a human approves, a branch rule passes, or a timer clears
  3. Provides an audit trail — the Environments view in Azure DevOps shows every deployment across all pipelines

In your team's pipelines, environments are referenced by name inside a deployment: job. The actual deploy logic (Octopus Create Release, etc.) runs after all checks pass.

Build stage runs, artifact published
Approval check pipeline pauses — waiting for reviewer
Reviewer approves ✓ deployment job now runs
Deploy stage Octopus Create Release executes

Referencing an environment in YAML

A deployment: job takes an environment: key. The value is the environment name as defined in Azure DevOps. If it doesn't exist yet, Azure Pipelines creates it on first run.

stages:
  - stage: Deploy
    dependsOn: Build
    condition: and(succeeded('Build'), eq(variables.isMaster, true))
    jobs:
      - deployment: DeployToProduction
        displayName: Deploy to Production
        pool:
          name: $(CILinuxAgentPool)
        environment: Production             # ← the environment name
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none
                - download: none
                - task: OctopusCreateRelease@3
                  inputs:
                    OctopusServerEndpoint: $(OctopusEndpoint)
                    Space: Default
                    Project: my-app
                    ReleaseNumber: $(GitVersion.SemVer)
                    Channel: Release

The environment: key is the gate trigger. Before the deploy: steps run, Azure Pipelines evaluates every check configured on the Production environment. If a required reviewer hasn't approved, the pipeline pauses here — not before the stage, but specifically before this job's steps execute.

You can also use the object form to specify a resource within an environment (relevant for Kubernetes / VMs). For Octopus-based pipelines, the string form is all you need:

environment: Production            # string form — sufficient for your team's pipelines
# environment:                     # object form — only needed for Kubernetes/VM targets
#   name: Production
#   resourceName: web-server-01

Configuring approval checks — the UI

Checks are not configured in YAML. They live in the Azure DevOps UI under:

Azure DevOps → Pipelines → Environments → [Environment name] → ⋯ → Approvals and checks

The four checks your team is most likely to use:

Required reviewers

Named users or groups must approve before the deployment job starts. Can require one reviewer or all of them.

Branch control

Only pipelines triggered from listed branches can use this environment. Blocks feature/* branches from reaching Production.

Exclusive lock

Only one pipeline run may deploy to this environment at a time. Subsequent runs queue and wait.

Business hours

Deployments only proceed during a configured time window. Useful for restricting production changes to working hours.

Branch control is critical for Production. Without it, anyone who can trigger a pipeline run — even from a feature/ branch — could potentially reach the Production environment (if the YAML conditions pass). The branch control check adds a second layer: even if the YAML condition evaluates to true, the check fails if the run's source branch isn't on the allowlist.

Exclusive lock — preventing concurrent deploys

When two pipeline runs reach the same environment simultaneously, you get a race condition: two Octopus releases deploying to the same slot at once. The Exclusive Lock check prevents this.

# What happens with Exclusive Lock enabled on 'Staging':
#
# Run A starts → acquires lock → deploy proceeds
# Run B starts → tries to acquire lock → waits
# Run A finishes → releases lock
# Run B → acquires lock → deploy proceeds
#
# The YAML doesn't change — the lock is enforced by the check in the UI.

You can configure whether queued runs wait or are cancelled when a newer run is triggered. "Cancel older" is common for non-production environments — the latest commit is what matters.

How this maps to your team's Octopus channels

Your team maps branches to Octopus channels, and pipeline stages use runtime conditions to decide which stage runs. Environments map one-to-one to those channels:

Branch Pipeline stage condition Azure Environment Octopus channel
develop isDevelop = true Development Development
release/*, hotfix/* isRelease = true Testing Testing
master, tags isMaster = true Production Release

In practice, the YAML stage condition already ensures only the right branch runs the deploy stage. The environment check is a second layer: even if someone modifies the condition, the branch control check on the environment stops non-production branches from reaching Production.

# A complete multi-channel deploy section matching your team's style

  - stage: DeployDevelopment
    dependsOn: Build
    condition: and(succeeded('Build'), eq(variables.isDevelop, true))
    jobs:
      - deployment: ToDevelopment
        environment: Development           # ← no approvals, fast feedback
        pool:
          name: $(CILinuxAgentPool)
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none
                - download: none
                - task: OctopusCreateRelease@3
                  inputs:
                    Channel: Development
                    ReleaseNumber: $(GitVersion.SemVer)

  - stage: DeployTesting
    dependsOn: Build
    condition: and(succeeded('Build'), eq(variables.isRelease, true))
    jobs:
      - deployment: ToTesting
        environment: Testing               # ← light approval (team lead)
        pool:
          name: $(CILinuxAgentPool)
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none
                - download: none
                - task: OctopusCreateRelease@3
                  inputs:
                    Channel: Testing
                    ReleaseNumber: $(GitVersion.SemVer)

  - stage: DeployProduction
    dependsOn: Build
    condition: and(succeeded('Build'), eq(variables.isMaster, true))
    jobs:
      - deployment: ToProduction
        environment: Production            # ← required reviewer + branch control
        pool:
          name: $(CILinuxAgentPool)
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none
                - download: none
                - task: OctopusCreateRelease@3
                  inputs:
                    Channel: Release
                    ReleaseNumber: $(GitVersion.SemVer)

Deployment history

Every time a deployment job completes, Azure DevOps records it against the environment. Navigate to Pipelines → Environments → Production to see:

  • Which pipeline run deployed, and when
  • Who triggered it and who approved it
  • The commit SHA and branch the run came from
  • Whether the deployment succeeded or failed

This is your team's production audit trail — useful for compliance, post-incident reviews, and answering "what's currently deployed?"

Tip: Even if you don't add any checks to an environment, the deployment history alone is worth referencing a named environment. It gives you a free, automatic deployment log without any extra work.
