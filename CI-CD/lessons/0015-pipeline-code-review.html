Pipeline Code Review: Find the Bug

Five real pipeline snippets, each with one misconfiguration. Study each one, work out what's wrong before you click Reveal. This is retrieval practice — the discomfort of not knowing is what makes the knowledge stick.

How to use this lesson: Read each pipeline snippet carefully. Try to name the bug out loud (or write it down) before you reveal the answer. After each reveal, rate yourself: Got it or Missed it. Your score is shown at the end.
1

The version calculation that sometimes fails in CI

Lesson 7

This pipeline works fine for the developer who cloned the repo last month — but a new team member set up a fresh clone today and the pipeline fails with "GitVersion.exe" exited with code 1: Unable to determine the commit count. The YAML is identical on both machines. What is the misconfiguration? 

- stage: Build
  jobs:
    - job: BuildJob
      pool:
        vmImage: 'ubuntu-latest'
      steps:
        - checkout: self

        - task: gitversion/setup@3.2.1
          inputs:
            versionSpec: '5.x'

        - task: gitversion/execute@3.2.1
          name: GitVersion

        - task: DotNetCoreCLI@2
          displayName: 'Build'
          inputs:
            command: 'build'
            arguments: '/p:Version=$(GitVersion.SemVer)'

Bug: checkout: self defaults to a shallow clone (fetchDepth: 1)

GitVersion reads the entire commit history to calculate a correct semantic version. A shallow clone gives it only one commit, so it cannot count commits since the last tag. On a fresh CI agent the history is always shallow unless you opt out.

Fix
- checkout: self
  fetchDepth: 0    ← add this line

fetchDepth: 0 fetches the full history. This is required on every repo that uses GitVersion. Without it, the error is intermittent — it depends on how many commits have been made since cloning.

2

The deploy that always errors "No package found at the specified location"

Lessons 5 + 13

The Build stage publishes a drop artifact and succeeds. The Deploy stage starts and immediately fails: "No package found at the specified location". The path looks reasonable to you. What is wrong? 

- stage: Build
  jobs:
    - job: BuildJob
      steps:
        - task: DotNetCoreCLI@2
          inputs:
            command: 'publish'
            zipAfterPublish: true
            arguments: '--output $(Build.ArtifactStagingDirectory)'
        - task: PublishPipelineArtifact@1
          inputs:
            targetPath: '$(Build.ArtifactStagingDirectory)'
            artifact: 'drop'

- stage: Deploy
  dependsOn: Build
  jobs:
    - deployment: DeployToDev
      environment: 'dev'
      strategy:
        runOnce:
          deploy:
            steps:
              - checkout: none
              - task: AzureWebApp@1
                inputs:
                  azureSubscription: '$(azureServiceConnection)'
                  appName: '$(appServiceNameDev)'
                  package: '$(Build.ArtifactStagingDirectory)/**/*.zip'

Bug: $(Build.ArtifactStagingDirectory) is the build agent's staging folder — a different machine

Build.ArtifactStagingDirectory is a local path on the build job's agent. The deployment job runs on a different agent. That folder does not exist there. Deployment jobs automatically download published artifacts into $(Pipeline.Workspace)/<artifact-name>/.

Fix
package: '$(Pipeline.Workspace)/drop/**/*.zip'

Pipeline.Workspace is the cross-agent workspace root that Azure Pipelines sets up on every agent in the pipeline run. Published artifacts land here, not in Build.ArtifactStagingDirectory.

3

The approval gate that was configured but never fires

Lesson 6

A team configured a manual approval on the prod environment in Azure DevOps (Pipelines → Environments → prod → Approvals). But every pipeline run deploys to production immediately — no one is ever asked to approve. Can you see why from the YAML? 

- stage: DeployProd
  displayName: 'Deploy → Prod'
  dependsOn: DeployDev
  condition: and(succeeded(), eq(variables.isMain, true))
  jobs:
    - job: DeployToProd
      pool:
        vmImage: 'ubuntu-latest'
      steps:
        - checkout: none
        - task: AzureWebApp@1
          inputs:
            azureSubscription: '$(azureServiceConnection)'
            appType: 'webAppLinux'
            appName: '$(appServiceNameProd)'
            package: '$(Pipeline.Workspace)/drop/**/*.zip'

Bug: job: instead of deployment: — environment checks are never consulted

The environment: keyword — which links a job to an Azure DevOps Environment and enforces its approval gates — is only available on deployment: jobs. A regular job: has no environment: property. The approval configured in the portal is simply never reached.

Fix
    - deployment: DeployToProd   ← change job → deployment
      pool:
        vmImage: 'ubuntu-latest'
      environment: 'prod'         ← add this — links to the approval gate
      strategy:
        runOnce:
          deploy:
            steps:
              - checkout: none
              - task: AzureWebApp@1
                ...

This is a silent misconfiguration — the pipeline runs successfully and deploys, but the governance control is completely absent. Teams often discover it only when auditing why a hotfix deployed to production without a sign-off.

4

Every pull request deploys to Dev — including draft PRs and WIP branches

Lessons 2 + 13

The team notices that opening a PR to develop triggers a full deployment to the Dev App Service — not just a build-and-test validation. This causes noise: Dev is constantly being overwritten by unfinished branches. Why does this happen? 

trigger:
  branches:
    include:
      - develop
      - 'release/*'

pr:
  branches:
    include:
      - develop

variables:
  - name: isDevelop
    value: $[eq(variables['Build.SourceBranch'], 'refs/heads/develop')]

stages:
  - stage: Build
    jobs:
      - job: BuildJob
        steps:
          - checkout: self
            fetchDepth: 0
          # ... build steps ...

  - stage: DeployDev
    dependsOn: Build
    condition: succeeded()
    jobs:
      - deployment: DeployToDev
        environment: 'dev'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    appName: '$(appServiceNameDev)'
                    package: '$(Pipeline.Workspace)/drop/**/*.zip'

Bug: DeployDev condition does not exclude PR builds

The pr: block correctly allows PR validation to run the Build stage. But the DeployDev condition is just succeeded() — it runs whenever Build succeeds, including when the trigger was a pull request. Build.Reason is PullRequest on PR runs, which is not checked anywhere in the deploy condition.

Fix
  - stage: DeployDev
    dependsOn: Build
    condition: |
      and(
        succeeded(),
        ne(variables['Build.Reason'], 'PullRequest')
      )

The ne(variables['Build.Reason'], 'PullRequest') guard gates the entire deploy stage. PR runs validate (Build stage), then stop. Only real branch pushes proceed to deployment.

5

Prod never deploys on release/* branches — but the condition looks correct

Lesson 13

On a push to release/2.0, the Build stage succeeds, DeployDev is skipped (intentionally — the team only deploys release branches to Dev via main), and DeployProd also gets skipped — even though its condition should make it run. No error, just "skipped." Why? 

variables:
  - name: isDevelop
    value: $[eq(variables['Build.SourceBranch'], 'refs/heads/develop')]
  - name: isMain
    value: $[eq(variables['Build.SourceBranch'], 'refs/heads/main')]
  - name: isRelease
    value: $[startsWith(variables['Build.SourceBranch'], 'refs/heads/release/')]

stages:
  - stage: Build
    ...

  - stage: DeployDev
    dependsOn: Build
    condition: |
      and(
        succeeded(),
        eq(variables.isDevelop, true)
      )

  - stage: DeployProd
    dependsOn: DeployDev
    condition: |
      and(
        succeeded(),
        or(
          eq(variables.isMain, true),
          eq(variables.isRelease, true)
        )
      )

Bug: succeeded() is false when the dependency was skipped

On a release/2.0 push, DeployDev is skipped (its condition is false). DeployProd depends on DeployDev. succeeded() only returns true if the dependency actually ran and passed. A skipped stage counts as not-succeeded, so the entire DeployProd condition is false before the isRelease check even runs.

Fix — two options

Option A: Allow the parent to be skipped:

  - stage: DeployProd
    dependsOn: DeployDev
    condition: |
      and(
        in(dependencies.DeployDev.result, 'Succeeded', 'Skipped'),
        or(
          eq(variables.isMain, true),
          eq(variables.isRelease, true)
        )
      )

Option B: Depend on Build instead of DeployDev:

  - stage: DeployProd
    dependsOn: Build     ← skip the middle stage entirely for release/main
    condition: |
      and(
        succeeded(),
        or(eq(variables.isMain, true), eq(variables.isRelease, true))
      )

Option A is safer for complex pipelines. Option B is simpler when Dev and Prod are truly independent paths.

The five bugs at a glance

Bug Root cause Symptom
1 — GitVersion Missing fetchDepth: 0 Intermittent "Unable to determine commit count"
2 — Artifact path Build.ArtifactStagingDirectory in deploy job "No package found at the specified location"
3 — Approval gate job: instead of deployment: Prod approval gate silently never fires
4 — PR deploys No Build.Reason != PullRequest guard Every PR overwrites Dev App Service
5 — Skipped parent succeeded() is false when parent was skipped DeployProd silently skipped on release branches

Your mental checklist for pipeline reviews

  1. checkout: self — is fetchDepth: 0 set if GitVersion is used?
  2. Deployment jobs — does package: use $(Pipeline.Workspace)/... not $(Build.ArtifactStagingDirectory)?
  3. Prod deploy jobs — is it deployment: with environment:, not job:?
  4. Deploy conditions — does each one include ne(variables['Build.Reason'], 'PullRequest')?
  5. Stage dependencies — does any stage depend on a stage that might be skipped? If so, does the condition use in(dependencies.X.result, 'Succeeded', 'Skipped')?

Primary source

Specify conditions — Azure Pipelines (Microsoft Learn)
