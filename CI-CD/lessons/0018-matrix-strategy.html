Matrix Strategy: One Job, Many Configurations

A matrix fans out one job definition into multiple parallel runs — each with a different value from your matrix. Test on Node 18 and Node 20 simultaneously. Test on Linux and Windows. Run five browser configs at once.

What the matrix does

Without a matrix you write one job, it runs once. With a matrix, Azure Pipelines clones the job once per matrix entry and sets a different variable in each clone. The jobs run in parallel (up to maxParallel).

node_18
Build & Test · Node 18 · ✓
node_20
Build & Test · Node 20 · ✓
node_22
Build & Test · Node 22 · ✗ (failing)

Each row is the same job definition. The only difference is the value of $(nodeVersion). You write the job once; Azure runs it three times.

Minimal example: Node.js version matrix

jobs:
  - job: TestMatrix
    displayName: 'Test on Node.js'
    pool:
      vmImage: 'ubuntu-latest'

    strategy:
      matrix:
        node_18:
          nodeVersion: '18.x'
        node_20:
          nodeVersion: '20.x'
        node_22:
          nodeVersion: '22.x'
      maxParallel: 3          # all three run at the same time

    steps:
      - checkout: self
        fetchDepth: 0

      - task: NodeTool@0
        displayName: 'Install Node.js $(nodeVersion)'
        inputs:
          versionSpec: $(nodeVersion)   # ← matrix variable

      - task: Npm@1
        displayName: 'npm ci'
        inputs:
          command: 'ci'

      - task: Npm@1
        displayName: 'npm test'
        inputs:
          command: 'custom'
          customCommand: 'test -- --ci --watchAll=false'
Matrix keys become job names. Azure Pipelines names each cloned job TestMatrix node_18, TestMatrix node_20, etc. — derived from the matrix key. Use underscores or alphanumerics only; hyphens in keys can cause display issues.

Matrix anatomy

strategy:
  matrix:
    node_18:                # ← job-suffix key (becomes part of the job name)
      nodeVersion: '18.x'     # ← variable set in this leg, readable as $(nodeVersion)
    node_20:
      nodeVersion: '20.x'
  maxParallel: 3                  # ← cap on concurrency (omit = no cap)

Matrix key (e.g. node_18)

Becomes the job-name suffix: the full job name is TestMatrix node_18. Used for display and in downstream dependsOn references. Choose descriptive, lowercase names.

Matrix property (e.g. nodeVersion)

A variable that is set only in that leg of the matrix. Reference it with $(nodeVersion) exactly as any other pipeline variable — in task inputs, scripts, or conditions.

maxParallel

Maximum number of matrix legs that run concurrently. maxParallel: 1 makes them run sequentially — useful when legs write to shared test data (as in the Applications.Apply nightly matrix you saw in Lesson 12).

Multiple variables per leg

Each key can define any number of variables. A cross-platform leg could set both nodeVersion and vmImage, giving you an implicit cross-product.

Multi-variable legs: OS × Node cross-matrix

strategy:
  matrix:
    linux_node20:
      vmImage: 'ubuntu-latest'
      nodeVersion: '20.x'
    windows_node20:
      vmImage: 'windows-latest'
      nodeVersion: '20.x'
    linux_node22:
      vmImage: 'ubuntu-latest'
      nodeVersion: '22.x'
  maxParallel: 3

pool:
  vmImage: $(vmImage)   # ← matrix variable used in pool definition

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: $(nodeVersion)
Explicit cross-products, not automatic. Azure Pipelines does not automatically compute 2×2 matrices. You define each leg explicitly. This is verbose but intentional — you often want to test specific combinations, not all possible ones. An 18 × 3 auto-matrix would create 54 jobs; you usually want fewer.

Conditional legs with matrix variables

You can use matrix variables in condition: expressions, allowing specific steps to only run in certain legs: 

steps:
  - script: npm run test:coverage
    displayName: 'Coverage report (LTS only)'
    condition: eq(variables['nodeVersion'], '20.x')   # ← only on the Node 20 leg

  - task: PublishCodeCoverageResults@2
    displayName: 'Publish coverage'
    condition: eq(variables['nodeVersion'], '20.x')
This is a common pattern: run the full test suite on all Node versions to catch compatibility issues, but only publish the coverage report once (from the primary/LTS leg). Publishing from every leg would create duplicate artifacts and inflate costs.

Depending on a matrix job downstream

A later stage or job that needs all matrix legs to pass uses the job name without the suffix: 

stages:
  - stage: Test
    jobs:
      - job: TestMatrix
        strategy:
          matrix:
            node_18: { nodeVersion: '18.x' }
            node_20: { nodeVersion: '20.x' }
        ...

  - stage: Deploy
    dependsOn: Test          # waits for ALL TestMatrix legs to complete
    condition: succeeded()   # only proceeds if ALL legs passed
    jobs:
      - deployment: DeployToDev
        ...
dependsOn: Test refers to the stage, which completes only when all jobs inside it (including all matrix legs) have finished. If any single matrix leg fails, the stage fails and Deploy is blocked. This is the correct behavior — you don't want to deploy code that failed on Node 18.

Real example: the nightly browser matrix (from Lesson 12)

You saw the Applications.Apply nightly pipeline use a matrix for 8 device/browser combos in Lesson 12. Now you can read exactly what it was doing: 

strategy:
  matrix:
    chromium_desktop:
      DEVICE_NAME: 'Desktop Chrome'
    chromium_tablet:
      DEVICE_NAME: 'Galaxy S8+'
    firefox_desktop:
      DEVICE_NAME: 'Desktop Firefox'
    # ... 5 more entries
  maxParallel: 1   # sequential — shared Dev environment, concurrent writes would conflict

steps:
  - task: DotNetCoreCLI@2
    inputs:
      command: 'test'
      arguments: >-
        --filter "Category=Regression"
        --logger "trx;LogFileName=results-$(DEVICE_NAME).trx"
    env:
      DEVICE_NAME: $(DEVICE_NAME)   # matrix variable injected as env var into test runner
