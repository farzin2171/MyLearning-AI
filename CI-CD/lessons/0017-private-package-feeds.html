Private Package Feeds in Azure Pipelines

Your team hosts internal npm and NuGet packages on Azure Artifacts. This lesson shows exactly how to authenticate to them from a pipeline — and why the same-org vs cross-org distinction determines which approach to use.

Why private feeds exist

Internal shared packages

Your team publishes reusable code (auth helpers, API clients, design tokens) as versioned packages. Internal consumers should not pull from npm/NuGet.org — those packages are private company code.

Upstream proxying

Azure Artifacts can proxy public registries (npm.org, nuget.org). Your pipeline only ever talks to one URL. If a public package disappears, the cached copy still exists in your feed.

The team's feed is hosted at pkgs.dev.azure.com/EquisoftOrganization/…/_packaging/Libraries/…. Services.Authorization uses it for NuGet; the team's JS projects use a separate npm feed reachable via the Equisoft.npm service connection.

The key distinction: same-org vs external feeds

Azure DevOps pipelines authenticate to feeds automatically — but only when both the pipeline and the feed live in the same Azure DevOps organization.

Same organization auto

Pipeline lives in EquisoftOrganization. Feed lives in EquisoftOrganization. Azure injects an auth token automatically — no service connection needed.

External / cross-org manual

Feed lives in a different Azure DevOps org, or on ProGet, GitHub Packages, etc. You must create a service connection and reference it explicitly.

Your pipeline → Service connection → PAT / workload identity → Private feed
Your team's situation: The npm feed uses an explicit service connection (Equisoft.npm) even though both pipeline and feed appear to be in the same org. This is common when the feed is scoped to a specific project and the pipeline lives in a different project — project-scoped feeds require explicit auth.

Implementation

Option A: customEndpoint in Npm@1 (simplest)

This is the team's pattern. The customEndpoint points to a service connection of type npm. The Npm@1 task injects auth credentials automatically at run time — you do not manage tokens yourself. 

- task: Npm@1
  displayName: 'npm ci (private registry)'
  inputs:
    command: 'ci'
    customRegistry: useNpmrc    # ← reads the registry URL from .npmrc
    customEndpoint: 'Equisoft.npm'   # ← service connection name that holds the PAT
customRegistry: useNpmrc — tells the task to read the registry URL from your .npmrc file rather than using the public registry. The task then authenticates to whichever URL is in .npmrc using the credentials from customEndpoint.

The .npmrc file (commit to repo)

; .npmrc — at the root of your repository
registry=https://pkgs.dev.azure.com/EquisoftOrganization/_packaging/LibrariesNpm/npm/registry/
always-auth=true
Do not put credentials in .npmrc. The URL and always-auth=true are safe to commit. The Npm@1 task injects the PAT at run time via the service connection. Hardcoded tokens in source files are a security incident waiting to happen.

Creating the npm service connection

In Azure DevOps: Project Settings → Service connections → New → npm

  • Registry URL: the pkgs.dev.azure.com/…/registry/ URL
  • Authentication: Username and Password (Token)
  • Password: a Personal Access Token with Packaging (Read) scope
  • Name: Equisoft.npm (match what you put in the pipeline)

Combining with the cache from Lesson 16

variables:
  npm_config_cache: $(Pipeline.Workspace)/.npm

steps:
  - task: Cache@2
    inputs:
      key: 'npm | "$(Agent.OS)" | package-lock.json'
      restoreKeys: |
        npm | "$(Agent.OS)"
      path: $(npm_config_cache)

  - task: Npm@1
    displayName: 'npm ci'
    inputs:
      command: 'ci'
      customRegistry: useNpmrc
      customEndpoint: 'Equisoft.npm'   # ← add this to your Lesson 16 snippet
The cache is keyed on package-lock.json — packages from the private registry are cached exactly like public packages. Authentication is only needed on a cache miss (first run or after a dependency change).

Option A: Inline feed selection (same-org feeds)

When your pipeline and feed are in the same Azure DevOps organization, reference the feed ID directly in DotNetCoreCLI@2. Azure injects auth automatically. 

- task: DotNetCoreCLI@2
  displayName: 'Restore'
  inputs:
    command: 'restore'
    projects: '**/*.csproj'
    feedsToUse: 'select'
    vstsFeed: '7b9dd9ad-2823-4d24-b1e5-6d0b0f3c4601/Libraries'
    # Format: /
    # OR just  for org-scoped feeds
Find the feed ID: In Azure DevOps, go to Artifacts → select the feed → Connect to feed → .NET → copy the source URL. The GUID is the project ID.

Option B: NuGet.Config + external credentials (Services.Authorization pattern)

The team's real approach. The NuGet.Config file is committed to the repo; the pipeline provides credentials for it via a service connection.

NuGet.Config (committed to repo)

<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <clear />
    <add key="DIT.Libraries"
         value="https://pkgs.dev.azure.com/EquisoftOrganization/7b9dd9ad-2823-4d24-b1e5-6d0b0f3c4601/_packaging/Libraries/nuget/v3/index.json" />
  </packageSources>
</configuration>

Pipeline YAML

- task: DotNetCoreCLI@2
  displayName: 'Restore'
  inputs:
    command: 'restore'
    projects: '**/*.csproj'
    feedsToUse: 'config'
    nugetConfigPath: 'NuGet.Config'
    externalFeedCredentials: 'Equisoft.nuget'   # ← NuGet service connection name
externalFeedCredentials maps a service connection name to the feed URL defined in NuGet.Config. The task rewrites the config at run time to inject the auth token — the committed config never contains secrets.
When to use Option B over A: Use Option B when your project already has a NuGet.Config that developers use locally (as Services.Authorization does). Developers get auth via their Azure DevOps credential provider; the pipeline gets it via the service connection. One config file serves both contexts.

Common errors & fixes

Error Cause Fix
401 Unauthorized Service connection not granted access to the pipeline, or PAT expired Library → Service connection → Security → add the pipeline; regenerate PAT if expired
404 Not Found on package Wrong feed URL, or package only exists in a different feed/scope Verify URL in Azure Artifacts → Connect to feed → copy the exact URL
npm ERR! code E401 (no .npmrc) .npmrc not committed, or registry URL missing Add .npmrc with registry URL + always-auth=true; commit to repo
Service connection not found Name in YAML does not exactly match the service connection name Check Project Settings → Service connections — names are case-sensitive
