Templates & extends:

How your team packages reusable pipeline logic and how app repos consume it safely.

Why this lesson

Every app pipeline in your team starts with roughly these three lines: 

resources:
  repositories:
    - repository: templates
      name: PipelineTemplates
      type: git
extends:
  template: /CI/build-dotnet-service-pipeline.yml@templates
  parameters:
    buildDocumentation: true
    deployOctopusRelease: false

Until you understand extends:, template parameters, and compile-time conditionals, you're reading this as magic. After this lesson, you'll see exactly what each line does.

Two ways to use templates

template:
Inline reference
inject steps / jobs / stages
vs
extends:
Full pipeline inheritance
security boundary

template: — inserts a chunk of YAML (steps, jobs, or stages) from a file into the current pipeline. Used heavily inside the PipelineTemplates repo itself, e.g. a stage template calling a step template. Templates docs →

extends: — the pipeline is the template. The app's azure-pipelines.yml hands full control to the template and can only pass parameters. This is a security boundary: admins can require that all pipelines extend an approved template, preventing ad-hoc steps from running arbitrary code. Template security →

Your team's pattern
App repos use extends: — the whole pipeline is controlled by PipelineTemplates. Inside PipelineTemplates, templates call each other with template:.

Step 1 — registering the template repo

Before you can reference any template from PipelineTemplates, you must declare the repo as a resource. The repository: value becomes the alias you use in @alias references. README.md 

resources:
  repositories:
    - repository: templates       # this becomes the @templates alias
      name: PipelineTemplates       # repo name in Azure DevOps
      ref: refs/heads/master       # which branch of templates to use
      type: git                     # git = Azure Repos

After this, any template reference appends @templates: 

# Inside an app repo — must use @templates suffix
- template: libs/git/gitversion-steps.yml@templates

# Inside PipelineTemplates itself — relative path, no @suffix needed
- template: deploy-release-steps.yml

The four template types

A template file's type is determined by what it defines at the top level, not by its filename.

Step template
Defines: steps:
gitversion-steps.yml
build-app-steps.yml
Job template
Defines: jobs:
build-db-on-linux-job.yml
create-git-tag-job.yml
Stage template
Defines: stages:
deploy-release-process.yml
run-tests-stage.yml
Pipeline template
Defines: parameters: + full pipeline
build-dotnet-service-pipeline.yml
ghas-scan-reactjs-pipeline.yml

Typed template parameters

Templates declare their inputs with a parameters: block. Each parameter has a type, a default, and an optional displayName. Parameter types → 

# From V2/build-dotnet-service-pipeline.yml
parameters:
  - name: dotNetVersion
    type: string            # string, number, boolean, object, step, stepList, job, jobList, stage, stageList
    default: '8.0.x'

  - name: runIntegrationTests
    type: boolean
    default: true

  - name: applicationBuildList   # a list of objects — used with ${{ each }}
    type: object

  - name: copyFilesStep         # caller provides an entire step definition
    type: step
    default:
      script: echo "no copy step provided"
type: step — a real team usage
build-configuration-pipeline.yml accepts a type: step parameter called copyFilesStep. The caller injects a custom step definition, and the template inserts it inline with ${{ parameters.copyFilesStep }}. This is how the template stays generic while each project customises one step. V2/build-configuration-pipeline.yml L9

Compile-time conditionals — ${{ if }}

Template expressions run at compile-time, so ${{ if }} can add or remove entire steps, jobs, or stages before the pipeline runs. This is how your templates implement optional features. V2/libs/net/build-with-tests-steps.yml L75 

steps:
  # Always runs
  - task: DotNetCoreCLI@2
    displayName: 'Build projects'
    ...

  # Only compiled into the pipeline if skipRunTests is not true
  - ${{ if ne(parameters.skipRunTests, true) }}:
    - task: DotNetCoreCLI@2
      displayName: 'Tests projects'
      ...

    - task: reportgenerator@5
      displayName: 'Generate test report'
      ...
${{ if }} vs condition:
${{ if }} removes steps entirely at compile-time — the agent never sees them. condition: runs at runtime and skips a step that is still in the plan. Use ${{ if }} for structural YAML changes (adding/removing whole steps); use condition: for runtime decisions (succeed/fail/branch-check).

Looping over object parameters — ${{ each }}

When a parameter is type: object, templates can iterate over it to generate steps or jobs dynamically. Your Octopus deploy templates do this to create one release per project in the list. V2/build-configuration-pipeline.yml L130 

# Template definition — iterates over the list passed by the caller
- ${{ each project in parameters.octopusProjectList }}:
    - task: OctopusCreateRelease@5
      displayName: 'Create release for ${{ project.projectName }}'
      inputs:
        Space: '${{ project.projectSpace }}'
        ProjectName: '${{ project.projectName }}'
# Caller (app repo) provides the list
extends:
  template: /CI/build-dotnet-service-pipeline.yml@templates
  parameters:
    applicationBuildList:
      - name: MyService
        packageType: publish
        octopusProjectList:
          - projectSpace: Spaces-222
            projectName: Base Config Common
            deployRelease: false

Nested ${{ if }} inside ${{ each }}

The Octopus deploy stage template combines both: loop over projects, then only deploy the ones where deployRelease is true. V2/libs/octopus/deploy-release-process.yml L25 

steps:
  - ${{ each project in parameters.octopusProjectList }}:
    - ${{ if eq(project.deployRelease, true) }}:    # filter within the loop
      - template: deploy-release-steps.yml
        parameters:
          octopusProjectName: '${{ project.projectName }}'
          octopusProjectEnvironment: "${{ coalesce(project.projectDevEnvironment, 'Development') }}"
coalesce()
coalesce(a, b) returns the first non-empty value. Here: if the caller didn't set projectDevEnvironment, the template falls back to 'Development'. It's the template equivalent of a default fallback. coalesce() docs →

The full call chain in your team

azure-pipelines.yml
App repo
extends: …@templates
→
build-dotnet-service-pipeline.yml
Pipeline template
stages + jobs defined
→
libs/net/build-with-tests-steps.yml
Step template
template: (relative)
→
libs/octopus/deploy-release-process.yml
Stage template
template: (relative)

Each level only knows its own parameters. The app repo can't see inside the step template — it can only pass values to the pipeline template, which passes subsets down the chain.
