Playwright Automation Tests in the Pipeline

How pipeline parameters travel through environment variables into C# test code — and how to debug a failed test run

What you're dealing with

The automated-tests.yml template runs C# + NUnit tests that use Microsoft.Playwright.NUnit — the .NET binding of Playwright (not the JavaScript version). Each test opens a real browser, navigates the live application, and asserts on what it sees. Understanding how the pipeline controls this is essential for debugging failed runs and for adding new tests.

How pipeline parameters reach the C# test code

The data flow has three hops: template parameter → environment variable → C# code reading it.

parameters.baseUrl
→
env: BASE_URL
→
Environment.GetEnvironmentVariable("BASE_URL")
— target URL for the live environment
parameters.deviceName
→
env: DEVICE_NAME
→
Environment.GetEnvironmentVariable("DEVICE_NAME")
— which device to emulate
parameters.browserName
→
Playwright.BrowserName runsetting
→
PageTest internals
— which browser engine (chromium, webkit, firefox)
parameters.expectTimout
→
Playwright.ExpectTimeout runsetting
→
PageTest internals
— how long Expect() waits (default 45 000 ms)
automated-tests.yml (pipeline)
- task: DotNetCoreCLI@2
  env:
    BASE_URL: "${{ parameters.baseUrl }}"
    DEVICE_NAME: "${{ parameters.deviceName }}"
  inputs:
    command: test
    arguments: >
      --filter "Category=Deployment"
      Playwright.BrowserName=
        ${{ parameters.browserName }}
      Playwright.ExpectTimeout=
        ${{ parameters.expectTimout }}
Main.cs (C# test)
public string GetBaseURL() {
  var baseURL =
    Environment.GetEnvironmentVariable(
      "BASE_URL");
  // throws if not set
  return baseURL;
}

public override BrowserNewContextOptions
  ContextOptions() {
  var deviceName =
    Environment.GetEnvironmentVariable(
      "DEVICE_NAME");
  var device = Playwright.Devices[deviceName];
  // uses Playwright device registry
}

The Playwright.BrowserName and Playwright.ExpectTimeout values are passed as MSTest/NUnit runsettings (-- KEY=VALUE after the double dash in dotnet test). They're consumed internally by Microsoft.Playwright.NUnit's PageTest base class — your code doesn't read them directly; Playwright's test infrastructure uses them to launch the correct browser and configure timeouts.

Why only some tests run in the pipeline

The pipeline passes --filter "Category=Deployment" to dotnet test. This runs only tests decorated with [Category("Deployment")].

Runs in pipeline smoke test
[Test, Category("Deployment")]
public async Task ApplyForms()
{
  // login, create app, delete app,
  // navigate pages — a full happy path
  await Login(Page, Users.User1);
  await Expect(Page.GetByRole(
    AriaRole.Heading,
    new() { Name = "Application Overview" }
  )).ToBeVisibleAsync();
  // ...
}
Does NOT run in smoke test
[Test]
// no Category("Deployment") —
// only runs in nightly full matrix
public async Task AppCaptureForms()
{
  // deeper AppCapture workflow
  // skipped in postRouteTraffic
}

The postRouteTraffic hook in the deployment stage runs a quick smoke test — one browser, one device (Desktop Chrome), only Category=Deployment tests. The nightly pipeline runs the full suite across 8 device/browser combinations with no category filter.

ContextFilterDevicesWhen
postRouteTraffic (deploy pipeline) Category=Deployment Desktop Chrome only After every Dev/QA deploy
azure-pipelines-tests.yml (nightly) No filter — all tests 8 device/browser combos 2 AM daily, always

How device emulation works

When DEVICE_NAME is set (e.g. "iPhone 15 Pro Max"), ContextOptions() in Main.cs looks it up in Playwright's built-in device registry:

// From Main.cs — ContextOptions()
var deviceName = Environment.GetEnvironmentVariable("DEVICE_NAME");

if (!string.IsNullOrEmpty(deviceName))
{
    var device = Playwright.Devices[deviceName];   // ← Playwright knows ~50 real devices
    context = new BrowserNewContextOptions(device)
    {
        BaseURL = context.BaseURL,
        Locale  = context.Locale,
        TimezoneId = context.TimezoneId
    };
}

IsMobile = context.IsMobile.GetValueOrDefault(false);
Viewport = context.ViewportSize;

The Playwright.Devices["iPhone 15 Pro Max"] lookup sets the viewport size, user agent string, touch support, and pixel ratio — exactly as if a real iPhone was browsing. The test code then reads IsMobile to decide between hamburger-menu paths and desktop-menu paths.

If DEVICE_NAME is empty, the test runs with a standard desktop viewport. This is what happens when you run the tests locally without setting the env var — desktop behaviour, no device emulation.

What gets captured when a test fails

The TearDown method in Main.cs automatically captures debugging artefacts when a test fails and attaches them to the Azure DevOps test result:

// From Main.cs — TearDown
[TearDown]
public async Task TearDown()
{
    if (!IsTestPassed)
    {
        // 1. Screenshot of the browser at the moment of failure
        var screenshotFile = Path.Combine(
            TestContext.CurrentContext.WorkDirectory,
            $"screenshot-{TestFullName}.png");
        await Page.ScreenshotAsync(new() { Path = screenshotFile });
        TestContext.AddTestAttachment(screenshotFile, "Error Snapshot:");

        // 2. Playwright trace — every action, DOM snapshot, network log
        var traceFile = Path.Combine(
            TestContext.CurrentContext.WorkDirectory,
            $"trace-{TestFullName}.zip");
        await Page.Context.Tracing.StopAsync(new() { Path = traceFile });
        TestContext.AddTestAttachment(traceFile, "Test Trace:");
    }
    // On pass: tracing is stopped without saving (no overhead in the artifact)
}

Tracing starts in SetUp (before each test) and stops in TearDown. On failure, both the PNG and the trace ZIP are attached to the test result in Azure DevOps.

To debug a failing Playwright test:

1. In Azure DevOps, open the pipeline run → click the failing job → click the Tests tab → click the failing test → click Attachments
2. Download the trace-*.zip file
3. Open it at trace.playwright.dev — you'll see a timeline of every click, navigation, and DOM snapshot, with network requests alongside

The nightly matrix: 8 browsers, one at a time

The nightly pipeline uses a matrix strategy to run the same test suite across 8 device/browser combinations. The critical detail: maxParallel: 1 means they run sequentially, not simultaneously.

# From azure-pipelines-tests.yml
strategy:
  matrix:
    'Desktop Chrome':
      deviceName: "Desktop Chrome"
      browserName: "chromium"
    'iPhone 15 Pro Max':
      deviceName: "iPhone 15 Pro Max"
      browserName: "webkit"
    # ... 6 more combinations
  maxParallel: 1          # ← run one at a time, not all 8 in parallel

With maxParallel: 1, one matrix job starts, runs all tests to completion, then the next starts. If it were set to 8, all eight browsers would hit the Dev environment simultaneously — creating interference between tests that create/modify/delete the same application records.

The 120-minute timeout (timeoutInMinutes: 120 on the job) is needed because 8 sequential runs of the full test suite can take well over an hour. If you're seeing nightly runs expire, this is where to look first.

The Page Object Model structure

The test project follows the Page Object Model (POM) pattern — each page/component of the app has a corresponding C# class that wraps its Playwright locators.

tests/Equisoft.Apply.AutomationTests/
├── Main.cs                    ← PageTest base: browser setup, env vars, tracing
├── BaseTest.cs                ← shared helpers: GetToken(), CreateSimplifiedIssueForm()
├── Common/
│   ├── Pages/
│   │   ├── ApplicationOverviewPage.cs   ← locators for the home page table
│   │   ├── TopMenu.cs                   ← login, logout, language switch
│   │   └── SignInPage.cs                ← AzureADB2C login form
│   ├── Modals/
│   │   ├── DeleteModal.cs               ← confirm-delete dialog
│   │   └── StartNewApplicationModal.cs  ← new application wizard
│   └── Controls/
│       └── CalendarControl.cs           ← date picker component
├── Pages/
│   └── SimplifiedIssue/                 ← page objects for the application form wizard
└── Tests/
    ├── HomePageHapyPath/
    │   └── HomePageHappyPath.cs         ← [Category("Deployment")] smoke test
    ├── SimplifiedIssue/                 ← full workflow tests (nightly only)
    └── SpecificPermissions/             ← permission-based test matrix

When a test fails, the trace zip shows you exactly which page object method was executing. Cross-reference the failing action in the trace against the page object class to find where to look in the code.
