Build Your Own CI/CD Pipeline to Azure App Service

A complete, step-by-step walkthrough — from empty repository to a running pipeline that builds, tests, and deploys to Azure App Service. No Octopus needed.

You have spent 12 lessons learning individual pipeline concepts. This lesson puts them all together into one coherent system you own end-to-end. The model is Services.Authorization (the team's sample .NET 8 service) — but the pipeline pattern here replaces Octopus with a direct Azure App Service deploy, so you can use it independently for any project.

What you will end up with

Push code → ① Build & Test → ② Deploy → Dev → Approval gate → ③ Deploy → Prod

How this differs from the team's current setup

A Prerequisites — Set up Azure resources (do once)

These are one-time tasks in the Azure portal and in Azure DevOps. You cannot run the pipeline until they exist.

A1 Create two Azure App Services

In the Azure portal:

  • Go to App Services → Create
  • Create a Dev instance — e.g. my-service-dev
  • Create a Prod instance — e.g. my-service-prod
  • Runtime stack: .NET 8 (LTS) · OS: Linux
  • Note both names — you will paste them into a variable group in step B1
Linux vs Windows: The pipeline uses appType: 'webAppLinux'. If you chose Windows, change that to appType: 'webApp'.
A2 Create a Service Connection in Azure DevOps

This is the credential that lets Azure Pipelines talk to your Azure subscription.

  • In Azure DevOps, go to Project Settings → Service connections → New service connection
  • Choose Azure Resource Manager
  • Authentication: Workload Identity Federation (recommended) or Service principal
  • Select the subscription that contains your App Services
  • Name it — e.g. my-azure-subscription — you will reference this name in the pipeline
  • Tick Grant access to all pipelines
Write down the exact connection name. The pipeline variable azureServiceConnection must match character-for-character.

B Azure DevOps — Environments & Variable Group

B1 Create a Variable Group

A variable group keeps secrets and environment-specific values out of your YAML. Go to Pipelines → Library → + Variable group.

Name it my-service-vars and add these three variables:

azureServiceConnection   = my-azure-subscription    (the name from A2)
appServiceNameDev        = my-service-dev            (name from A1)
appServiceNameProd       = my-service-prod           (name from A1)
You can mark azureServiceConnection as a secret if you like, but since it's just a name (not a password) it doesn't need to be.
B2 Create the Dev environment
  • Go to Pipelines → Environments → New environment
  • Name: dev · Resource: None
  • Leave approvals off — dev deploys automatically
B3 Create the Prod environment with an approval gate
  • Create another environment named prod
  • Click ⋯ → Approvals and checks → + Add → Approvals
  • Add yourself (or your team lead) as approver
  • Save
Why environments matter: The pipeline's environment: 'prod' key links the deployment job to this approval gate. You learned this in Lesson 6 — the gate lives in Azure DevOps, not in the YAML.

C Repository Setup

C1 Create a new repository
  • In Azure DevOps, go to Repos → + New repository (or use GitHub)
  • Initialize with a .gitignore (choose VisualStudio)
  • Clone it locally: git clone <url>
C2 Add GitVersion.yml

At the repository root, create GitVersion.yml:

mode: ContinuousDeployment
branches: {}
ignore:
  sha: []
Why ContinuousDeployment? Every commit gets its own unique version number — never a duplicate. You learned this in Lesson 7. Services.Authorization uses the identical config.
C3 Set up your branch strategy

Create the branches the pipeline expects:

git checkout -b develop
git push -u origin develop
  • Day-to-day work: commit to develop → auto-deploys to Dev
  • Release prep: branch to release/1.0 → deploys to Dev (waiting for Prod approval)
  • Production: merge to main + tag → deploys to Prod after approval

D The Pipeline YAML — Full File

Create azure-pipelines.yml at your repository root and paste the following. Every section maps to a concept you've already learned — annotations below explain the connections. 

# azure-pipelines.yml
# ─────────────────────────────────────────────────────────────────
# CI/CD pipeline for <your service>
# Deploys to Azure App Service (no Octopus required)
# ─────────────────────────────────────────────────────────────────

trigger:
  branches:
    include:
      - main
      - develop
      - 'release/*'
      - 'hotfix/*'
  tags:
    include:
      - '*'

pr:
  branches:
    include:
      - main
      - develop

variables:
  - group: 'my-service-vars'          # ← variable group from step B1
  - name: isDevelop
    value: $[eq(variables['Build.SourceBranch'], 'refs/heads/develop')]
  - name: isMain
    value: $[eq(variables['Build.SourceBranch'], 'refs/heads/main')]
  - name: isRelease
    value: $[startsWith(variables['Build.SourceBranch'], 'refs/heads/release/')]
  - name: isHotfix
    value: $[startsWith(variables['Build.SourceBranch'], 'refs/heads/hotfix/')]
  - name: isTag
    value: $[startsWith(variables['Build.SourceBranch'], 'refs/tags/')]
  - name: isPR
    value: $[eq(variables['Build.Reason'], 'PullRequest')]
  - name: buildConfiguration
    value: 'Release'
  - name: dotnetVersion
    value: '8.0.x'

stages:

  # ── ① BUILD ───────────────────────────────────────────────────
  - stage: Build
    displayName: '① Build & Test'
    jobs:
      - job: BuildJob
        displayName: 'Restore → Build → Test → Publish'
        pool:
          vmImage: 'ubuntu-latest'
        workspace:
          clean: all
        steps:
          - checkout: self
            fetchDepth: 0             # GitVersion needs full history (Lesson 7)

          - task: gitversion/setup@3.2.1
            displayName: 'Install GitVersion'
            inputs:
              versionSpec: '5.x'

          - task: gitversion/execute@3.2.1
            displayName: 'Calculate Version'
            name: GitVersion

          - script: echo "##[section]Building version $(GitVersion.SemVer)"
            displayName: 'Log version'

          - task: UseDotNet@2
            displayName: 'Install .NET $(dotnetVersion)'
            inputs:
              version: '$(dotnetVersion)'

          - task: DotNetCoreCLI@2
            displayName: 'Restore'
            inputs:
              command: 'restore'
              projects: '**/*.csproj'

          - task: DotNetCoreCLI@2
            displayName: 'Build'
            inputs:
              command: 'build'
              projects: '**/*.csproj'
              arguments: >-
                --no-restore
                --configuration $(buildConfiguration)
                /p:Version=$(GitVersion.SemVer)
                /p:AssemblyVersion=$(GitVersion.AssemblySemVer)

          - task: DotNetCoreCLI@2
            displayName: 'Unit Tests'
            inputs:
              command: 'test'
              projects: '**/*.UnitTests.csproj'
              arguments: >-
                --no-build
                --configuration $(buildConfiguration)
                --collect:"XPlat Code Coverage"
            publishTestResults: true

          - task: DotNetCoreCLI@2
            displayName: 'Publish'
            inputs:
              command: 'publish'
              publishWebProjects: true
              arguments: >-
                --no-build
                --configuration $(buildConfiguration)
                --output $(Build.ArtifactStagingDirectory)
              zipAfterPublish: true   # ← produces the .zip AzureWebApp@1 expects

          - task: PublishPipelineArtifact@1
            displayName: 'Upload drop artifact'
            inputs:
              targetPath: '$(Build.ArtifactStagingDirectory)'
              artifact: 'drop'

  # ── ② DEPLOY → DEV ────────────────────────────────────────────
  - stage: DeployDev
    displayName: '② Deploy → Dev'
    dependsOn: Build
    condition: |
      and(
        succeeded(),
        ne(variables.isPR, true),
        or(
          eq(variables.isDevelop, true),
          eq(variables.isRelease, true),
          eq(variables.isHotfix, true),
          eq(variables.isTag, true)
        )
      )
    jobs:
      - deployment: DeployToDev
        displayName: 'Azure App Service: Dev'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'dev'            # ← links to Azure DevOps environment (Lesson 6)
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none      # no source needed — artifact downloaded automatically

                - task: AzureWebApp@1
                  displayName: 'Deploy to $(appServiceNameDev)'
                  inputs:
                    azureSubscription: '$(azureServiceConnection)'   # from variable group
                    appType: 'webAppLinux'
                    appName: '$(appServiceNameDev)'
                    package: '$(Pipeline.Workspace)/drop/**/*.zip'
                    deploymentMethod: 'auto'
                    appSettings: |
                      -ASPNETCORE_ENVIRONMENT Staging
                      -ASPNETCORE_URLS "http://+:80"

  # ── ③ DEPLOY → PROD ───────────────────────────────────────────
  - stage: DeployProd
    displayName: '③ Deploy → Prod'
    dependsOn: DeployDev
    condition: |
      and(
        succeeded(),
        or(
          eq(variables.isMain, true),
          eq(variables.isTag, true)
        )
      )
    jobs:
      - deployment: DeployToProd
        displayName: 'Azure App Service: Prod'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'prod'           # ← approval gate enforced here (Lesson 6)
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none

                - task: AzureWebApp@1
                  displayName: 'Deploy to $(appServiceNameProd)'
                  inputs:
                    azureSubscription: '$(azureServiceConnection)'
                    appType: 'webAppLinux'
                    appName: '$(appServiceNameProd)'
                    package: '$(Pipeline.Workspace)/drop/**/*.zip'
                    deploymentMethod: 'auto'
                    appSettings: |
                      -ASPNETCORE_ENVIRONMENT Production
                      -ASPNETCORE_URLS "http://+:80"

Key YAML decisions — annotated

fetchDepth: 0 on checkout — GitVersion reads the full commit history to calculate a valid semantic version. Without this, you get "Unable to determine the commit count" errors (Lesson 7).
zipAfterPublish: true on DotNetCoreCLI publish — this is what produces the .zip file that AzureWebApp@1 expects via package: '$(Pipeline.Workspace)/drop/**/*.zip'. Without it, you'd get a directory, and the task wouldn't know what to upload.
checkout: none in deployment jobs — deployment jobs automatically download all published artifacts. You don't need source code to deploy — only the zip. Skipping checkout saves ~30 seconds per environment (Lesson 5).
Condition on DeployDev — excludes PR builds (ne(variables.isPR, true)) so that pull request validation never triggers a real deployment. You learned this pattern in Lesson 2.
Condition on DeployProd — only runs on main or a tag. A release branch commit goes to Dev, then stops. Only a merge to main (or a tag) triggers Prod.
appSettings block — overrides App Service application settings at deploy time. This is equivalent to the environment variables you saw in docker-compose.yml in Services.Authorization. Each line is -KEY value (note the leading dash).

E Register the Pipeline & Run It

E1 Commit and push
git add azure-pipelines.yml GitVersion.yml
git commit -m "ci: initial pipeline"
git push
E2 Register the pipeline in Azure DevOps
  • Go to Pipelines → New pipeline
  • Where is your code? → Azure Repos Git (or GitHub)
  • Select your repository
  • Configure: Existing Azure Pipelines YAML file
  • Path: /azure-pipelines.yml
  • Click Continue → Run
E3 Watch the first run

From the develop branch, you should see three stages:

  • ① Build & Test — runs, produces a drop artifact
  • ② Deploy → Dev — runs automatically (no approval needed)
  • ③ Deploy → Prod — skipped (not on main or a tag)
Success check: Open your Dev App Service URL in a browser. Your app should be running. The version is visible via the GitVersion-stamped assembly: check /hc or your health-check endpoint.
E4 Promote to Prod
git checkout main
git merge develop
git tag v1.0.0
git push origin main --tags
  • The Build stage runs again on the tag
  • DeployDev runs (tag is in the condition)
  • DeployProd starts — pauses at the approval gate
  • You receive an email / Teams notification → click Approve
  • Deployment proceeds to your Prod App Service

F Adapting for Node.js / React

If your project is JavaScript/TypeScript instead of .NET, replace the Build stage steps with the following. The Deploy stages are identical — Azure App Service doesn't care about the language. 

      steps:
        - checkout: self
          fetchDepth: 0

        - task: gitversion/setup@3.2.1
          inputs:
            versionSpec: '5.x'

        - task: gitversion/execute@3.2.1
          name: GitVersion

        - task: NodeTool@0
          displayName: 'Install Node.js'
          inputs:
            versionSpec: '20.x'

        - task: Npm@1
          displayName: 'npm ci'
          inputs:
            command: 'ci'

        - task: Npm@1
          displayName: 'npm test'
          inputs:
            command: 'custom'
            customCommand: 'test -- --watchAll=false --ci'

        - task: Npm@1
          displayName: 'npm build'
          inputs:
            command: 'custom'
            customCommand: 'run build'

        # Zip the build output so AzureWebApp@1 can deploy it
        - task: ArchiveFiles@2
          displayName: 'Zip build output'
          inputs:
            rootFolderOrFile: '$(System.DefaultWorkingDirectory)/build'
            includeRootFolder: false
            archiveType: 'zip'
            archiveFile: '$(Build.ArtifactStagingDirectory)/drop.zip'
            replaceExistingArchive: true

        - task: PublishPipelineArtifact@1
          displayName: 'Upload drop artifact'
          inputs:
            targetPath: '$(Build.ArtifactStagingDirectory)/drop.zip'
            artifact: 'drop'
Node.js App Service runtime: Set the App Service runtime to Node 20 LTS in the Azure portal, and add a startup command node server.js (or pm2 start server.js for production). For a React SPA served statically, you might prefer Azure Static Web Apps instead.

Common problems & fixes

SymptomLikely causeFix
"No package found" in AzureWebApp@1 zipAfterPublish is false, or artifact name mismatch Confirm zipAfterPublish: true and artifact name is drop
GitVersion error: "Unable to determine commit count" fetchDepth is 1 (shallow clone) Set fetchDepth: 0 on checkout
DeployProd never runs, even after merge to main DeployDev condition skipped the Dev stage, so DeployProd's dependsOn sees a skipped parent Check DeployDev condition — it must succeed (not skip) for a main branch push
"Service connection not found" Variable group not linked to the pipeline, or connection name typo Check Library → Variable group → Pipeline permissions; verify spelling
App deployed but returns HTTP 500 Missing App Service application settings (connection strings etc.) Add settings in appSettings block, or configure in Azure portal under Configuration
