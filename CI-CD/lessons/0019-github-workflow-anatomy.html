GitHub Actions Workflow Anatomy

Three levels instead of four — and how every Azure Pipelines concept you know maps across.

Why this lesson

You know the Azure Pipelines hierarchy cold: Pipeline → Stage → Job → Step. GitHub Actions uses the same ideas with different names, a flatter structure, and a few important quirks. Once the mapping clicks, any GitHub workflow file becomes readable in seconds — and you'll write them with the same confidence you now have with Azure YAML.

The three-level hierarchy

GitHub Actions uses three levels instead of Azure's four. The key difference: there is no stages: keyword.

Workflow
One .yml file under .github/workflows/ = one workflow. Defines the events that trigger it and contains all jobs. Equivalent to: Pipeline
↳
Job
A named group of steps that runs on one runner. Jobs run in parallel by default. Use needs: to sequence them. Equivalent to: Job (same concept, same name)
↳
Step
A single unit of work: either uses: an Action, or run: a shell command. Steps are sequential within a job. Equivalent to: Step (same concept, same name)
The missing Stage
Azure Pipelines has four levels; GitHub Actions has three. Where you used a stages: block to group and sequence jobs, in GitHub Actions you wire job ordering with needs:. The GitHub UI renders the dependency graph automatically — it looks exactly like a stage pipeline, just without the YAML keyword.

Where workflow files live

Unlike Azure Pipelines (where you configure the YAML file path in the DevOps UI), GitHub auto-discovers any .yml file under .github/workflows/. Every file there becomes its own workflow. No registration required — just push and it's active. 

# Typical multi-workflow layout
.github/
  workflows/
    ci.yml         # build + test on every push / PR
    deploy.yml     # deploy to Azure on main push
    release.yml    # publish package on tag push
    nightly.yml    # scheduled integration tests

Annotated workflow: Node.js CI

A complete workflow that builds and tests a TypeScript project on every push and PR. Read the numbers alongside the annotations: 

name: CI                              ①

on:                                   ②
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:                                  ③
  build:                               ④
    runs-on: ubuntu-latest           ⑤
    steps:
      - uses: actions/checkout@v4    ⑥
      - uses: actions/setup-node@v4  ⑦
        with:
          node-version: '20'
          cache: 'npm'
      - name: Install
        run: npm ci                    ⑧
      - name: Build
        run: npm run build
      - name: Test
        run: npm test

  deploy:
    needs: build                      ⑨
    if: github.ref == 'refs/heads/main'⑩
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Azure
        run: echo "deploy step here"
  • ① Display name shown in the GitHub Actions UI. Optional but useful — without it GitHub uses the filename.
  • ② The event block. Everything that can trigger this workflow goes under a single on: key. Replaces Azure's separate top-level trigger: and pr: keys.
  • ③ All jobs live under jobs:. No stages: wrapper — jobs are the top-level unit.
  • ④ The job ID (build) is a YAML key. It's used by needs: and appears in the UI. Unlike Azure job IDs, it's also the display name unless you add a name: field.
  • ⑤ Runner selection. GitHub-hosted: ubuntu-latest, windows-latest, macos-latest. Replaces Azure's pool: vmImage:.
  • ⑥ Checkout is NOT automatic. You must always add this step — unlike Azure where the agent checks out by default. Forgetting this is the #1 beginner mistake.
  • ⑦ Actions use uses: owner/repo@ref. The with: block passes inputs — like Azure's inputs: on a task. cache: 'npm' enables npm caching without an extra step.
  • ⑧ run: executes shell commands. Bash is the default on Linux runners. Replaces Azure's script: or bash: step type.
  • ⑨ needs: build makes this job wait for the build job to succeed. Replaces Azure's dependsOn:.
  • ⑩ if: conditions jobs (or steps). github.ref is the branch/tag ref. This replaces Azure's condition: with runtime expressions.

Sequencing jobs — the needs: pattern

Without stages, needs: is the only sequencing mechanism. A job with needs: [build, test] waits for both to succeed before starting: 

jobs:
  build:                          # runs immediately
    runs-on: ubuntu-latest
    steps: # ...

  test:
    needs: build                  # waits for build
    runs-on: ubuntu-latest
    steps: # ...

  deploy:
    needs: [build, test]          # waits for BOTH
    runs-on: ubuntu-latest
    steps: # ...

GitHub renders this as a visual graph in the workflow run view. The result looks identical to a three-stage Azure pipeline — Build → Test → Deploy.

Common events — beyond push and PR

on:
  # Scheduled run (cron — same syntax as Azure schedules)
  schedule:
    - cron: '0 2 * * 1-5'        # weekdays at 02:00 UTC

  # Manual trigger — like clicking "Run pipeline" in Azure DevOps
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deploy target'
        required: true
        default: 'staging'
        type: choice
        options: [staging, production]

  # Called by another workflow — like an Azure template
  workflow_call:
    inputs:
      node-version:
        type: string
        required: false
        default: '20'
workflow_dispatch vs workflow_call
workflow_dispatch = triggered manually by a human (UI button or gh workflow run). workflow_call = triggered programmatically by another workflow. One workflow can declare both, making it both manually runnable and callable as a reusable workflow.

Context variables — what replaces $(Build.SourceBranch)

Azure Pipelines has predefined variables like $(Build.SourceBranch). GitHub Actions has contexts — objects accessible via ${{ context.property }}: 

# Common context values
${{ github.ref }}              # refs/heads/main  (AZ: Build.SourceBranch)
${{ github.ref_name }}         # main             (AZ: Build.SourceBranchName)
${{ github.sha }}              # abc1234…         (AZ: Build.SourceVersion)
${{ github.run_number }}       # 42               (AZ: Build.BuildNumber)
${{ github.actor }}           # username         (AZ: Build.RequestedFor)
${{ github.repository }}      # owner/repo       (no direct AZ equivalent)
${{ secrets.MY_SECRET }}      # repo secret      (AZ: $(secret-variable))
${{ vars.MY_VAR }}            # repo variable    (AZ: variable group value)
