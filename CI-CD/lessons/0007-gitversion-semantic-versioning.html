GitVersion and Semantic Versioning

Where $(GitVersion.SemVer) comes from and why it changes by branch

The problem GitVersion solves

You've seen $(GitVersion.SemVer) and $(GitVersion.AssemblySemVer) scattered through your team's templates. These aren't pipeline magic — they're variables set by the GitVersion tool, which reads your git history and calculates what version number this build should be.

Without it, you'd either hard-code versions (and forget to bump them) or invent your own branch-naming → versioning logic. GitVersion applies Semantic Versioning rules consistently based on your branches and git tags.

Semantic Versioning in 30 seconds

A SemVer version has up to four parts:

1MAJOR
.
3MINOR
.
0PATCH
-
beta.5pre-release label

Rules: bump MAJOR for breaking changes, MINOR for new features, PATCH for fixes. Pre-release labels (-beta.5, -rc.2) mark builds that are not yet stable. A build with a pre-release label sorts lower than the clean version — 1.3.0-rc.2 < 1.3.0.

The two tasks

GitVersion runs in two steps. Setup installs the tool onto the agent. Execute reads your git history and sets pipeline variables.

steps:
  - task: gitversion/setup@3.2.1
    displayName: Setup GitVersion
    inputs:
      versionSpec: '6.x'            # which GitVersion version to install

  - task: gitversion/execute@3.2.1
    displayName: Execute GitVersion
    inputs:
      useConfigFile: true
      configFilePath: GitVersion.yml # config at repo root (optional but recommended)

Everything after these two steps can use $(GitVersion.*) variables — they're injected into the pipeline variable scope by the execute task.

Critical: fetchDepth: 0 required. GitVersion needs the full git history to trace back to the most recent tag and count commits since then. A shallow clone (fetchDepth: 1) breaks GitVersion — it can't see the tags.

Your team uses fetchDepth: 1 for fast PR validation jobs, but build jobs that run GitVersion must set fetchDepth: 0.

jobs:
  - job: Build
    workspace:
      clean: all
    steps:
      - checkout: self
        fetchDepth: 0        # ← full history for GitVersion

      - task: gitversion/setup@3.2.1
        inputs:
          versionSpec: '6.x'

      - task: gitversion/execute@3.2.1
        inputs:
          useConfigFile: true
          configFilePath: GitVersion.yml

      # All subsequent steps can use $(GitVersion.SemVer) etc.

Branch → version mapping

GitVersion calculates the version based on the current branch name and the nearest tag in git history. Here's how your team's branch strategy maps to version output:

Branch Example $(GitVersion.SemVer) Meaning
feature/my-feat 1.3.0-my-feat.3 pre 3 commits into the feature branch
develop 1.3.0-beta.12 pre 12 commits since the last tag on develop
release/1.3.0 1.3.0-rc.4 pre 4th release candidate build
hotfix/fix-login 1.2.5-beta.1 pre Patch bump, pre-release until merged
master (after merge) 1.3.0 stable Clean stable version after release merge
Tag v1.3.0 on master 1.3.0 stable Exact tag match — the base for future builds

The commit count resets at a tag. When you push v1.3.0 to master, the next commit on develop becomes 1.4.0-beta.1 — GitVersion has seen the 1.3.0 tag and increments to the next planned version automatically.

Key variables

GitVersion sets many variables. These are the ones you'll encounter in your team's templates:

Variable Example on develop Example on master Use for
$(GitVersion.SemVer) 1.3.0-beta.12 1.3.0 Octopus release version, npm package version
$(GitVersion.MajorMinorPatch) 1.3.0 1.3.0 When you need the stable number without the pre-release label
$(GitVersion.AssemblySemVer) 1.3.0.0 1.3.0.0 .NET assembly versioning (four-part format)
$(GitVersion.NuGetVersionV2) 1.3.0-beta0012 1.3.0 NuGet packages (no dots in pre-release label)
$(GitVersion.InformationalVersion) 1.3.0-beta.12+Branch.develop.Sha.abc1234 1.3.0+Branch.master.Sha.def5678 Full audit version including git SHA

When you see the team's template passing ReleaseNumber: $(GitVersion.SemVer) to Octopus, you now know exactly what that string looks like — and why it changes depending on which branch triggered the pipeline.

GitVersion.yml — the configuration file

A GitVersion.yml at your repo root customises the versioning strategy. You don't need to memorise the full schema, but it's worth knowing the file exists and what it controls.

# GitVersion.yml — minimal example used with GitFlow strategy
mode: ContinuousDeployment
branches:
  develop:
    regex: develop
    tag: beta
    increment: Minor
  release:
    regex: release\/.*
    tag: rc
    increment: None
  hotfix:
    regex: hotfix\/.*
    tag: beta
    increment: Patch

The mode controls how GitVersion determines version increments. ContinuousDeployment is the most common for team workflows — every commit on develop produces a new pre-release version. You'll find your team's actual config in the root of any repo that uses GitVersion.

How it fits in a team build job

jobs:
  - job: Build
    pool:
      name: $(CILinuxAgentPool)
    workspace:
      clean: all
    steps:
      - checkout: self
        fetchDepth: 0                        # full history for GitVersion

      - task: gitversion/setup@3.2.1
        displayName: Setup GitVersion
        inputs:
          versionSpec: '6.x'

      - task: gitversion/execute@3.2.1
        displayName: Execute GitVersion
        inputs:
          useConfigFile: true
          configFilePath: GitVersion.yml

      - task: NodeTool@0
        inputs:
          versionSpec: 24.x

      - task: Npm@1
        displayName: npm ci
        inputs:
          command: ci
          customRegistry: useNpmrc
          customEndpoint: 'Equisoft.npm'

      - task: Npm@1
        displayName: npm run build
        inputs:
          command: custom
          customCommand: run build

      - task: CopyFiles@2
        inputs:
          SourceFolder: dist
          TargetFolder: $(Build.ArtifactStagingDirectory)

      - task: PublishPipelineArtifact@1
        inputs:
          targetPath: $(Build.ArtifactStagingDirectory)
          artifactName: drop-$(GitVersion.SemVer)   # ← version baked into artifact name
          publishLocation: pipeline

And downstream in the Octopus deploy stage:

- task: OctopusCreateRelease@3
  inputs:
    OctopusServerEndpoint: $(OctopusEndpoint)
    Project: my-app
    ReleaseNumber: $(GitVersion.SemVer)          # ← e.g. "1.3.0-beta.12" on develop
    Channel: Development
