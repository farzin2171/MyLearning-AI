Triggers & Conditions

Control when a pipeline runs and what runs based on the outcome.

Why this lesson

Knowing the YAML hierarchy (Lesson 1) tells you how a pipeline is structured. Triggers and conditions tell you when it fires and how it reacts to failures. In a team workflow, getting these wrong means either too many noisy builds — or PRs that deploy to production unchecked.

The three trigger types

trigger:
Fires on push to specified branches or when matching paths change.
pr:
Fires when a pull request targets a branch. Validates code before merge.
⚠ Azure Repos Git: has no effect — see below.
schedules:
Fires on a cron schedule, regardless of code changes.

1. CI trigger — push-based

The CI trigger fires every time code is pushed to a matching branch. Specify include or exclude lists for branches and paths. Schema reference → 

trigger:
  branches:
    include:
      - main
      - develop
      - feature/*    # wildcards supported
    exclude:
      - feature/wip-* # skip work-in-progress branches
  paths:
    include:
      - src/**         # only trigger when src/ changes
    exclude:
      - **/*.md        # doc-only changes won't run the pipeline
Path filters — monorepo pattern
If your repo has multiple apps, point paths.include at the relevant app folder. Changing apps/dashboard/** won't trigger the apps/api pipeline — clean, fast builds.

To disable the CI trigger entirely (manual or PR only): 

trigger: none

2. PR trigger — the Azure Repos gotcha

Critical — Azure Repos Git behaviour
If your repo is in Azure Repos Git, the YAML pr: keyword is silently ignored. It works for GitHub and Bitbucket Cloud, not Azure Repos.

For Azure Repos, PR build validation is configured through branch policies in the Azure DevOps UI: Repos → Branches → ⋯ → Branch policies → Build validation. Branch policies docs →

If you're on GitHub and have connected it to Azure Pipelines, the pr: keyword works as expected: 

# Only for GitHub / Bitbucket Cloud repos
pr:
  branches:
    include:
      - main
  paths:
    exclude:
      - docs/**

3. Scheduled trigger — cron

Use schedules: to run nightly builds, weekly dependency audits, etc. The cron syntax is: scheduled triggers docs →

0Minute
2Hour (UTC)
*Day of month
*Month
1-5Day of week

↑ This runs at 02:00 UTC, Monday–Friday (day-of-week 1=Mon, 5=Fri, 0=Sun)

schedules:
  - cron: '0 2 * * 1-5'     # 02:00 UTC, Mon–Fri
    displayName: Nightly build
    branches:
      include:
        - main
    always: true         # run even if no new commits
Note
always: true forces the schedule to run even when there are no new commits since the last build. Without it, Azure skips the run if nothing changed — usually what you want, but not for dependency audits.

Conditions — controlling what runs

Conditions apply to stages, jobs, or steps and decide whether that unit runs based on the outcome of what came before. Conditions docs →

Expression Runs when… Run if cancelled? Use case
succeeded() All dependencies succeeded No Default — normal happy-path steps
succeededOrFailed() Dependencies succeeded or failed No Cleanup, artifact upload — must run after failures but respects cancel
always() Always Yes Critical notifications — even on cancel
failed() At least one dependency failed No Failure-only alerts or rollback steps

The default is succeeded() — you don't need to write it explicitly on every step. You only need a condition when you want non-default behaviour.

Practical example — test report even on failure

A common pattern: always publish the test results file even if npm test fails, so you can see which tests failed in the Azure DevOps UI: 

steps:
  - script: npm ci

  - script: npm test -- --reporter=junit --output=results.xml
    displayName: 'Run tests'

  - task: PublishTestResults@2
    displayName: 'Publish test results'
    condition: succeededOrFailed()   # ← runs even if tests fail
    inputs:
      testResultsFormat: 'JUnit'
      testResultsFiles: 'results.xml'

Combining conditions with variables

You can write richer conditions using the expression language. A common one: only deploy from main: 

condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
Read-only built-in variables
Build.SourceBranch, Build.Reason, Build.BuildId are automatically set by Azure Pipelines on every run. You'll use them often in conditions and scripts. Full list →
