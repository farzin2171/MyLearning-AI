Lesson 11 of Azure CI/CD with Azure Pipelines — comparing Applications.Apply and Services.Authorization

Lean Pipelines and Pipeline Completion Triggers

Why two teams can achieve the same CI/CD outcome with very different YAML lengths — and how pipelines can trigger other pipelines

Two repos, two philosophies

Both Applications.Apply and Services.Authorization build, test, package, and deploy via Octopus. Yet their main pipeline files look nothing alike.

Applications.Apply / azure-pipelines.yml
~270
lines of YAML
  • 6 local template files
  • Inline stage/job definitions
  • Service containers declared here
  • Lifecycle hooks written out in full
  • Team owns the build + deploy logic
Services.Authorization / azure-pipelines.yml
~30
lines of YAML
  • No local templates
  • Single extends: call
  • Only parameters (names, IDs, flags)
  • Build + deploy logic lives in V2 template
  • Team only defines "what", not "how"

The Authorization pipeline isn't simpler — it does everything Apply does. It's just expressed entirely through extends: with parameters. This is the lean pipeline pattern: push complexity into the shared template, keep the app repo's YAML minimal.

The lean pipeline: complete file

Here is the entire main pipeline for Services.Authorization — unabridged:

resources:
  repositories:
    - repository: templates
      type: git
      name: PipelineTemplates

trigger:
  branches:
    include:
      - 'develop'
      - 'release/*'
      - 'hotfix/*'
  tags:
    include:
      - '*'

extends:
  template: /V2/build-dotnet-service-pipeline.yml@templates
  parameters:
    buildDocumentation: true
    documentationProjectName: 'Authorization'

    applicationBuildList:
      - name: 'AuthorizationService'
        projectName: 'Equisoft.AuthorizationService.csproj'
        packageId: 'microservices.authorization'
        packageTitle: 'Equisoft Authorization Microservice'
        packageDescription: 'The service responsible for the authorization...'
        packageType: 'publish'
        packageDocs: true
        octopusProjectList:
          - projectSpace: 'Spaces-222'
            projectGroup: 'Microservices (IIS)'
            projectName: 'Authorization Service (IIS)'
            deployRelease: true
          - projectSpace: 'Spaces-222'
            projectGroup: 'Microservices'
            projectName: 'Authorization Service'
            channel: 'Release'

That's it. GitVersion, CodeQL scanning, NuGet packaging, Octopus release creation, deployment — all handled by /V2/build-dotnet-service-pipeline.yml@templates. The app team provides the project-specific values; the template provides everything else.

The applicationBuildList parameter is an object list — the template iterates it with ${{ each item in parameters.applicationBuildList }} (the same ${{ each }} pattern from Lesson 4). Each entry can define multiple Octopus projects, flags like deployRelease: true, and packaging options — all without the app team needing to know how Octopus releases are created.

Explicit (Apply) vs lean (Authorization) — when to choose

Explicit (Apply-style)Lean extends-first (Auth-style)
Good whenUnique build steps, multiple front-ends, bespoke deploy targetsServices following the same standard pattern as a dozen other repos
MaintenanceTeam owns the logic — changes need PRs in this repoTemplate team updates V2 once; all services inherit the improvement
VisibilityEverything is explicit; easy to read and debug without contextRequires knowing what the template does; harder to debug template bugs
SecurityAnyone who can edit this repo can change the pipelineextends: enables required templates — org can enforce security gates

Most services that follow a standard pattern should use the lean approach. The Apply application is unusual: it has a React front-end, integration test service containers, Playwright smoke tests, and Docker publishing — all custom. A single shared template couldn't cover all of it without becoming unwieldy.

Pipeline completion triggers — a new kind of trigger

The Authorization repo's GHAS scan pipeline uses something you haven't seen before: a pipeline completion trigger. Instead of watching for commits, it fires when another pipeline run finishes.

Here are all three ways azure-pipelines-ghas-scan.yml can be triggered:

git tag pushed
→
ghas-scan.yml
trigger: tags
main pipeline completes
on release/* or hotfix/*
→
ghas-scan.yml
resources.pipelines — new!
3 AM every night (develop)
→
ghas-scan.yml
schedules: 
# From azure-pipelines-ghas-scan.yml

resources:
  pipelines:
    - pipeline: AuthorizationService          # a name for this reference
      source: Services\Services.Authorization # the OTHER pipeline's display name in Azure DevOps
      trigger:
        branches:
          include:
            - 'release/*'
            - 'hotfix/*'                      # only trigger when those branches build

trigger:
  tags:
    include:
      - '*'

schedules:
  - cron: "0 3 * * *"
    displayName: Daily Nightly scan
    branches:
      include:
        - develop
    always: true                              # run even if develop has no new commits

Why use a pipeline completion trigger here? The GHAS security scan is expensive and only makes sense after a real build artifact exists. Rather than integrating the scan into the main pipeline (which would slow every build), it runs after the main pipeline finishes on release branches. The two pipelines stay independent — the scan can be updated or rerun without rebuilding the app.

always: true on scheduled triggers

By default, a scheduled pipeline run is skipped if the target branch hasn't changed since the last run. always: true overrides that — the schedule fires regardless of whether there are new commits.

schedules:
  - cron: "0 3 * * *"
    displayName: Daily Nightly scan
    branches:
      include:
        - develop
    always: true          # ← without this, a quiet develop branch skips the run

For security scans, always: true is essential — new vulnerabilities are discovered in dependencies every day. If develop has no commits for a week, you still want a daily scan to catch newly-disclosed CVEs.

Don't use always: true on expensive pipelines you don't actually need to run daily. Full integration builds on every schedule even without changes wastes agent minutes. Reserve it for monitoring pipelines (security scans, dependency audits, uptime checks) where freshness matters more than efficiency.

How resources.pipelines works

KeyMeaning
pipeline:An alias for this reference (used elsewhere in your YAML)
source:The display name of the other pipeline, exactly as shown in Azure DevOps
trigger.branches.include:Only fire this pipeline when the other pipeline ran from these branches
trigger: trueShort form — fire on any branch of the source pipeline
trigger: noneNever auto-trigger; use it as a data source only (to download artifacts)

You can also use the pipeline resource to download artifacts from the triggering run — useful when a deploy pipeline needs the build artifact from the build pipeline that triggered it:

resources:
  pipelines:
    - pipeline: myBuild
      source: MyApp Build Pipeline
      trigger: true

steps:
  - download: myBuild      # downloads artifacts from the run that triggered this pipeline
    artifact: drop
