Variables & Variable Groups

How your team injects config, secrets, and branch-based logic — drawn from your PipelineTemplates repo.

Why this lesson

Look at the top of any pipeline in your team's V2/ folder: 

variables:
- group: Pipeline_Main
- group: pipeline-templates
- template: variables/resources.yml

Three variable sources — a variable group for shared org secrets, another group for template-specific config, and a template file for image names. You can't read or maintain these pipelines without understanding what each source does and when each expression syntax applies.

The three expression syntaxes

Azure Pipelines has three syntaxes for referencing variables. They look similar but evaluate at different times. Expressions docs →

Macro — runtime
$(variableName)
Expanded when the step runs on the agent. Works for most values: script args, task inputs, condition strings. Most common syntax.
Template expression — compile-time
${{ expression }}
Expanded when the pipeline is compiled, before it runs. Used for parameters, conditionals, and loops in templates. Only works with parameters and static values.
Runtime expression
$[expression]
Expanded at runtime but before a step runs. Used for assigning variables from other variables or conditions on stages/jobs.
The most common mix-up
${{ parameters.x }} works in template files because it's compile-time. In a non-template pipeline, you can't use ${{ variables.x }} — use $(x) instead. If a variable isn't resolving, check which syntax you're using against when it's evaluated.

Variable scopes — where you define them

Variables can be defined at four levels. Inner scopes override outer ones.

Pipeline
Visible everywhere. Use for values all stages share (agent pool name, build config).
Stage
Visible only within that stage. Your team puts branch-detection variables here.
Job
Visible within that job only. Often used for job-specific toggles.
Step
Set from a script using a logging command — becomes available to subsequent steps.

Variable groups — the team pattern

A variable group is a collection of name/value pairs defined in Pipelines → Library → Variable groups in the Azure DevOps UI. They let multiple pipelines share the same config without duplication. Variable groups docs →

Your team uses two groups in every pipeline: V2/build-dotnet-service-pipeline.yml

variables:
- group: Pipeline_Main     # CILinuxAgentPool, CIWindowsAgentPool, shared secrets
- group: pipeline-templates # dotnet-msbuild-linux, template-specific config
- template: variables/resources.yml  # dbSQLImage, rabbitmqImage (from the template repo)

The agent pool name $(CILinuxAgentPool) is never hardcoded — it comes from Pipeline_Main. If the team switches pools, they change it once in the library, not in every pipeline file.

Secret variables in groups

Mark a variable as secret in the UI (lock icon). Azure then:

  • Masks the value in all logs — it never appears in plain text
  • Prevents it from being passed between jobs automatically
  • Requires you to explicitly map it to an environment variable in scripts that need it
steps:
  - bash: echo "Deploying with token..."
    env:
      DEPLOY_TOKEN: $(mySecretToken) # explicitly mapped — secret won't auto-inject

Branch-detection variables — the stage pattern

Your team's pipelines compute branch flags once per stage using runtime expressions and then use them in conditions throughout: V2/build-dotnet-service-pipeline.yml L126–130 

stages:
  - stage: Publish
    variables:
      # $[...] = runtime expression — evaluated when stage starts
      isMaster:  $[eq(variables['Build.SourceBranch'], 'refs/heads/master')]
      isDevelop: $[eq(variables['Build.SourceBranch'], 'refs/heads/develop')]
      isRelease: $[contains(variables['Build.SourceBranch'], 'refs/heads/release')]
      isHotfix:  $[contains(variables['Build.SourceBranch'], 'refs/heads/hotfix')]
      isTag:     $[contains(variables['Build.SourceBranch'], 'refs/tags')]
    condition: and(succeeded('Build'),
                  or(eq(variables.isMaster, 'true'),
                     eq(variables.isDevelop, 'true'),
                     eq(variables.isRelease, 'true'),
                     eq(variables.isHotfix, 'true'),
                     eq(variables.isTag, 'true')))
Why 'true' in quotes?
Runtime expression variables are always strings, even boolean-looking ones. eq(variables.isMaster, 'true') — with quotes — is correct. Without quotes it won't match.

Output variables — passing data between jobs

Jobs run on different agents, so they can't share in-memory variables. The solution: write a value to stdout using a logging command, then read it in the next job via dependencies. Your team uses this in the GHAS pipeline to pass Jira ticket counts. scans/ghas/ghas-scan-reactjs-pipeline.yml L104–120

Step 1 — write the output variable

- task: PowerShell@2
  name: outputJiraList          # step name — required to reference output
  inputs:
    targetType: inline
    script: |
      # isOutput=true makes it visible to other jobs
      Write-Host "##vso[task.setvariable variable=jiraIssueCount;isOutput=true]42"

Step 2 — read it in a downstream job

- job: SendSlackNotification
  dependsOn: CreateJiraTickets
  variables:
    # dependencies.JobName.outputs['stepName.variableName']
    ticketsCount: $[ dependencies.CreateJiraTickets.outputs['outputJiraList.jiraIssueCount'] ]
  steps:
    - script: echo "$(ticketsCount) tickets found"
Logging command
##vso[task.setvariable variable=X]value — the ##vso prefix is what Azure Pipelines intercepts. Works in bash, PowerShell, and cmd scripts. isOutput=true is required for cross-job visibility. Logging commands →

Predefined variables your team uses

# Used in your team's templates:
$(Build.SourceBranch)          # refs/heads/master, refs/heads/develop, refs/tags/v1.0.0
$(Build.BuildNumber)           # used as Octopus release number
$(Build.Repository.Name)       # used in Slack notification messages
$(Build.Reason)                # 'PullRequest', 'Manual', 'IndividualCI', etc.
$(System.PullRequest.PullRequestId)  # used in commitlint PR pipeline
$(System.AccessToken)          # pipeline's OAuth token — used in Azure DevOps REST calls
$(System.DefaultWorkingDirectory)    # root of checked-out code

# Set by GitVersion task after it runs:
$(GitVersion.SemVer)           # e.g. 1.4.2-alpha.5
$(GitVersion.AssemblySemVer)   # e.g. 1.4.2.0
$(GitVersion.InformationalVersion)  # full string with commit hash

Variables in scripts — the Octopus channel pattern

Your team sets an Octopus Deploy channel based on the source branch using a PowerShell script that writes back to the pipeline via a logging command. This is the output variable pattern in real production use: V2/build-dotnet-service-pipeline.yml L243–259 

- powershell: |
    # Decide which Octopus channel based on branch
    $branch = "$(Build.SourceBranch)"
    $channel = "Development"

    if ($branch.StartsWith("refs/heads/release/") -or
        $branch.StartsWith("refs/heads/hotfix/")) {
        $channel = "Testing"
    } elseif ($branch -eq "refs/heads/master" -or
              $branch.StartsWith("refs/tags/")) {
        $channel = "Release"
    }

    # Write back to pipeline — available as $(channel) in subsequent steps
    Write-Host "##vso[task.setvariable variable=channel]$channel"
  displayName: 'Set Octopus channel'

After this step runs, any subsequent step in the same job can reference $(channel). Because isOutput=true is not set, it's local to this job only.
