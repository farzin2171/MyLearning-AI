Lesson 5 of Azure CI/CD with Azure Pipelines

Pipeline Artifacts

Passing build output between stages so build and publish never share an agent

The problem artifacts solve

Each stage in an Azure Pipeline can run on a different agent — and by default it does. The build agent compiles your TypeScript, runs tests, and produces a dist/ folder. Then it's done. The publish stage starts on a brand-new agent that has never seen your source tree.

Artifacts are how you hand the dist/ folder from one stage to the next. Azure Pipelines stores them in a managed artifact store attached to the run, outside any agent's filesystem.

Publishing an artifact

Use PublishPipelineArtifact@1 at the end of your build job.

- task: PublishPipelineArtifact@1
  displayName: Publish drop
  inputs:
    targetPath: $(Build.ArtifactStagingDirectory)   # what to upload
    artifactName: drop                               # the name other stages use to reference it
    publishLocation: pipeline                        # always 'pipeline' (not 'filepath')
InputMeaning
targetPath Folder or file to upload. Anything not under this path is not included.
artifactName The name you'll use in download: downstream. Defaults to drop if omitted.
publishLocation Always use pipeline. The legacy filepath mode requires a file share.

Convention: copy files into $(Build.ArtifactStagingDirectory) before publishing. This is a dedicated staging folder that's always clean at the start of each build. Never publish directly from $(Build.SourcesDirectory) — you'd upload your entire repo.

A typical Node build job ends like this:

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: 24.x

  - task: Npm@1
    displayName: npm ci
    inputs:
      command: ci

  - task: Npm@1
    displayName: npm run build
    inputs:
      command: custom
      customCommand: run build

  - task: CopyFiles@2
    displayName: Stage build output
    inputs:
      SourceFolder: dist
      TargetFolder: $(Build.ArtifactStagingDirectory)

  - task: PublishPipelineArtifact@1
    displayName: Publish drop
    inputs:
      targetPath: $(Build.ArtifactStagingDirectory)
      artifactName: drop
      publishLocation: pipeline

Downloading an artifact

In a downstream stage, use either the task form or the shorthand:

Shorthand (preferred)

steps:
  - download: current        # 'current' = this pipeline run
    artifact: drop           # matches the artifactName above
    # Downloaded to: $(Pipeline.Workspace)/drop/

Task form (when you need more control)

- task: DownloadPipelineArtifact@2
  inputs:
    source: current
    artifact: drop
    targetPath: $(Build.ArtifactStagingDirectory)

Default download path: The shorthand puts files at $(Pipeline.Workspace)/<artifactName>/. If you used artifactName: drop, look in $(Pipeline.Workspace)/drop/.

The task form lets you redirect to $(Build.ArtifactStagingDirectory) — the same path your publish stage expects — which keeps scripts consistent.

download: none — skipping the auto-download

Deployment jobs (job: blocks under a deployment: key) automatically download all artifacts at startup. This is convenient but wasteful if the job only triggers a remote action (e.g., creating an Octopus Deploy release).

jobs:
  - deployment: PublishToOctopus
    displayName: Create Octopus release
    environment: Production
    strategy:
      runOnce:
        deploy:
          steps:
            - download: none     # ← skip the auto-download; we don't need the files here
            - task: OctopusCreateRelease@3
              ...

Regular jobs do not auto-download — they start with an empty workspace. Only deployment jobs have the auto-download behaviour. If you're in a regular job and need an artifact, always add an explicit download: step.

Team template pattern

In your team's PipelineTemplates repo, the Build stage ends with a publish step, and the subsequent Publish/Deploy stages set checkout: none (no source needed) and use download: current. Here's a condensed example matching that style:

stages:
  - stage: Build
    condition: and(succeeded(), or(eq(variables.isDevelop, true), eq(variables.isMaster, true)))
    jobs:
      - job: BuildApp
        pool:
          name: $(CILinuxAgentPool)
        workspace:
          clean: all
        steps:
          - task: NodeTool@0
            inputs: { versionSpec: 24.x }

          - task: Npm@1
            inputs: { command: ci, customRegistry: useNpmrc, customEndpoint: 'Equisoft.npm' }

          - task: Npm@1
            inputs: { command: custom, customCommand: run build }

          - task: CopyFiles@2
            inputs:
              SourceFolder: dist
              TargetFolder: $(Build.ArtifactStagingDirectory)

          - task: PublishPipelineArtifact@1
            inputs:
              targetPath: $(Build.ArtifactStagingDirectory)
              artifactName: drop
              publishLocation: pipeline

  - stage: Publish
    dependsOn: Build
    condition: and(succeeded('Build'), eq(variables.isDevelop, true))
    jobs:
      - deployment: CreateOctopusRelease
        environment: Development
        strategy:
          runOnce:
            deploy:
              steps:
                - checkout: none             # ← no source code needed
                - download: current
                  artifact: drop             # ← grab what Build published
                - task: OctopusCreateRelease@3
                  inputs:
                    OctopusServerEndpoint: ...
                    Space: Default
                    Project: my-app
                    Packages: |
                      MyApp.Web:$(GitVersion.SemVer)

Notice: the Publish stage does two things in order — checkout: none to skip the source clone, then download: current to get the artifact. This keeps the publish job fast: it only touches the pre-built package, never the source tree.
