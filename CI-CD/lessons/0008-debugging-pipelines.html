Lesson 8 of Azure CI/CD with Azure Pipelines

Debugging Failing Pipelines

Where to look, what the error means, and how to fix it fast

The three-click rule

Any pipeline failure can be reached in three clicks: run overview → job → step. Before reading anything else, narrow down where the failure is. Everything else is just reading carefully.

✓Build
✗Publish
—Deploy

The run overview shows every stage at a glance. Click the red stage → click the red job → click the red step → read the log.

Diagnostic decision tree

  • Pipeline didn't start at all?
    • YAML parse error — open the pipeline in the editor; the "Issues" tab shows line numbers
    • Trigger didn't fire — check trigger config in YAML; did you push to the right branch?
  • Stage shows "Skipped"?
    • Stage condition evaluated to false — add a diagnostic echo step; see section below
  • Stage shows "Waiting"?
    • Approval gate pending — go to Pipelines → Environments → approve it
  • Job failed with an error?
    • First red step — open that step's log; read from the bottom upward
    • "Task not found" — extension not installed in your org
    • "exited with code N" — your script failed; look at lines printed before this message
    • "Template not found" — wrong path, wrong @alias, or repo resource not authorised
    • "Resource not authorized" — service connection or agent pool needs pipeline permission

Reading the log: what the ##[prefix] means

Every line in an Azure Pipelines log is tagged with a severity prefix. Knowing these tells you what to ignore and what to focus on.

##[section]Starting: npm run build
##[command]/bin/bash -e /tmp/azagent/_work/_temp/build.sh
> my-app@1.3.0-beta.12 build
> tsc --noEmit && vite build
##[warning]Found 2 deprecated API usages. Build will succeed.
##[error]src/App.tsx(42,5): error TS2345: Argument of type 'string' is not assignable to parameter of type 'number'.
##[error]TypeScript compilation failed. Exit code: 2
##[section]Finishing: npm run build
Prefix Colour Meaning
##[section]tealStep started / finished — structural marker, not an error
##[command]blueExact command the agent ran — useful for reproducing locally
##[warning]yellowNon-fatal issue — step still succeeds unless failOnStderr: true
##[error]redFatal — step will fail; these are the lines you actually need to read
##[debug]greyOnly visible when System.Debug is true — very verbose internal detail

Read error logs bottom-up. The root cause is usually the first ##[error] line. Everything after it is cascading fallout. Scroll to the first red line, not the last.

The five most common errors

1 — Script exited with non-zero code
##[error]Bash exited with code '1'.

Your shell script failed. The ##[error] line is just the wrapper — the actual error is in the lines above it. Search upward for the real message (a compiler error, a missing file, a failed HTTP call).

Quick check: Can you reproduce it by running the same command locally with the same Node version?

2 — Task not found
##[error]Task 'gitversion/setup' not found. The pipeline references a task that does not exist.

The Azure DevOps extension that provides this task is not installed in your organisation. Go to Organisation Settings → Extensions → Browse marketplace and install it.

Also happens with a typo in the task name or a wrong major version (e.g. Npm@2 instead of Npm@1).

3 — Template or repository not found
##[error]Unable to resolve template: /CI/build-pipeline.yml@templates. Verify that the referenced template exists and that the pipeline has access to the repository.

One of three causes: (a) the path in template: is wrong — check spelling and the leading /; (b) the @alias doesn't match the name in resources.repositories; or (c) the pipeline hasn't been granted access to the PipelineTemplates repo — go to Project Settings → Repositories → [repo] → Security and allow the pipeline.

4 — Resource not authorized
##[error]Resource not authorized. The pipeline is not permitted to use this service connection.

The service connection (e.g. the Octopus endpoint, the npm registry endpoint) hasn't been granted permission to this specific pipeline. Go to Project Settings → Service connections → [connection] → Security and add the pipeline, or tick "Grant access permission to all pipelines".

5 — Variable is empty or undefined
##[warning]Input required: versionSpec
or: release number is "" — Octopus rejects empty strings

The variable that should hold a value is empty at runtime. Common causes: (a) a ##vso[task.setvariable] step didn't run or used a wrong scope; (b) GitVersion didn't run (often a fetchDepth: 1 shallow clone problem); (c) the variable group isn't linked to this pipeline.

Quick check: Add a script: echo "var=$(MyVar)" step immediately before the failing step to confirm what value it actually sees.

Enabling diagnostic logging

When the log doesn't show enough detail, turn on System.Debug. This floods the log with internal agent activity — task resolution, variable expansion, HTTP calls — but it's invaluable for tricky failures.

Option A — for a single run: when queuing a run manually, click Variables and add System.Debug = true. It applies to that run only.

Option B — in YAML (temporarily):

variables:
  - group: Pipeline_Main
  - name: System.Debug
    value: true          # ← remove when you're done debugging

Don't leave System.Debug: true permanently — it makes logs enormous and can expose sensitive variable values in plain text. Add it, reproduce the failure, then remove it.

Debugging a "Skipped" stage

A stage that shows Skipped means its condition: evaluated to false at runtime. The challenge: conditions run silently — the UI just says "skipped", not "skipped because isDevelop was false".

Step 1 — print the condition variables before the failing stage:

- stage: Diagnostics
  condition: always()           # always runs, gives you the values
  jobs:
    - job: PrintVars
      steps:
        - script: |
            echo "Build.SourceBranch:     $(Build.SourceBranch)"
            echo "Build.SourceBranchName: $(Build.SourceBranchName)"
            echo "isDevelop:              $(isDevelop)"
            echo "isMaster:               $(isMaster)"
            echo "isRelease:              $(isRelease)"
          displayName: Print branch variables

Run the pipeline and check what those variables actually contain. A common gotcha: the branch runtime expression sets isDevelop using a startsWith check on refs/heads/develop, and the actual branch name doesn't match that pattern exactly.

Step 2 — temporarily remove the condition to confirm the stage itself works, then re-add it once you know what value to check against.

Remember: runtime expressions (${{ }} compile-time vs $[] runtime vs $() macro) evaluate at different times. A $[ condition that references a variable not yet set will silently evaluate to empty string — which is falsy. See Lesson 3 for the expression timing rules.

Downloading the full log

The UI truncates long logs. For anything involving npm output, compiler errors, or verbose test runners, click Download logs (top-right of the job view) to get a zip of all step logs as plain text files — searchable in your editor.

Also useful: the Raw log link in each step expands the full text without pagination.
