Lesson 9 of Azure CI/CD with Azure Pipelines — from Applications.Apply

Service Containers

Running SQL Server and Redis alongside your build job — no external infrastructure needed

The problem: integration tests need real services

The Applications.Apply build job runs integration tests against a real SQL Server database and a real Redis cache. Without containers, you'd need either a shared test database (fragile, slow, causes flaky tests) or mocks (which don't catch SQL dialect bugs or Redis TTL edge cases).

Service containers solve this: Azure Pipelines spins up SQL Server and Redis as Docker containers alongside your build job, on the same agent network. The containers start before your first step runs and are torn down when the job finishes. Your tests connect to them as if they were local services.

The job container (net10) runs your steps. The sidecar containers (db, redis) are network-reachable services. The alias you give a sidecar (db, redis) becomes its hostname inside the job container.

Step 1 — define containers as pipeline resources

Containers are declared in the top-level resources.containers block, just like repositories. Each entry needs a name (alias), an image, and optionally ports and environment variables.

# From azure-pipelines.yml in Applications.Apply

resources:
  repositories:
    - repository: templates
      type: git
      name: PipelineTemplates

  containers:
    - container: net10                            # alias — used to reference this container
      image: "mcr.microsoft.com/dotnet/sdk:10.0"

    - container: db
      image: "mcr.microsoft.com/mssql/server:2022-latest"
      hostname: db                                # hostname other containers use to reach it
      ports:
        - '1433:1433'                             # host:container port mapping
      env:
        MSSQL_SA_PASSWORD: '$(sa_password)'       # secret from variable group
        ACCEPT_EULA: "Y"                          # required by SQL Server image

    - container: redis
      hostname: redis
      image: redis
      ports:
        - '6379:6379'

Secrets in container env:: Pass sensitive values like MSSQL_SA_PASSWORD via pipeline variables ($(sa_password) from the variable group), never hard-coded. The value is masked in logs automatically.

Step 2 — attach containers to the job

Inside the job, two keys control which containers are attached:

  • container: — the job container; your steps run inside this image
  • services: — sidecar containers; they run alongside and are reachable by hostname
# From build-app.yml in Applications.Apply

jobs:
  - job: BuildApp
    container: net10        # steps execute inside the .NET SDK container
    services:
      db: db                # left = hostname the job uses, right = resource alias from above
      redis: redis
    steps:
      - checkout: self

      # Integration tests connect using "db" as the server hostname
      # e.g. "Server=db,1433;Database=Apply;User=sa;Password=$(sa_password)"
      - task: DotNetCoreCLI@2
        displayName: "Tests"
        env:
          ASPNETCORE_ENVIRONMENT: Integration    # picks up appsettings.Integration.json
        inputs:
          command: test
          arguments: '--no-build --configuration Release --collect "XPlat Code Coverage"'

ASPNETCORE_ENVIRONMENT: Integration is the connection between the container and the test configuration. The test project's appsettings.Integration.json file contains a connection string pointing to Server=db,1433. The hostname matches the sidecar alias.

Using services without a job container

You don't have to run your steps inside a container. services: works on its own — the steps run directly on the agent, but the sidecars are still reachable:

jobs:
  - job: IntegrationTests
    # no container: key — steps run on the host agent
    services:
      db: db         # SQL Server still starts as a sidecar
      redis: redis   # Redis still starts as a sidecar
    steps:
      - script: dotnet test ...

The Applications.Apply build uses a job container (net10) because it needs a specific .NET SDK version that may not be installed on the host agent. For teams running on self-hosted agents with the right SDK, the job container can be omitted.
