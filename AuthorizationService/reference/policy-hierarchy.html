Policy / Role / Permission — Quick Reference

See Lesson 04 for full walkthrough.

Access Key Format

Root.Child.Grandchild.PermissionKey (permission inside a nested policy) Root.Child.RoleKey (role inside a child policy) Each segment = the Key field of a Policy, Role, or Permission entity.

How Keys Are Built ([dbo].[GetUserAccessKeys])

Step 1 — Recursive CTE builds policy identifiers: root policy Key = "platform" → Identifier = "platform" child policy Key = "reports" → Identifier = "platform.reports" grandchild Key = "export" → Identifier = "platform.reports.export" Step 2 — Final SELECT joins identifiers to mapped permissions/roles: Identifier + "." + Permission.Key → "platform.reports.export.run" Identifier + "." + Role.Key → "platform.admin.administrator"

User → Access Chain

JWT (external identity) → ExternalRolesProvider → role name strings → tblExternalRole (ExternalSourceId matches name) → tblExternalRoleToPermission → tblPolicyPermission → tblPolicy → tblExternalRoleToRole → tblPolicyRole → tblPolicy → [dbo].[GetUserAccessKeys] (recursive CTE + JOIN) → List<string> of dot-separated access keys → cached in Redis under authz.eval.{userId}

Delete Cascade

Deleting a policy cascades through: PolicyPermissionMapping → ExternalRoleToPermissionMapping, PermissionGroupToPermissionMapping
PolicyRoleMapping → ExternalRoleToRoleMapping
Then all child policies recursively.
Effect on users: cached entries remain until TTL expires. Force invalidation with DELETE /cache/permissions/{userId}.
