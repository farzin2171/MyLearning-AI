What Is the Authorization Service?

The big picture: what problem it solves, how it is structured, and the four core building blocks.

The Problem It Solves

Every microservice on the platform needs to answer one question before it does any work: "Is this caller allowed to do this?" Without a central answer, each service would have to maintain its own copy of permission rules — a recipe for drift, inconsistency, and security bugs.

The Authorization Service is the single source of truth for that question. Every other service that needs to make an access decision calls it first.

The core contract

Call GET /api/v1/authorize?key={accessKey} and get back "Allow" or "Deny". That's it. The complexity of policies, roles, external providers, and caching is hidden behind this simple answer.

The Four Building Blocks

Authorization decisions are the result of combining four concepts:

1. Permission

The most granular unit. A Permission is a named capability: something like policy.read, client.create, or report.export. Each permission has a unique Key within a tenant. Permissions are what access keys ultimately resolve to.

2. Role

A Role is a named group of permissions (and sometimes other roles). Rather than assigning dozens of permissions to a user directly, you assign them a role: underwriter, admin, readonly-agent. This is the "internal role" — distinct from external roles (see below).

3. Policy

A Policy is the assignment vehicle. It connects who to what. A policy bundles Roles and Permissions together, and is assigned to users. Crucially, policies are hierarchical: a child policy inherits everything from its parent. This lets you build a tree of access — product-level → tenant-level → feature-level — without repeating rules.

Policy Tree (example) [Root Policy: BasicAccess] ├── permission: dashboard.view └── [Child Policy: UnderwriterAccess] ├── role: underwriter └── permission: policy.submit A user assigned UnderwriterAccess gets dashboard.view + underwriter role + policy.submit

4. External Role

The platform does not control all identity systems. Users may have roles defined in Azure AD, OIPA, a custom API, or via JWT claims. An External Role maps an identity from one of these external providers into the internal Role/Permission system.

At authorization time, the service queries the configured external providers for the user, translates those external roles into internal ones, and evaluates the whole picture together.

Why External Roles matter

A user might never appear in the service's own database. Their identity lives in Azure AD. External roles are the bridge — without them, those users would always be denied.

The Four-Layer Architecture

The service is built on clean architecture with four distinct layers:

┌─────────────────────────────────────────────────┐ │ Equisoft.AuthorizationService (Web API) │ │ Controllers · Filters · DI modules │ ├─────────────────────────────────────────────────┤ │ Equisoft.AuthorizationService.Services │ │ Business logic · Factory pattern · Providers │ ├─────────────────────────────────────────────────┤ │ Equisoft.AuthorizationService.Data │ │ EF Core DbContext · Repositories · Dapper SPs │ ├─────────────────────────────────────────────────┤ │ Equisoft.AuthorizationService.Domain │ │ Domain models · Cache manager · Constants │ └─────────────────────────────────────────────────┘

Each layer depends only on layers below it. The Web API layer never talks to the database directly.

The Two Primary Endpoints

Everyday consumers of the service use exactly two endpoints:

Endpoint Returns Use case
GET /authorize?key=X Allow / Deny Guard a specific feature (one key = one permission check)
GET /evaluate string[] of access keys Get the full permission set for a user (to populate a UI)
Management endpoints are service-account only

POST /management/policies, /management/roles, etc. require a service account token. Regular user JWTs are rejected by ServiceAccountAuthorizeFilter.

Quick Reference: Key Terms

Permission
A named, atomic capability. The finest grain of access control.
Role
A named bundle of permissions (internal to this service).
Policy
Assigns roles/permissions to users. Forms a hierarchy (parent → child inheritance).
External Role
A role from an outside provider (Azure AD, OIPA, custom API, claims) mapped into internal roles/permissions.
External Role Provider
A plugin (one of five types: CustomApi, Oipa, Claims, Assistant, MicrosoftGraph) that retrieves external roles for a user.
Access Key
A string identifier for a specific permission, used as the lookup key in /authorize.
TenantId
All entities (policies, roles, permissions) are scoped to a tenant. The service is multi-tenant by design.
