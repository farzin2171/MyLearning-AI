Lesson 04 · Authorization Service · Data Model

Policy, Role & Permission Hierarchy

What access keys actually look like, how the policy tree produces them, and how to predict a user's access from first principles.

You know from Lesson 02 that GET /authorize?key=X checks whether the string X is in a cached list. You know from Lesson 03 how external roles feed into that list. This lesson answers the question you haven't fully answered yet: what do those key strings actually look like, and how are they generated?

Access Keys Are Paths, Not Names

An access key is not just a permission name like "view". It is a dot-separated path encoding the full location of a permission in the policy tree:

The access key format

RootPolicy.ChildPolicy.GrandchildPolicy.PermissionKey
RootPolicy.ChildPolicy.RoleKey

Every segment is the Key field of a Policy, or the Key of a Permission/Role. The hierarchy is encoded in the string itself.

This is why a caller can ask for "platform.reports.export.run" — not just "run". The path makes access keys globally unique across the entire tree, even if two policies each have a permission named "view".

Worked Example: Reading a Policy Tree

Consider this policy tree. Each indented level is a child policy.

platform (root policy key: "platform") ├── Permission: "login" ├── reports (child policy key: "reports") │ ├── Permission: "view" │ └── export (grandchild policy key: "export") │ └── Permission: "run" └── admin (child policy key: "admin") ├── Role: "administrator" └── Permission: "user.manage"

Now suppose an external role Underwriter is mapped to:

  • The login permission inside platform
  • The view permission inside platform.reports

The stored procedure builds these access keys for the Underwriter role:

Access keys returned by [dbo].[GetUserAccessKeys]
platform.login platform.reports.view

A call to /authorize?key=platform.reports.export.run for an Underwriter user would return Deny — that path is not in the list, because the Underwriter role is not mapped to the export permission inside platform.reports.export.

How the Stored Procedure Builds Paths: The Recursive CTE

The stored procedure [dbo].[GetUserAccessKeys] uses a SQL recursive CTE to flatten the entire policy tree into identifiers before joining to permissions. 

-- Step 1: traverse the entire tree, building dot-separated identifiers
WITH cte_data AS (
    -- anchor: root policies (ParentPolicyId IS NULL)
    SELECT PolicyId, CAST([Key] AS NVARCHAR(MAX)) AS Identifier
    FROM tblPolicy WHERE ParentPolicyId IS NULL

    UNION ALL

    -- recursive: each child appends its Key to its parent's Identifier
    SELECT Child.PolicyId,
           CAST(Parent.Identifier + '.' + Child.[Key] AS NVARCHAR(MAX))
    FROM tblPolicy AS Child
    INNER JOIN cte_data AS Parent ON Child.ParentPolicyId = Parent.PolicyId
)
-- Step 2: for each external role, find its PolicyPermission/PolicyRole records
-- and append the Permission/Role Key to form the final access key string

The result of Step 1 for the example tree above would be:

PolicyId | Identifier ---------+---------------------------- 1 | platform 2 | platform.reports 3 | platform.reports.export 4 | platform.admin

Step 2 then joins those identifiers to the permission/role records the external role is linked to, appending .PermissionKey or .RoleKey to produce the final strings.

Only the Policy table is hierarchical

Roles and Permissions are flat lists. The tree structure lives entirely in tblPolicy via the self-referential ParentPolicyId foreign key. The hierarchy is resolved at query time by the recursive CTE — it is not stored pre-computed.

How a User Gets Assigned to a Policy

There is no direct "user → policy" table. The chain is always:

External role name (from provider) → tblExternalRole (ExternalSourceId matches the name) → tblExternalRoleToPermission → tblPolicyPermission → tblPolicy → tblExternalRoleToRole → tblPolicyRole → tblPolicy

When you say "a user is assigned to the reports policy", what you really mean is: the user has an external role that is mapped (via junction tables) to one or more PolicyPermission or PolicyRole records that belong to the reports policy.

This is configured via the management API with a service account: 

// POST /api/v1/management/policies
{
  "policies": [
    {
      "key": "platform",
      "permissions": ["login"],
      "policies": [
        {
          "key": "reports",
          "permissions": ["view"],
          "policies": [
            { "key": "export", "permissions": ["run"] }
          ]
        }
      ]
    }
  ]
}

The repository creates this entire tree recursively in one transaction, handling junction table entries automatically.

Practice: Predict the Access Keys

These exercises build the skill of reading a policy tree and predicting access keys. Work out the answer before revealing it.

Exercise 1

Given this policy tree:

crm (root) ├── Permission: "contact.view" └── premium (child) ├── Permission: "contact.edit" └── Permission: "contact.delete"

External role SalesRep is mapped to: the contact.view permission in crm, and the contact.edit permission in crm.premium.

What access keys does a SalesRep user get?

crm.contact.view — the contact.view permission lives in the crm root policy.
crm.premium.contact.edit — the contact.edit permission lives in the premium child policy (path: crm → premium).

crm.premium.contact.delete is not returned — SalesRep is not mapped to that permission.

Exercise 2

Given this policy tree:

insurance (root) └── underwriting (child) └── life (grandchild) ├── Role: "actuary" └── Permission: "quote.approve"

External role Actuary is mapped to: the actuary role and the quote.approve permission, both inside insurance.underwriting.life.

What access keys does an Actuary user get?

insurance.underwriting.life.actuary — the actuary role lives at the grandchild level; full path is prefixed.
insurance.underwriting.life.quote.approve — the quote.approve permission also at grandchild level.

Note how deep nesting makes the keys long — and how the caller must use the full string insurance.underwriting.life.quote.approve, not just quote.approve.

Exercise 3 — deletion edge case

Given this policy tree:

app (root) ├── Permission: "settings.view" └── admin (child) └── Permission: "settings.edit"

External role Admin is mapped to the settings.edit permission in app.admin. A developer now deletes the admin child policy via the management API.

After deletion, what access keys does an Admin user get?

None. When a policy is deleted, the repository cascades: it deletes the PolicyPermissionMapping first, then the ExternalRoleToPermissionMapping pointing to it. The Admin external role now has no mappings left. The user gets an empty list — Deny on everything.

Additionally, any cached results for Admin users remain valid until TTL expires, so the effect may be delayed by the Redis TTL. Use DELETE /api/v1/cache/permissions/{userId} to force immediate effect.

Retrieval practice — from memory first

2. Two policies in the same tenant both have a permission with Key = "view". What prevents these from colliding as access keys?

3. An external role is mapped to a PolicyPermission that belongs to a child policy. The recursive CTE also generates an identifier for the root policy. Does the external role automatically receive the root policy's permissions too?

4. A PermissionGroup contains three PolicyPermission entries. An assistant acting OnBehalfOf a user has this group assigned. The user has 10 access keys. How many keys can the assistant act with?
