The Authorization Request Flow

What actually happens — step by step — when another service asks "is this caller allowed?"

In Lesson 01 you learned the four building blocks (Permission, Role, Policy, External Role) and the two primary endpoints. Now we trace a real GET /authorize?key=some.access.key request through the entire codebase, from JWT to JSON response.

Step by Step

1

Controller reads the identity context

AuthorizeController.GetAsync() receives the validated JWT. The DIT framework middleware has already parsed it into an IIdentityContext injected into the controller. The controller copies the relevant fields into an AuthorizeParameters object: OwnerId, ExternalOwnerId, TenantId, Claims, and the requested Key.

The Key is the single access key being tested — e.g. "policy.submit".

2

Factory selects the right authorize service

AuthorizeServiceFactory.Get() inspects the identity context and picks one of four IAuthorizeService implementations, checked in this exact order:

ContextDetection conditionCache key format
OnBehalfOf _identityContext.IsOnBehalfOf == true authz.eval.{ExternalOwnerId}_{OnBehalfOfExternalId}
Guest Claims contain service_isGuestAccount=true authz.eval.guest (fixed)
ServiceAccount IdentityType == IdentityType.Service authz.eval.{role_name}
User Default (none of the above) authz.eval.{OwnerId}

Each implementation is registered with .NET 8's keyed services and retrieved via GetRequiredKeyedService<IAuthorizeService>(nameof(ServiceFactoryType.User)).

3

Cache check — the fast path

AuthorizeCacheService.HasPermissionAsync() asks the cache manager for a List<string> stored at the context-specific cache key. The cache manager first checks Redis; if Redis is down (circuit breaker open), it falls back to the in-memory cache.

Cache Hit If a list exists, the method simply checks keysSet.Contains(authorizeParameters.Key) and returns true (Allow) or false (Deny) immediately. No database access. No external calls.

Cache Miss If the list is absent (null), the evaluation pipeline is triggered.

4

Evaluation: resolve external roles

Slow path EvaluateServiceFactory.Get() mirrors the authorize factory — same four contexts. The evaluate service's job is to compute the complete list of access keys for this identity and write it to the cache.

For a regular user: ExternalRolesService.GetRolesForUserAsync() queries every configured External Role Provider in priority order and returns the user's external role names (e.g. ["underwriter", "BasicAccess"]).

For a service account: the role is read directly from the service_role claim — no external provider call needed.

For a guest: two hardcoded role names are used: "Guest" and "Guest{TenantId}".

5

Stored procedure: flatten roles to access keys

AccessRepository.GetFlattenAccessKeysAsync() passes the role names to SQL Server via Dapper: 

exec [dbo].[GetUserAccessKeys] @TenantId, @Roles
-- @Roles is a comma-separated string: "underwriter,BasicAccess"
-- Returns: rows with an Identifier column (the access key strings)

This stored procedure walks the Policy → Role → Permission hierarchy in SQL and returns the flat list of all permission keys the given roles grant within the tenant. Dapper maps the rows to a List<string> of identifiers.

6

Cache write — the result is stored

EvaluateCacheService writes the computed key list to Redis with a TTL (configurable via persistence.redis.ExpiryTimeSpanInSecs). It also caches two ID mappings:

  • authz.external.{ExternalOwnerId} → internal OwnerId
  • authz.external.{OnBehalfOfExternalId} → OnBehalfOf internal ID (if applicable)

The write uses a distributed lock (key + "_lock") in Redis to prevent two concurrent requests for the same user from each triggering a full evaluation.

7

Final decision

Back in AuthorizeCacheService, the freshly cached list is retrieved and the same Contains(key) check runs. The boolean result becomes AuthorizeResult.Allow or AuthorizeResult.Deny, wrapped in an AuthorizeResponseViewModel and returned as HTTP 200. 

{ "result": "Allow" }   // or "Deny"

The Resilience Layer: Circuit Breaker

The AuthorizationCacheManager wraps every Redis call in a Polly circuit breaker. If Redis calls fail repeatedly, the breaker opens and all further requests immediately fall back to an in-memory cache — avoiding a flood of failing Redis calls.

Consistency implication

When Redis is healthy, all instances of the service share one cache. When the circuit breaks, each instance has its own in-memory cache. Cache invalidation (e.g. after a permission change) won't propagate to in-memory instances. This means a permission change may not take effect immediately during a Redis outage.

The Special Case: OnBehalfOf

The OnBehalfOf context is the most complex. An example: an AI assistant acting for a human user. The evaluation service runs two external role lookups and an intersection:

  1. Resolve external roles for the impersonating user (e.g. the assistant).
  2. Check that the impersonating user has the special permission "authorization.use.onBehalfOf". If not — Deny immediately.
  3. Resolve external roles for the target user (the human being acted for).
  4. Flatten the target user's roles to access keys (stored proc call).
  5. If the impersonating identity is an assistant, intersect the target user's keys with the assistant's permission groups (a second stored proc: GetUserAccessKeysFromPermissionGroups).
  6. Cache the intersected result under a composite cache key.
Why the intersection?

An assistant should never be able to act with more permission than its own scope allows, even if the user it's acting for has broader rights. The intersection enforces the principle of least privilege for assistants.
