External Role Providers

Where user roles come from, how the service chooses between them, and what goes wrong when they fail.

In Lesson 02 you saw that on a cache miss, the service calls ExternalRolesService.GetRolesForUserAsync() before it can touch the database. That call is the subject of this lesson. It is the layer responsible for most "why is this user denied?" puzzles in production.

What a Provider Does

A provider answers one question: what role names does this user have in an external system? It takes a user identity and returns a list of ExternalRoleInformation objects — each carrying a Name string that must later match a configured record in tblExternalRole.

The name must match exactly

If a provider returns the string "Underwriter" but the database has ExternalRole.ExternalSourceId = "underwriter" (lowercase), the stored procedure won't find a match and the user gets no access from that role. Case sensitivity in provider responses is a common source of subtle bugs.

The Five Provider Types

Claims Never throws

type: "Claims" No external call ClaimType param

Reads roles directly from the JWT claims already in the request. Filters claims by the configured ClaimType (defaults to System.Security.Claims.ClaimTypes.Role) and returns each matching claim value as a role name.

Failure mode: If no claims match the type, returns an empty list — it never throws. The safest provider to configure; has zero external dependencies.

Assistant Throws on unexpected HTTP

type: "Assistant" Assistant Management Service No config params

Calls GET /api/v2/assistants/{externalUserId}/exists on the Assistant Management Service. Returns a single role named "Assistant" if the user exists there (HTTP 200/204). Returns empty on 404 (user is not an assistant).

Failure mode: Any other HTTP status throws an exception — which stops the entire provider cascade.

MicrosoftGraph Empty on 404, throws otherwise

type: "MicrosoftGraph" Azure AD / Entra ID ClientId, ClientSecret, TenantId

Calls the Microsoft Graph API to get the user's Azure AD group memberships. Returns each group's DisplayName as the role name.

Failure mode: 404 (user not found in Azure AD) returns empty. Any other Graph API error or network failure throws, stopping the cascade.

CustomApi Throws on HTTP failure or bad schema

type: "CustomApi" Any REST API ServiceEndpoint + token strategy

Makes a GET request to a configurable ServiceEndpoint URL (may include a {userId} placeholder). Expects a JSON response matching: {"securityRoles": [{"key": "...", "description": "..."}]}

Token strategy: either JWT forwarding (the incoming user token is forwarded) or client credentials (fetches a new token using TokenClientId + TokenClientSecret).

Failure mode: Throws on missing config, HTTP errors, or JSON schema mismatch.

OIPA Throws on missing config or SOAP failure

type: "Oipa" OIPA Exposed Computation (SOAP) Url, Username, Password, ComputationId

Calls the OIPA insurance platform's SOAP web service using computation ID "API_M037" (configurable). The service returns a list of role strings specific to the insurance system.

Failure mode: Throws if Url, Username, or Password are missing, or if the SOAP call fails.

How the Service Chooses: Priority + First-Match-Wins

ExternalRolesService loads all active providers for the tenant, sorts them by Priority ascending (lower number = runs first), then iterates:

The three rules of provider execution
  1. Non-empty result → return immediately (remaining providers are skipped)
  2. Empty result → try next provider
  3. Exception → log error, return empty, stop (remaining providers are skipped)

This is the exact loop from the code:

foreach (var provider in providers.Where(x => x.IsActive).OrderBy(x => x.Priority))
{
    try
    {
        var roles = await provider.GetRolesForUserAsync(parameters, providerId);
        if (roles.Any())
            return roles;        // ← stop: first winner takes all
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Exception occurred...");
        return new List<ExternalRoleInformation>();  // ← stop: exception = empty
    }
}
return new List<ExternalRoleInformation>();  // ← all exhausted

A worked example

Suppose a tenant has three active providers:

PriorityTypeWhat happens for User A
1ClaimsReturns empty (user has no matching claims)
2MicrosoftGraphReturns ["Underwriter", "BasicAccess"] ← non-empty → return immediately
3OIPANever called

Now change the scenario: MicrosoftGraph throws an exception.

PriorityTypeWhat happens
1ClaimsReturns empty
2MicrosoftGraphThrows → log error, return empty [ ]
3OIPANever called — exception stopped the cascade

The result: User A gets zero access, even though OIPA could have resolved their roles.

This is the most dangerous production behaviour

An intermittent failure in a higher-priority provider silently denies all users for whom that provider throws — even if a lower-priority provider would have succeeded. When debugging unexpected Deny results, always check logs for provider exceptions first.

How Role Names Become Access Keys: The Two-Level Mapping

A provider returning "Underwriter" doesn't directly grant permissions. There is a two-level indirection through the database:

Provider returns Name = "Underwriter" │ ▼ [tblExternalRole] ExternalSourceId = "Underwriter" (matched by stored proc) TenantId = ... │ ├──[tblExternalRoleToRole]──────────────────────────────────────────┐ │ ExternalRoleId → PolicyRoleId │ │ ▼ │ [tblRole] → [tblPolicyRole] → [tblPermission] │ └──[tblExternalRoleToPermission]────────────────────────────────────┐ ExternalRoleId → PolicyPermissionId │ ▼ [tblPermission] → Permission.Key │ ▼ Stored proc returns: ["policy.submit", "dashboard.view", ...]

The stored procedure [dbo].[GetUserAccessKeys] receives the role names as a CSV string and walks this graph entirely in SQL, returning the flat list of permission key strings.

What this means for configuration

Adding a new external provider is not enough. For a role that provider returns to grant access, you must also create an ExternalRole record in the database with a matching ExternalSourceId, and attach it to at least one internal Role or Permission via the junction tables.

Debugging: "Why Is This User Denied?"

When a user gets Deny unexpectedly, work through this checklist:

  1. Check for provider exceptions in the logs. Look for LogError from ExternalRolesService. An exception in any provider stops the cascade and returns empty.
  2. Check which providers are active for the tenant (query tblExternalRoleProvider where TenantId = ? AND IsActive = 1). A deactivated provider silently contributes nothing.
  3. Verify the role name returned by the provider exactly matches an ExternalRole record (SELECT * FROM tblExternalRole WHERE TenantId = ? AND ExternalSourceId = 'the-name'). Case mismatch here is silent — no error, just no permissions.
  4. Check the ExternalRole has active mappings (tblExternalRoleToRole or tblExternalRoleToPermission with IsActive = 1). A mapped role/permission that was deactivated gives nothing.
  5. Check the cache. If the deny happened just after a permission change, the user's old key list may still be cached. Call DELETE /api/v1/cache/permissions/{userId} to force recomputation.
