Debugging Authorization Failures

Using the complete mental model to diagnose "user is denied" in production — systematically, not by guessing.

Lessons 01–04 gave you the mental model. This lesson applies it. The mission goal is: "Debug common authorization failures confidently." By the end you will have a repeatable diagnostic process and three worked cases under your belt.

The Five Root Causes of Unexpected Deny

1

Provider exception

A higher-priority provider threw an exception. The cascade stopped. No roles were resolved. All users hit by that provider get Deny on everything, not just one key.

2

Role name mismatch

The provider returned a name (e.g. "Compliance_Officers") that does not exactly match any tblExternalRole.ExternalSourceId. Silent — no error, no match.

3

Missing or inactive mapping

The ExternalRole record exists, but tblExternalRoleToPermission / tblExternalRoleToRole has no active row linking it to the relevant PolicyPermission/PolicyRole.

4

Stale cache

A permission was added or removed in the database, but the user's Redis cache entry (TTL not yet expired) still holds the old key list. The change hasn't taken effect yet.

5

Wrong access key string

The caller is requesting X but the actual key in the database is x (casing), or the policy nesting is different from what the caller assumed. The key simply isn't in the list.

The Diagnostic Toolkit

1. Check logs for provider exceptions

Search structured logs (Seq/Graylog) for LogError from Equisoft.AuthorizationService.Services.V1.Services.ExternalRole.ExternalRolesService around the time of the failure. The message contains "Exception occurred while fetching external roles".

2. Check active providers for the tenant

SELECT ExternalRoleProviderId, Type, Priority, IsActive, Description FROM tblExternalRoleProvider WHERE TenantId = 'TENANT-GUID' ORDER BY Priority ASC; -- Verify: correct types active, expected priority order

3. Check ExternalRole record by name

SELECT ExternalRoleId, ExternalSourceId, Name, IsActive FROM tblExternalRole WHERE TenantId = 'TENANT-GUID' AND ExternalSourceId = 'the-exact-role-name-from-provider'; -- Must be an exact match (case-sensitive). Zero rows = cause 2.

4. Check mappings for an ExternalRole

-- Permission mappings (cause 3 check) SELECT er.Name AS ExternalRole, pol.Key AS PolicyKey, perm.Key AS PermissionKey, erp.IsActive AS MappingActive FROM tblExternalRole er JOIN tblExternalRoleToPermission erp ON er.ExternalRoleId = erp.ExternalRoleId JOIN tblPolicyPermission pp ON erp.PolicyPermissionId = pp.PolicyPermissionId JOIN tblPermission perm ON pp.PermissionId = perm.PermissionId JOIN tblPolicy pol ON pp.PolicyId = pol.PolicyId WHERE er.TenantId = 'TENANT-GUID' AND er.ExternalSourceId = 'the-role-name'; -- Zero rows = cause 3. IsActive=0 row = mapping disabled.

5. Simulate what the stored procedure produces

EXEC [dbo].[GetUserAccessKeys] @TenantId = 'TENANT-GUID', @Roles = 'RoleName1,RoleName2'; -- The full list the user's cache SHOULD contain once recomputed. -- If your expected key is absent here, the mapping is missing (cause 3 or 5). -- If it IS here and the user still gets Deny, the cache is stale (cause 4).

6. Force cache invalidation

-- REST call (service account token required) DELETE /api/v1/cache/permissions/{userId}?executedFor={userId} DELETE /api/v1/cache/clear/assistants/{assistantId} -- for assistant contexts

Case Studies

Case 1 — "It worked yesterday"
Reported symptoms
Multiple users from the same tenant lost all access simultaneously this morning. No code was deployed. No policy changes were made in the Authorization Service. The maintenance window last night involved patching the network switches.

What is your first diagnostic step?

You search Seq for LogError from ExternalRolesService between 02:00–06:00.

ERROR ExternalRolesService Exception occurred while fetching external roles Type: MicrosoftGraph Message: The operation was canceled (network timeout after 30s) TenantId: acme-tenant-id

The MicrosoftGraph provider (priority 1) timed out after the network switch patching disrupted Azure AD connectivity. The cascade stopped — no roles resolved for any user who relied on this provider.

Root cause: Provider exception (Cause 1) MicrosoftGraph provider at priority 1 is timing out due to network disruption.

Immediate fix: Temporarily set IsActive = 0 for the MicrosoftGraph provider in tblExternalRoleProvider so the cascade falls through to the next provider. Then force cache invalidation for affected users with DELETE /cache/permissions/{userId}. Once Azure AD connectivity is restored, re-enable the provider.

Long-term fix: If OIPA can serve as a fallback for this tenant, verify it is configured and active at a lower priority.
Case 2 — "It never worked"
Reported symptoms
A new Azure AD group "Compliance_Officers" (with underscore) was created last week and its members should have access to platform.compliance.reports.view. None of them can access it. Only users in this group are affected.

No provider exceptions in logs. Which query do you run first?

You query tblExternalRole for the name.

SELECT ExternalSourceId, Name, IsActive FROM tblExternalRole WHERE TenantId = 'acme-tenant-id' AND ExternalSourceId LIKE '%Compliance%'; Result: ExternalSourceId | Name | IsActive -------------------+---------------------+--------- ComplianceOfficers | Compliance Officers | 1

The ExternalRole record exists and is active — but the ExternalSourceId is "ComplianceOfficers" (no underscore). MicrosoftGraph returns the AD group DisplayName, which is "Compliance_Officers" (with underscore). The names don't match — the stored procedure receives "Compliance_Officers" but looks for "ComplianceOfficers".

Root cause: Role name mismatch (Cause 2) The ExternalSourceId in the database doesn't match the string MicrosoftGraph returns.

Fix: Update the ExternalRole record so ExternalSourceId = 'Compliance_Officers' (with underscore), matching the Azure AD group DisplayName exactly. Then force cache invalidation for all affected users. No code change needed — this is a data fix via the management API or directly in the DB (with care).
Case 3 — "The change isn't taking effect"
Reported symptoms
An analyst's role was updated 20 minutes ago — a new permission analytics.dashboard.export was added. The analyst still gets Deny when they try to use it. The developer who made the change confirms the database was updated correctly.

No provider exceptions. You confirm the SP produces the new key. What's next?

You call the cache invalidation endpoint with a service account token.

DELETE /api/v1/cache/permissions/{analyst-user-id}?executedFor={analyst-user-id} → HTTP 200 OK

The user's Redis cache entry held the old key list computed before the permission was added. The TTL (say, 1 hour) had not yet expired. Forcing invalidation evicts the entry — the next request triggers a fresh evaluation, which now returns the new key.

Root cause: Stale cache (Cause 4) The database had the correct data, but the user's cached key list hadn't been invalidated.

Fix already applied: Cache invalidated — user now gets Allow on the next request.

Process improvement: When making permission changes that need immediate effect, always include a cache invalidation step. Document this in your deployment checklist.
