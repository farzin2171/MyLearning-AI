Single Sign-On in the DIT Ecosystem

Lesson 11 of the Identity series ~25 min SSO · Session Management · External Apps
Mission tie-in You can now register clients and configure external identity providers. This lesson answers the next natural question: once a user logs in, how does that session carry across Apply, Illustrate, and potentially apps we don't own?

1. What SSO actually means in the Equisoft IDG

Single Sign-On (SSO) is not a feature you enable with a flag. It is a consequence of one central fact:

The key insight The Identity Gateway (IDG) creates a browser session cookie when a user first authenticates. Every OIDC client that redirects to the same IDG will hit that cookie, and the IDG can issue tokens without asking for credentials again.

The IDG (Duende IdentityServer) writes a cookie — typically named idsrv.session — to the IDG's own domain (e.g. identity.equisoft.com). That cookie is the SSO session. Apply doesn't touch it, Illustrate doesn't touch it. It lives entirely at the IDG level.

The SSO "bubble"

Apply and Illustrate are already inside the SSO bubble because they are both registered OIDC clients of the same IDG. SSO between them is not a feature request — it is already active. The question is only whether the applications are configured to leverage it.

2. The SSO flow step by step

Scenario: User logs into Apply, then navigates to Illustrate

  1. Apply: no token → OIDC challenge

    User hits Apply. No local auth cookie exists. Apply triggers Challenge(OpenIdConnectDefaults.AuthenticationScheme) and redirects the browser to the IDG's /connect/authorize endpoint with client_id=apply.TENANT.

  2. IDG: no session → show login page

    The IDG has no idsrv.session cookie yet. It redirects the browser to the external IdP (Azure B2C or Okta, depending on the tenant), where the user enters credentials.

  3. IDG: creates session, issues code to Apply

    After the external IdP authenticates the user, the IDG creates its own session (idsrv.session cookie on the identity.equisoft.com domain), then redirects back to Apply's /signin-oidc with an authorization code.

  4. Apply: exchanges code for tokens, writes its own auth cookie

    Apply calls the IDG's /connect/token endpoint with the code + PKCE verifier, receives access_token and id_token, and writes its own application auth cookie. The user is now logged into Apply.

  5. User navigates to Illustrate: OIDC challenge again

    Illustrate has no Apply cookie (different domain / different app cookie). It issues its own OIDC challenge, redirecting to the IDG's /connect/authorize with client_id=illustrate.TENANT.

  6. IDG: session found → no login required

    The IDG sees the idsrv.session cookie it wrote in step 3. The user is already known. The IDG immediately redirects back to Illustrate's callback with a new authorization code — no credentials prompt.

  7. Illustrate: gets its own tokens

    Illustrate exchanges the code for tokens scoped to illustrate.TENANT. The access token carries the same user claims, same tenant claim, same identityRole. User is seamlessly logged into Illustrate.

Takeaway Each application gets its own access token scoped to its own client. The IDG session is the invisible bridge. Neither Apply nor Illustrate shares tokens with each other — they share the IDG session.

3. What makes Apply and Illustrate SSO-capable

The OIDC client configuration governs whether SSO flows correctly. Here are the critical settings in the client entry (inside dit-base-config-common):

Config fieldTypical valueWhy it matters for SSO
AllowedGrantTypes ["authorization_code"] Authorization Code + PKCE is the browser-side SSO-compatible grant. Never use implicit for SSO.
RequirePkce true Prevents code interception. Required for SSO security in public/browser clients.
RedirectUris ["https://apply.T.com/signin-oidc"] IDG will only redirect back here. Must match exactly or SSO redirect fails.
PostLogoutRedirectUris ["https://apply.T.com/signout-callback-oidc"] Required for coordinated logout (see §5).
FrontChannelLogoutUri "https://apply.T.com/signout-oidc" The IDG pings this URL via iframe when a logout happens in any other connected app.
AllowedScopes ["openid","profile","email","tenant_access"] Scopes determine which claims appear in the access token. Must include openid for OIDC.

On the application side, Apply's AddOpenIdConnect must not force re-authentication. The footgun here is setting prompt = "login" — that tells the IDG to always show the login page, breaking SSO completely.

// ✓ Correct — no prompt override, SSO works
options.Events.OnRedirectToIdentityProvider = ctx => {
    // Only set prompt when you deliberately want to force re-login
    // e.g. for a "switch account" feature
    return Task.CompletedTask;
};

// ✗ Breaks SSO for everyone
options.Events.OnRedirectToIdentityProvider = ctx => {
    ctx.ProtocolMessage.Prompt = "login";  // forces credentials every time
    return Task.CompletedTask;
};
Common footgun Some developers add prompt=login during debugging to avoid cached sessions. This accidentally disables SSO for all users in that environment. Remove it before merging.

4. Enabling SSO for an application outside DIT

An external application becomes part of the SSO bubble by becoming an OIDC client of the IDG. It does not matter whether the app is Java, Python, Node.js, or .NET — the protocol is the same.

What you do in the IDG (dit-base-config-common)

  • Add a new client entry to the clients config
  • Specify its redirect and logout URIs
  • Assign it allowed scopes
  • Generate and store a client secret (hashed)

What the external app must do

  • Configure the IDG as its OIDC Authority
  • Set the ClientId and ClientSecret
  • Handle the /signin-oidc callback
  • Optionally wire up logout to /signout-oidc

Step 1 — Add the client in dit-base-config-common

Open the clients config file (the same one where Apply and Illustrate entries live). Add a new entry. The minimal structure for a browser-based SSO app:

{
  "ClientId": "ext-myapp",
  "ClientName": "My External Application",

  "ClientSecrets": [
    { "Value": "<SHA256-hashed-secret>" }
  ],

  "AllowedGrantTypes": ["authorization_code"],
  "RequirePkce": true,

  "RedirectUris": [
    "https://myapp.external.com/signin-oidc"
  ],
  "PostLogoutRedirectUris": [
    "https://myapp.external.com/signout-callback-oidc"
  ],
  "FrontChannelLogoutUri": "https://myapp.external.com/signout-oidc",

  "AllowedScopes": ["openid", "profile", "email"],

  "AllowOfflineAccess": false,

  "Properties": {
    "EcosystemTenant": "MYTENANT"
  }
}
EcosystemTenant is required Without the EcosystemTenant property, the IDG's token generation logic will not inject the tenant claim into tokens issued for this client. SSO will work, but downstream DIT services relying on the tenant claim will see nothing.

Step 2 — Configure the external app to use the IDG as its authority

For an ASP.NET Core application, the configuration mirrors exactly what Apply does:

// Program.cs (or Startup.cs)
builder.Services.AddAuthentication(options => {
    options.DefaultScheme = CookieAuthenticationDefaults.AuthenticationScheme;
    options.DefaultChallengeScheme = OpenIdConnectDefaults.AuthenticationScheme;
})
.AddCookie()
.AddOpenIdConnect(options => {
    // Point at the Equisoft IDG — this is the only authority
    options.Authority = "https://identity.equisoft.com";

    options.ClientId     = "ext-myapp";
    options.ClientSecret = "<plain-text secret from vault>";
    options.ResponseType = "code";   // Authorization Code
    options.UsePkce      = true;

    options.Scope.Clear();
    options.Scope.Add("openid");
    options.Scope.Add("profile");
    options.Scope.Add("email");

    options.SaveTokens = true;

    // The IDG discovery doc is at:
    // https://identity.equisoft.com/.well-known/openid-configuration
    // — AddOpenIdConnect fetches this automatically on startup.
});

For non-.NET apps (Node.js, Python, Java), use any OIDC-certified library. They all need the same three pieces of information:

ParameterValue
issuer / authorityhttps://identity.equisoft.com
client_idext-myapp
client_secret(from secrets vault)

Step 3 — Verify SSO is working

A quick manual test:

  1. Log into Apply

    Open the browser, log in normally. Confirm you are fully authenticated inside Apply.

  2. Open your external app in the same browser

    Navigate to the external app's protected route. It should redirect you to the IDG.

  3. Watch the address bar

    You should see the IDG URL appear briefly, then immediately redirect back to your external app — with no login prompt. If a login page appears, check: (a) same IDG authority? (b) EcosystemTenant matches the tenant you logged into Apply with? (c) no prompt=login on the OIDC redirect?

5. Silent authentication: prompt=none

Sometimes an app wants to check whether the user has an active SSO session without visibly redirecting to the IDG (for example, in an iframe or during a background token refresh check). This is done by adding prompt=none to the authorize request.

// Sending a prompt=none request
GET https://identity.equisoft.com/connect/authorize
  ?response_type=code
  &client_id=ext-myapp
  &redirect_uri=https://myapp.external.com/signin-oidc
  &scope=openid+profile
  &prompt=none          // ← silent check
  &nonce=abc123

Two possible outcomes:

SituationIDG responseYour app should
Active SSO session exists Redirect with code=… — success Exchange code → get tokens silently
No session / session expired Redirect with error=login_required Fall back to a visible OIDC challenge (normal login flow)
When is this used? Primarily for single-page apps (SPAs) that use iframes to silently refresh tokens, and for apps that want to restore a session on page load without a visible redirect. DIT's MVC apps mostly rely on the standard Authorization Code flow with refresh tokens for this.

6. Single Sign-Out (SLO)

SSO without coordinated logout is incomplete. If a user logs out of Apply but stays logged in at the IDG, navigating to Illustrate will silently re-authenticate them — which is usually not what "log out" means.

Front-Channel Logout (browser-based)

The IDG renders a hidden page with <img> or <iframe> tags pointing to each connected client's FrontChannelLogoutUri. Each client clears its own session cookie when hit. Simple, no server-to-server auth needed. Unreliable if a browser blocks third-party cookies or iframes.

Back-Channel Logout (server-to-server)

The IDG sends a signed JWT (logout token) via HTTP POST to each registered client's back-channel endpoint. The client validates the token and clears the session. More reliable than front-channel but requires the client to expose an HTTPS endpoint reachable by the IDG.

How to trigger SLO correctly

When a user clicks "Log Out" in Apply (or any DIT app), the application must:

  1. Clear the local app cookie

    Call HttpContext.SignOutAsync(CookieAuthenticationDefaults.AuthenticationScheme).

  2. Redirect to the IDG end-session endpoint

    Call HttpContext.SignOutAsync(OpenIdConnectDefaults.AuthenticationScheme). ASP.NET Core's OIDC handler automatically redirects to https://identity.equisoft.com/connect/endsession?id_token_hint=<id_token>&post_logout_redirect_uri=<your-uri>.

  3. IDG destroys session + notifies all clients

    The IDG removes the idsrv.session cookie and fires logout notifications (front-channel or back-channel) to every other client that was active in that session.

// Apply's Logout action (simplified)
public async Task<IActionResult> Logout()
{
    await HttpContext.SignOutAsync(CookieAuthenticationDefaults.AuthenticationScheme);

    // This line triggers the redirect to IDG /connect/endsession
    // and passes the id_token_hint automatically (if SaveTokens = true)
    await HttpContext.SignOutAsync(OpenIdConnectDefaults.AuthenticationScheme);

    return new EmptyResult(); // redirect is handled by OIDC middleware
}
The id_token_hint matters Without id_token_hint, the IDG may show a "do you really want to log out?" confirmation page. Always include it. ASP.NET Core's OIDC handler handles this automatically when SaveTokens = true is set on the client.

External apps and SLO

An external app participates in SLO by registering its FrontChannelLogoutUri in the IDG client config (step 1 above). When any DIT app triggers end-session, the IDG's logout page will include an iframe pointing to the external app's logout URI, clearing the external app's session.

7. Summary: DIT app vs. external app SSO

DIT App (Apply, Illustrate) External App
Client already registered? Yes — in dit-base-config-common No — must be added
Authority URL Same IDG per environment Same IDG per environment
Tenant claim in tokens? Yes — EcosystemTenant set Only if EcosystemTenant is added to config
SSO participation Automatic — already in the bubble Automatic once registered as OIDC client
Logout participation Yes — FrontChannelLogoutUri registered Must add FrontChannelLogoutUri to client config
Code change needed? None (already configured) App must implement OIDC auth + callback handler

8. SSO debugging checklist

When SSO is not working (user gets a login prompt they shouldn't), work through these in order:

SymptomMost likely causeFix
Login prompt appears on second app Different IDG URL in the second app's config Ensure Authority is identical across all apps (same host, no trailing slash differences)
Login prompt with prompt=login in URL App is forcing it via OnRedirectToIdentityProvider Remove the prompt=login override
invalid_client error ClientId or ClientSecret mismatch Verify client config in IDG matches what the app sends
unauthorized_client error Grant type not allowed for this client Add authorization_code to AllowedGrantTypes
SSO works but no tenant claim Missing EcosystemTenant in client config Add Properties.EcosystemTenant to the client entry
Logout from Apply, Illustrate still logged in Illustrate's FrontChannelLogoutUri not registered / being blocked Confirm URI is registered; test with browser dev tools network panel during logout
