Services.User: The User Attributes Gateway

Lesson 9 of 9 ~25 min Identity Architecture · Services.User · Connector Pattern

After Lesson 8 you understand how Apply authenticates users — via OIDC cookies for regular users, and local cookies for guests. But authentication answers who are you? A second question remains: what can you do? — i.e. what is your role? This lesson explains why answering that question requires a separate service, what Services.User actually is, and how Apply calls it.

1. The Three-Service Split: Who Owns What?

A common mistake is to think one service handles "users". In reality, three services share responsibility — each owning a different slice of what we loosely call "user information":

Identity Gateway (IDG)

"Who are you?"
  • Authenticates credentials
  • Issues access tokens & ID tokens
  • Provides sub / externalobjectidentifier
  • Manages sessions & refresh tokens
  • Federated IdPs (Azure B2C, Okta)

Apply (local User table)

"Do you exist here?"
  • Local user record (GUID primary key)
  • ExternalId → links to IDG's sub
  • IsGuest flag
  • TenantId for multi-tenancy
  • Created/updated on first OIDC sign-in

Services.User

"What is your role / profile?"
  • Routes attribute queries to upstream sources
  • Azure Graph, WebAPIs, claim connectors
  • IdentityRole, AgentNumber, email
  • No User table of its own
  • Returns UserInformation DTO
The core insight Services.User has no User entity in its database. Its SQL schema stores connectors, handlers, and configurations — not user rows. When you ask it "what is Alice's role?", it fetches that answer live from Azure Graph (or a WebAPI, or a configured claim) and returns it. It is a routing layer, not a store.

2. The Sign-In Flow: How Apply's Local User Record Is Created

When a user logs in for the first time, their Apply user record does not yet exist. It is created automatically by a cookie event hook — not by a dedicated "register" endpoint.

src/Equisoft.Apply/Infrastructure/Authentication/Functions/CookieAuthenticationFunctions.cs — OnSignedInFunction

Inside OnSignedInFunction, the code does:

  1. Extract externalobjectidentifier claim from the newly-issued cookie principal. This is the user's identity in the Identity Gateway — their sub.
  2. Look up User row in Apply's database by ExternalId == externalobjectidentifier.
  3. If not found: create a new User row with ExternalId, FirstName, LastName, Email from the OIDC claims. IsGuest = false.
  4. If found: update FirstName, LastName, Email if they changed (the user may have updated their profile in the IdP).
Tip — OnSignedIn only fires for OIDC users Remember from Lesson 8: OnValidatePrincipal short-circuits for isGuest = "true". But OnSignedIn is only invoked when the OIDC handler issues a new cookie — guests never go through OIDC, so they never trigger this event. Their user record is created directly in GuestAuthenticationHelper.

3. The Role-Check Flow: How Apply Calls Services.User

After login, Apply often needs to know: "is this user an advisor? a manager?" It cannot store the answer locally because the source of truth for roles lives in Azure AD (or another connector-backed source). This is where Services.User comes in.

src/Equisoft.Apply.Data/Externals/Clients/UserServiceClient.cs 
// Apply's outbound HTTP call to Services.User
GET /api/v2/user/identities/role/{externalUserId}
Authorization: Bearer {service-account-token}

Notice the token used here is a service account token — not the user's own access token. Apply authenticates itself to Services.User as a service, not on behalf of the user. The externalUserId parameter carries the user's identity.

src/Equisoft.Apply.Services/Users/UserService.cs 
public async Task<bool> UserHasRoleAsync(string externalUserId, string role)
{
    var userRole = await _userServiceClient.GetUserRoleByIdAsync(externalUserId);
    return !string.IsNullOrWhiteSpace(userRole) && userRole == role;
}

4. What Lives Inside Services.User

The connector pattern

Services.User doesn't hard-code any data source. Instead it has a pluggable Connector → Handler system. Each tenant can have different connectors configured. When a role query arrives, the service walks the configured connectors in order (with cascading fallback) until one returns a result.

Connector types

ServiceDbContext
  • ClaimConnectorConfiguration — reads a claim from the token
  • AzureGraphConnectorConfiguration — queries Microsoft Graph API
  • WebApiConnectorConfiguration — calls a custom REST endpoint with configurable routes

What the DB actually stores

No User rows
  • Connector definitions (type, config)
  • Handlers (which operation each connector covers)
  • Tenant-specific connector-handler mappings
  • Cascading fallback chains

Services.User authentication

Services.User validates inbound JWTs from the same Identity Gateway: 

// appsettings.Development.json
"auth": {
    "jwt": {
        "authority": "https://api.dev.equisoft.com/identity",
        "audience": "userapi"   // ← "userapi" scope required on the token
    }
}

The ServiceAccountAuthorizeFilter further restricts the management endpoints (/api/v2/management/…) to callers whose token carries IdentityType == Service — regular user tokens are rejected.

Services.User also calls the Identity Gateway

For ID conversion queries (GET /api/v2/useridentity/convert), Services.User itself calls back to the Identity Gateway's /user/convert/{userId} endpoint, using a service account Bearer token obtained from a token endpoint. The IDG is simultaneously:

  • The token authority that validates incoming requests to Services.User
  • A downstream API that Services.User calls for ID conversion

Watch out — two "UserService" classes exist Equisoft.Apply.Services.Users.UserService is Apply's local service for its own User table. UserService.Services.V2.Services.UserService is the internal logic class inside Services.User. They have the same name but live in different projects and serve completely different purposes. When reading code, always check the namespace first.

7. Knowledge Check

Quiz — User Service Architecture

Services.User has no User table. What does its database actually store?

✓ Correct. The schema stores Connectors, Handlers, ConnectorHandler mappings, and their tenant-specific configurations. These rows describe how to find user attributes — not the attributes themselves.

Which token does Apply use when it calls GET /api/v2/user/identities/role/{id}?

✓ Correct. UserServiceClient uses IServiceAccountTokenRepository to obtain a client-credentials token for the Apply service account. The user's own token is not forwarded.

When does Apply's local User row get created for a first-time OIDC user?

✓ Correct. CookieAuthenticationFunctions.OnSignedInFunction is fired by ASP.NET Core right after the OIDC handler issues the cookie. It extracts externalobjectidentifier and upserts the User row.

What does the ServiceAccountAuthorizeFilter in Services.User check?

✓ Correct. The filter reads IIdentityContext.IdentityType from the resolved token claims. If the type is not Service, a 403 Forbidden is returned. Regular user tokens are rejected from management endpoints.
