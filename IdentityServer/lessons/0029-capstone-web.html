Capstone — Build WorkSpace Web & End-to-End Test

~35 minutes Final capstone lesson Prerequisites: IS (5001) + Calendar API (5003) running

This is the last piece: a user opens a browser, clicks "Log in", is redirected to WorkSpace IS, enters their credentials, is sent back to the web app, and sees their calendar events — all fetched from the Calendar API using their access token. Build it, run it, watch the full flow complete.

Token flow recap OIDC login → IS issues ID token + access token + refresh token → web app stores them in a cookie (SaveTokens = true) → web app reads access token from cookie → sets Bearer header → calls Calendar API → API validates token → returns tenant-filtered events.

2

Wire OIDC authentication in Program.cs

Two schemes: Cookie (stores the session after login) + OpenIdConnect (talks to WorkSpace IS). SaveTokens = true is the critical line — it stores the access token inside the session cookie so controllers can retrieve it later.

Program.cs
using Microsoft.AspNetCore.Authentication.Cookies;
using Microsoft.AspNetCore.Authentication.OpenIdConnect;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllersWithViews();

builder.Services
    .AddAuthentication(options =>
    {
        options.DefaultScheme          = CookieAuthenticationDefaults.AuthenticationScheme;
        options.DefaultChallengeScheme = OpenIdConnectDefaults.AuthenticationScheme;
    })
    .AddCookie(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddOpenIdConnect(OpenIdConnectDefaults.AuthenticationScheme, options =>
    {
        options.Authority    = "https://localhost:5001";
        options.ClientId     = "workspace-web";
        options.ResponseType = "code";      // Authorization Code flow

        // Scopes to request
        options.Scope.Clear();
        options.Scope.Add("openid");
        options.Scope.Add("profile");
        options.Scope.Add("email");
        options.Scope.Add("calendar.read");
        options.Scope.Add("offline_access");    // request a refresh token

        // Store access + refresh tokens in the session cookie
        options.SaveTokens = true;

        // Trust dev cert on IS (remove in production)
        options.BackchannelHttpHandler = new HttpClientHandler
        {
            ServerCertificateCustomValidationCallback = (_, _, _, _) => true
        };
    });

builder.Services.AddHttpClient("CalendarApi", client =>
{
    client.BaseAddress = new Uri("https://localhost:5003");
    // Trust Calendar API dev cert
}).ConfigurePrimaryHttpMessageHandler(() => new HttpClientHandler
{
    ServerCertificateCustomValidationCallback = (_, _, _, _) => true
});

var app = builder.Build();

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllerRoute(name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

app.Run();
3

Create the CalendarController

Replace the default HomeController or create a new controller. The key line is GetTokenAsync("access_token") — it reads the token that SaveTokens = true stored in the cookie.

Controllers/CalendarController.cs
using Microsoft.AspNetCore.Authentication;
using Microsoft.AspNetCore.Authorization;

[Authorize]
public class CalendarController : Controller
{
    private readonly IHttpClientFactory _factory;

    public CalendarController(IHttpClientFactory factory) => _factory = factory;

    public async Task<IActionResult> Index()
    {
        // Retrieve the access token saved in the session cookie
        var accessToken = await HttpContext.GetTokenAsync("access_token");

        var client = _factory.CreateClient("CalendarApi");
        client.DefaultRequestHeaders.Authorization =
            new System.Net.Http.Headers.AuthenticationHeaderValue("Bearer", accessToken);

        var response = await client.GetAsync("/api/calendar");
        response.EnsureSuccessStatusCode();

        var events = await response.Content
            .ReadFromJsonAsync<List<CalendarEventDto>>();

        return View(events);
    }
}

// Simple DTO matching the Calendar API's JSON shape
public record CalendarEventDto(int Id, string Title, string TenantId, DateTime Start);

Add login and logout endpoints

Create Controllers/AccountController.cs. Logout must sign out of both the local cookie and the IS session (RP-Initiated Logout):

Controllers/AccountController.cs
using Microsoft.AspNetCore.Authentication;
using Microsoft.AspNetCore.Authentication.Cookies;
using Microsoft.AspNetCore.Authentication.OpenIdConnect;

public class AccountController : Controller
{
    // Trigger OIDC login — redirects to WorkSpace IS
    public IActionResult Login(string returnUrl = "/")
        => Challenge(new AuthenticationProperties { RedirectUri = returnUrl },
                     OpenIdConnectDefaults.AuthenticationScheme);

    // Sign out locally AND send the browser to IS logout (RP-Initiated Logout)
    public IActionResult Logout()
        => SignOut(new AuthenticationProperties { RedirectUri = "/" },
                   CookieAuthenticationDefaults.AuthenticationScheme,
                   OpenIdConnectDefaults.AuthenticationScheme);
}
Verify → dotnet build succeeds. Start all three services (IS on 5001, CalendarApi on 5003, WorkSpaceWeb on 5002) in three separate terminals, then proceed to the smoke test.

The End-to-End Flow

What happens when Alice browses to https://localhost:5002/Calendar:

Browser → GET /Calendar No session → 302 to OIDC challenge
Web App → 302 → https://localhost:5001/connect/authorize?client_id=workspace-web&…
IS → 302 → /Account/Login (no session on IS either)
Alice → POST alice / alice IS validates, creates auth cookie
IS → 302 → https://localhost:5002/signin-oidc?code=abc&state=…
Web App → POST /connect/token (code exchange, back-channel) OIDC middleware exchanges code for tokens
Web App → Stores access_token + refresh_token in session cookie SaveTokens = true does this
Web App → GET /api/calendar Authorization: Bearer eyJ… CalendarController reads token from cookie
Calendar API → Validates JWT, filters by tenant-acme, returns 2 events
Browser → Renders: "Welcome, Alice Smith — ACME All-Hands, ACME Sprint Review"

Smoke Test Checklist

  • ✓ 1 Browse to https://localhost:5002/Calendar — you are redirected to the IS login page
  • ✓ 2 Log in as alice / alice — you return to the web app and see "ACME All-Hands" and "ACME Sprint Review"
  • ✓ 3 The tenant label shows tenant-acme — the claim came through IProfileService
  • ✓ 4 Click "Sign out" — browser is redirected to IS for session termination, then back to /
  • ✓ 5 Log in as bob / bob — you see "Globex Kickoff" and tenant label tenant-globex
  • ✓ 6 Check the Reports Worker console — "Calendar data: []" (empty list, no tenant_id on machine token)

Knowledge Check

1. What does SaveTokens = true in the OIDC options do?

2. The Logout action signs out of two schemes. What would happen if it only signed out of the Cookie scheme?

3. Alice's access token is stored in an encrypted session cookie. A network attacker intercepts the cookie. What stops them from using it indefinitely?

4. The OIDC middleware handles the code exchange back-channel (web app → IS token endpoint). Why is this better than having the browser exchange the code directly?
