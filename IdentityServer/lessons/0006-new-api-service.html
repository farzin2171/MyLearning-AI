Building a New API Service End-to-End

~40 minutes Hands-on implementation Prerequisites: All previous lessons

This is the capstone lesson. You will follow the exact pattern used by Services.AssistantManagement to wire up a new ASP.NET Core API service with JWT authentication, policy-based authorization, service-account token calls, and authorization service integration. Every step references real code from the existing services.

What you'll build A new API service (MyNewService) that: accepts JWT tokens from the Identity Gateway, enforces scope-based policies, reads identity context via IIdentityContext, calls the Authorization Service for fine-grained permission checks, and calls another service using a service-account token.

The Four Responsibilities of Any Downstream Service

1. Authentication

  • Validate JWT signature from IdG
  • Reject expired or malformed tokens
  • Support both user and service tokens

2. Authorization

  • Enforce scope-based policies per endpoint
  • Distinguish user vs. service accounts
  • Delegate fine-grained checks to Authorization Service

3. Identity context

  • Extract tenantId for data isolation
  • Read sub (user ID) and externalId
  • Handle on-behalf-of flows

4. Outbound calls

  • Get a service-account token from IdG
  • Attach it to outbound API requests
  • Cache and renew before expiry

Phase 2: Configure Authentication & Authorization

Phase 2

Model your Startup.cs on AssistantManagement. The key lines: 

// Startup.cs — ConfigureServices

// Step 1: Authentication — validate incoming JWTs from the Identity Gateway
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme, builder =>
{
    builder.AddJwt();       // DIT wrapper for Microsoft.AspNetCore.Authentication.JwtBearer
    builder.AddBasicAuth(); // For health check endpoints only
});

// Step 2: Authorization — define policies that controllers will use
services.AddAuthorization(options =>
{
    // Scope-based policies:
    options.AddPolicy("mynewservice.read",
        p => p.Requirements.Add(new HasScopeRequirement("mynewservice.read")));
    options.AddPolicy("mynewservice.write",
        p => p.Requirements.Add(new HasScopeRequirement("write:mynewservice")));
});

// Register the DIT scope requirement handler:
services.AddScoped<IAuthorizationHandler, HasScopeRequirementHandler>();

// Step 3: Register IIdentityContext (reads claims from the validated JWT)
services.AddIdentityPrincipal(); // or equivalent DIT registration
// Startup.cs — Configure (middleware pipeline)
app.UseAuthentication();    // MUST come before UseAuthorization
app.UseAuthorization();
app.UseIdentityPrincipal(); // DIT: populates IIdentityContext from the validated principal
Order matters in the pipeline UseAuthentication → UseAuthorization → UseIdentityPrincipal. Reversing any of these causes 401s even with valid tokens.

Phase 3: Controllers — Enforcing Authorization

Phase 3

Apply [Authorize] attributes at the controller or action level. Model on AssistantsController.cs from AssistantManagement: 

[ApiController]
[Route("api/v1/[controller]")]
public class WidgetsController : ControllerBase
{
    private readonly IIdentityContext _identity;
    private readonly IWidgetService   _service;

    public WidgetsController(IIdentityContext identity, IWidgetService service)
    {
        _identity = identity;
        _service  = service;
    }

    // Anyone with a valid JWT and the read scope can list widgets:
    [HttpGet]
    [Authorize("mynewservice.read")]
    public async Task<IActionResult> GetWidgets()
    {
        // tenantId for data isolation — ALWAYS filter by this:
        var tenantId = _identity.TenantId;

        // sub — the user's local ID in the IdG database:
        var userId = _identity.ResourceOwnerId;

        var widgets = await _service.GetWidgetsAsync(tenantId, userId);
        return Ok(widgets);
    }

    // Write operations require the write scope:
    [HttpPost]
    [Authorize("mynewservice.write")]
    public async Task<IActionResult> CreateWidget([FromBody] CreateWidgetRequest request)
    {
        var tenantId = _identity.TenantId;
        var result   = await _service.CreateWidgetAsync(tenantId, request);
        return Created($"api/v1/widgets/{result.Id}", result);
    }
}

// Management endpoints: service accounts only
[ApiController]
[Route("api/v1/management/[controller]")]
[ServiceFilter(typeof(AuthorizeServiceAccountFilter))] // rejects non-service tokens
public class WidgetsManagementController : ControllerBase
{
    // Only service-account tokens (service_isService: true) reach here
}

Implementing AuthorizeServiceAccountFilter

Copy the pattern from AuthorizeServiceAccountFilter.cs in both AssistantManagement and Authorization Service: 

public class AuthorizeServiceAccountFilter : IAuthorizationFilter
{
    private readonly IIdentityContext _identityContext;

    public AuthorizeServiceAccountFilter(IIdentityContext identityContext)
        => _identityContext = identityContext;

    public void OnAuthorization(AuthorizationFilterContext context)
    {
        if (!_identityContext.IsAuthenticated)
        {
            context.Result = new UnauthorizedResult();
            return;
        }
        if (_identityContext.IdentityType != IdentityType.Service)
        {
            context.Result = new ForbidResult();
        }
    }
}

// Register in Startup.cs:
services.AddScoped<AuthorizeServiceAccountFilter>();

Phase 4: Reading Identity Context

Phase 4

IIdentityContext gives you everything you need about the caller. Add the extension method from IdentityContextExtensions.cs to extract the service account name (useful for audit logging): 

// Infrastructure/Extensions/IdentityContextExtensions.cs
public static class IdentityContextExtensions
{
    // Extract service account name from the client_id claim:
    public static string GetServiceAccountName(this IIdentityContext identityContext)
    {
        var claim = identityContext.Claims
            .FirstOrDefault(c => c.Type == "client_id");
        return claim?.Value ?? string.Empty;
    }
}

// Usage in a controller or service:
if (_identity.IdentityType == IdentityType.Service)
{
    var accountName = _identity.GetServiceAccountName();
    // e.g. "assistmgntapi"
}

Phase 5: Outbound Service-Account Calls

Phase 5

When your service needs to call another API on its own behalf (not the user's), use the DIT token client. Model on AssistantManagement's configuration: 

// appsettings.json — service account config:
{
  "ServiceAccounts": {
    "TokenEndpoint": "https://api.dev.equisoft.com/identity/connect/token",
    "PrincipalPrefix": "mynewserviceapi",    // your service's client_id
    "TenantSecrets": {
      "ACME": "the-client-secret-for-acme-tenant"
    }
  }
}

// Startup.cs — wire up the token client:
services.ConfigureTokenClient(Configuration.GetSection("ServiceAccounts"));
// Define the outbound client interface (Refit):
// Infrastructure/Clients/IUserServiceClient.cs
[Headers("Accept: application/json")]
public interface IUserServiceClient
{
    [Get("/v2/user/identities")]
    Task<UserInfo> GetUserAsync(
        [Header("Authorization")] string authorization, // "Bearer {token}"
        [Query] string externalUserId);
}

// Register in Startup.cs:
services.AddRefitClient<IUserServiceClient>()
    .ConfigureHttpClient(c => c.BaseAddress =
        new Uri(Configuration["services:userService:endpoint"]));
// In your service class — obtain and use a service-account token:
public class WidgetService : IWidgetService
{
    private readonly IUserServiceClient _userClient;
    private readonly ITokenClient       _tokenClient; // from DIT

    public async Task<Widget> GetWidgetWithOwnerAsync(Guid tenantId, Guid widgetId)
    {
        var token = await _tokenClient.GetTokenAsync(tenantId);
        var user  = await _userClient.GetUserAsync($"Bearer {token}", externalUserId);
        // ...
    }
}

Phase 6: Fine-Grained Authorization Service Checks

Phase 6

For operations that need finer control than scopes alone, forward the user's token to the Authorization Service. Model on how AssistantManagement calls the cache endpoint: 

// Infrastructure/Clients/IAuthorizationClient.cs (Refit)
public interface IAuthorizationClient
{
    [Get("/api/v1/Authorize")]
    Task<HttpResponseMessage> AuthorizeAsync(
        [Header("Authorization")] string authorization,
        [Query] string key);
}

// appsettings.json:
{
  "services": {
    "authorization": {
      "endpoint": "https://api.dev.equisoft.com/authorization"
    }
  }
}

// Usage in a service:
public async Task<IActionResult> DeleteWidget(Guid widgetId,
    [FromHeader] string authorization)
{
    // Check fine-grained permission before deleting:
    var response = await _authzClient.AuthorizeAsync(
        authorization,         // forwarded user token
        key: "widgets.Delete"  // the access key to check
    );

    if (!response.IsSuccessStatusCode)
        return Forbid();

    await _service.DeleteWidgetAsync(widgetId);
    return NoContent();
}
Forward the token, don't re-validate it Pass the Authorization: Bearer … header value straight through to the Authorization Service. It validates the token itself — you don't need to decode or verify it again.
