OAuth 2.0 — The Delegation Problem

~20 minutes Conceptual + interactive No prerequisites

Every OAuth tutorial starts with "here's how to configure a client." This lesson starts one step earlier: why does OAuth exist at all? Understanding the problem it solves makes every other concept — flows, tokens, scopes, grant types — click naturally into place.

The Problem: Third-Party Apps Want Your Data

Imagine it's 2006. You've just signed up for a travel-planning app that wants to import your contacts to find friends who are already using it. It shows you this:

The old way (pre-OAuth) "To import your Gmail contacts, please enter your Gmail username and password."

You type your password. What just happened?

  • Full access, not scoped access. The travel app can now read your email, send email as you, change your account settings — not just read contacts.
  • No expiry. Your password doesn't expire automatically. The app stores it indefinitely.
  • No selective revocation. The only way to cut off the travel app is to change your password — which also breaks every other app using that password.
  • The travel app becomes a high-value target. If its database is breached, attackers get your Google password — not just your contacts.

This was the reality. Twitter's early API, LinkedIn, Facebook Connect — all asked for passwords. It was an industry-wide failure of security design, and OAuth 2.0 was designed specifically to end it.

The OAuth Solution: Delegated Authorization

The core insight is simple: you don't need to share your password for an app to act on your behalf. Instead, you tell a trusted server "I authorize this app to access this specific thing," and the server issues a limited-power token for that purpose.

The travel app never sees your password. It gets a token that can only read contacts, expires in an hour, and can be revoked at any time without touching your password.

The core shift OAuth 2.0 replaces "I trust you with my master key" with "here is a key that opens only the door you need, for only as long as you need it."

The Four OAuth 2.0 Actors

Every OAuth interaction involves exactly four parties. Learning to spot them in any scenario is the foundation of all OAuth analysis. These terms come directly from RFC 6749 §1.1.

Resource Owner

The User

The entity that owns the protected data. Usually a human. They must grant permission before a client receives a token.

Client

The Application

The app that wants access. Could be a web app, mobile app, or a background service. It requests tokens — it never touches the user's password.

Authorization Server

The Token Issuer

The trusted server that authenticates the user and issues tokens. This is exactly what Duende IdentityServer implements.

Resource Server

The Protected API

The API that holds the data. It validates tokens and serves requests only when the token has the right scope.

Try it now — name the actors "A mobile banking app displays your transaction history from your bank's API."

Resource Owner = you  ·  Client = mobile app  ·  Authorization Server = bank's auth system  ·  Resource Server = bank's transaction API

Access Tokens: Limited-Authority Credentials

The clearest analogy for an OAuth access token is a hotel key card:

Property Key card (access token) Master key (your password)
Scope of access Room 512 only Every room, every floor
Expiry Expires at checkout Does not expire automatically
Revocability Deactivate this card; others still work Change the lock; breaks everything
If stolen Attacker gets room 512 until checkout Attacker gets the whole hotel

OAuth 2.0 defines three token types you'll encounter across this course:

  • Access token — short-lived (minutes to hours); sent to Resource Servers to prove authorization.
  • Refresh token — long-lived; sent only to the Authorization Server to obtain new access tokens when the current one expires.
  • ID token — introduced by OpenID Connect (Lesson 16); carries claims about who the user is, not just what they can do.

Scopes: What the Token Is Allowed to Do

A scope is a named permission string. The client requests scopes, the user approves them, and the Authorization Server encodes the approved scopes into the token. The Resource Server checks the token's scopes before serving any request.

ScopeWhat the client is permitted to do
contacts:readRead address book entries
calendar:readRead calendar events
calendar:writeCreate and edit calendar events
email:sendSend email on the user's behalf

The consent screen the user sees is generated from the requested scopes. If a travel app importing contacts shows email:send in the consent screen, that is a red flag — it is requesting far more than it needs.

Principle of least privilege Clients should request only the scopes their current operation requires. Over-scoped tokens increase the blast radius if a token is stolen or a client is compromised.
