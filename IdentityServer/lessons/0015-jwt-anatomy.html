JWT Anatomy — Inside the Token

~25 minutes Conceptual + interactive Prerequisite: Lesson 13

Every OAuth access token and OpenID Connect ID token you will encounter in this course is a JWT — a JSON Web Token. You've seen them referenced in every lesson so far; now you'll open one up. Understanding JWT structure is a prerequisite for building a Duende IdentityServer and for writing correct token validation code.

JWTs are not encrypted by default A JWT is digitally signed (tamper-proof) but not encrypted (not confidential). Anyone can decode the payload without a key. Never put secrets — passwords, PII that shouldn't be exposed, internal system details — in JWT claims unless you're using a JWE (encrypted JWT).

The Three-Part Structure

A JWT is three Base64URL-encoded strings joined by dots:

header payload signature

Paste any JWT at jwt.io to decode it instantly. The site decodes the Base64URL on the left and verifies the signature on the right.

Part 1: The Header

Decoded, the header above is:

{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-2024-01"
}
ClaimMeaningWhat to know
alg Signing algorithm RS256 = RSA + SHA-256 (asymmetric). The Authorization Server signs with a private key; Resource Servers verify with the public key. Prefer over HS256 (symmetric, requires sharing the secret).
typ Token type Always JWT for standard tokens. Some systems use at+JWT for access tokens per RFC 9068.
kid Key ID Identifies which public key to use for verification. The Authorization Server rotates keys over time; the kid tells the Resource Server which entry in the JWKS endpoint to use.

Part 2: The Payload (Claims)

Decoded payload:

{
  "iss": "https://auth.example.com",
  "sub": "usr-42",
  "aud": "https://api.example.com",
  "exp": 1717170000,
  "iat": 1717166400,
  "jti": "uuid-abcd",
  "scope": "calendar:read email"
}

These are claims — statements the Authorization Server is making about the token. They are defined in RFC 7519 §4.1. See the JWT Claims Reference for a complete table.

ClaimNameMeaning
issIssuerThe Authorization Server that issued this token. Validate this matches the expected AS URL.
subSubjectThe principal the token represents — usually the user's unique ID. In Client Credentials tokens, it's the client_id.
audAudienceWho the token is for. The Resource Server MUST reject any token whose aud does not include its own identifier.
expExpirationUnix timestamp. The token MUST NOT be accepted at or after this time.
iatIssued AtUnix timestamp when the token was issued. Used for age checks.
jtiJWT IDUnique identifier for this token. Used by the AS to enable revocation; can be checked against a denylist.
scopeScopeSpace-separated list of approved OAuth scopes. The Resource Server checks this before serving a request.

Part 3: The Signature

The signature is computed over the encoded header and payload: 

signature = RSA.sign(
  SHA256( base64url(header) + "." + base64url(payload) ),
  authorizationServer.privateKey
)

If anyone modifies even a single byte of the header or payload, the signature will not match when verified — the token is rejected. The private key never leaves the Authorization Server.

How Resource Servers get the public key — JWKS

Every compliant Authorization Server publishes its public keys at a JWKS endpoint (JSON Web Key Set). The Resource Server fetches keys from here, caches them, and uses them for verification.

Discovery document (auto-generated by Duende IS):
GET https://auth.example.com/.well-known/openid-configuration

→ returns JSON including:
"jwks_uri": "https://auth.example.com/.well-known/jwks"

JWKS endpoint lists all current public keys:
GET https://auth.example.com/.well-known/jwks

→ returns JSON like:
{ "keys": [ { "kty": "RSA", "kid": "key-2024-01", "n": "...", "e": "AQAB" } ] }

The kid in the JWT header matches a key in the JWKS list. If the Resource Server sees an unknown kid, it refreshes its key cache — this is how key rotation works transparently.

The Validation Checklist

Accepting a JWT with a valid signature is not enough. A Resource Server must check all of the following. Skipping even one check creates a real vulnerability.

  • 1 Signature — verify using the public key matching the token's kid. Fetch from JWKS if not cached.
  • 2 iss must exactly match the Authorization Server's URL you trust. A token from a different AS (even a valid one) must be rejected.
  • 3 aud must contain this Resource Server's identifier. Without this check, a token issued for API-A can be replayed against API-B.
  • 4 exp must be in the future. Use server time; allow a small clock-skew tolerance (30–60 seconds) for distributed systems.
  • 5 nbf (if present) must be in the past — the token is not yet valid before this time.
  • 6 scope must include the scope(s) required for this specific endpoint. A token with calendar:read must not access a write endpoint.
ASP.NET Core handles this for you When you call AddJwtBearer() in an ASP.NET Core app, the middleware performs all six checks automatically — as long as you configure Authority (→ sets expected issuer and fetches JWKS) and Audience (→ validates aud). Lesson 23 covers the exact configuration.
The confused deputy attack If a Resource Server skips the aud check, an attacker who obtains a valid token for Service A can use it against Service B. Always configure Audience validation. This is the most common JWT misconfiguration in real systems.

Custom Claims

Authorization Servers can add any additional claims they need. These are called private claims in RFC 7519. Duende IdentityServer lets you add them in the Profile Service. When you reach Module 3, you'll implement this.

Examples of custom claims you'll see throughout this course:

ClaimAdded byPurpose
tenantIdEquisoft token generatorIdentifies which tenant the user belongs to
identityRoleEquisoft token generatorThe user's role within the tenant
emailOIDC profile scopeUser's email address (Module 2)
