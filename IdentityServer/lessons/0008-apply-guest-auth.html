Apply: MVC Configuration & Guest Authentication

Lesson 8 of 8 ~25 min Authentication · ASP.NET Core MVC · Cookie Auth

The Apply application is Equisoft's insurance application-filling product. Unlike internal services that always talk to the Identity Gateway, Apply must handle a special case: guests — unauthenticated visitors who fill in a form without ever creating an account. This lesson shows you how the MVC app wires up its authentication stack, and traces the complete guest flow from the first HTTP request to a cookie-authenticated session.

1. How the MVC App Is Configured

Apply is an ASP.NET Core 6+ MVC application. Authentication is set up in two places: Startup.cs delegates to a separate extension method, and the actual scheme registration lives in the authentication infrastructure folder.

The call chain

src/Equisoft.Apply/Startup.cs → ConfigureServices() 
// Startup.cs — two delegation calls
services.ConfigureAuthentication(Configuration, Environment);
services.ConfigureAuthorization();
src/Equisoft.Apply/Infrastructure/Authentication/Extensions/IServiceCollectionExtensions.cs

Inside ConfigureAuthentication(), three schemes are registered:

Cookie default

CookieAuthenticationDefaults

Used for browser sessions. Both regular (OIDC) users and guest users end up with a cookie. The cookie name is Apply.{tenantKey}.

OpenID Connect challenge

OpenIdConnectDefaults

Used as the default challenge scheme — any unauthorised browser request triggers a redirect to the Identity Gateway at https://api.dev.equisoft.com/identity.

JWT Bearer

JwtBearerDefaults

Used for machine-to-machine API calls (service accounts). Authority is the same Identity Gateway.

Custom OIDC Handler The default OpenIdConnectHandler is replaced with IframeAwareOpenIdConnectHandler. When a request comes from inside an iframe (detected via query param popupAuth=true), it redirects to /account/IframeLogin instead of the standard OIDC redirect, opening a popup window for login. This prevents the login page from being trapped inside an iframe.

Cookie authentication events

src/Equisoft.Apply/Infrastructure/Authentication/Functions/CookieAuthenticationFunctions.cs
EventWhat it does
OnSignedInUpserts user record in DB from claims (creates if not exists, updates name/email if changed)
OnSigningOutDeletes stored refresh tokens; for AJAX returns 401 instead of redirect
OnValidatePrincipalOn every request: checks if access token is expired and refreshes it — skipped entirely for guest users
Key insight — guests skip token validation Inside OnValidatePrincipal the very first check is if (isGuest) return;. Guest sessions have no access token at all — they are pure cookie sessions, never touching the Identity Gateway.

2. The Guest Flow — End to End

A guest user is someone who fills in a form without logging in. The flow creates a local-only "user" record in Apply's database, signs the browser in with a cookie, and never involves the Identity Gateway.

Browserno session
→
GuestControllerPOST /guest/forms
→
GuestAuthenticationHelperGetOrCreate…
→
UserServiceCreateGuestUserAsync
→
CookieHttpContext.SignInAsync

Step-by-step

  1. Request arrives at POST /guest/forms with a formCode body parameter. The action is decorated [AllowAnonymous] — the [Authorize] on the class is overridden. No cookie or token is required to reach this endpoint.
  2. GuestController calls _guestAuthenticationHelper.GetOrCreateAndSignInGuestAsync(). If the user is already authenticated (e.g. revisiting after 30 days), the existing user is returned immediately — no duplicate user is created.
  3. GuestAuthenticationHelper creates a new User row via IUserService.CreateGuestUserAsync(): 
    var guestUser = new User {
    ExternalId = Guid.NewGuid().ToString(), // random GUID, no IDG involved
    IsGuest    = true,
    TenantId   = _tenantContext.Tenant.Id,
    CreatedOn  = DateTime.UtcNow
};
    Note: ExternalId is normally required to be non-null for regular users. A special validation branch allows null/empty only when IsGuest = true.
  4. Claims are assembled locally — no round-trip to IDG: 
    new Claim(ClaimTypes.NameIdentifier,      guestUser.Id.ToString())
new Claim("externalobjectidentifier",     guestUser.ExternalId)
new Claim("isGuest",                      "true")         // ← the sentinel
new Claim("tenant",                       tenant.Key)
// Optional if available:
new Claim(ClaimTypes.GivenName,           guestUser.FirstName)
new Claim(ClaimTypes.Surname,             guestUser.LastName)
new Claim(ClaimTypes.Email,               guestUser.Email)
  5. Cookie sign-in via HttpContext.SignInAsync(CookieAuthenticationDefaults.AuthenticationScheme, principal) with IsPersistent = true and an expiry of 30 days. The browser receives a Set-Cookie header: Apply.{tenantKey}, HttpOnly, SameSite=None, Secure.
  6. Form is created in Apply's database via IApplicationService.CreateApplicationAsync() with the guest as both owner and creator. The controller then redirects to /Form/{id}.
GET shortcut: /guest/start?formCode=… The [HttpGet] Start() action simply delegates to the same CreateForm() logic. This allows shareable links instead of requiring a form POST.

3. Guest User Lifecycle

Guest users are ephemeral by design. Three things limit their lifespan:

Cookie expiry — 30 days

Persistent cookie

After 30 days of inactivity the browser cookie expires and the guest can no longer access their form without re-entering via a /guest/start link.

CleanupGuestRecordsHostedService — 90 days

Background service · daily

Runs once a day. Deletes guest User rows (and their forms) that haven't been accessed in 90+ days. This prevents unbounded DB growth from abandoned guest sessions.

6. How It All Fits Together

The critical distinction: a regular user's cookie wraps an OIDC session backed by access/refresh tokens stored in DB, while a guest's cookie wraps a local-only session with no tokens at all. The isGuest claim is the sentinel that lets all downstream code — cookie validation, authorization handlers, services — know which path to take.

7. Knowledge Check

Quiz — Apply Guest Auth

Which claim tells Apply code that the current user is a guest (no IDG session)?

✓ Correct. The custom claim isGuest is set to "true" and is checked in CookieAuthenticationFunctions.OnValidatePrincipal to skip token refresh logic.

What happens during OnValidatePrincipal when isGuest = "true"?

✓ Correct. The code short-circuits with return; for guest users — there is no access token to validate or refresh.

What is the default challenge scheme in Apply and what triggers it?

✓ Correct. The default challenge scheme is OpenID Connect. When an unauthenticated request hits an [Authorize] endpoint, ASP.NET Core challenges with OIDC, redirecting the browser to the Identity Gateway.

After how many days of inactivity does CleanupGuestRecordsHostedService delete a guest user's record?

✓ Correct. 90 days. Note that the cookie itself expires in 30 days, so a guest whose cookie expired can still have their DB record persist for another 60 days before cleanup.
