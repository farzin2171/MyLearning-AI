Registering Clients & APIs in the Identity Gateway

~30 minutes Practical walkthrough Prerequisites: Lesson 2

Every application that requests tokens and every API that accepts them must be registered in the Identity Gateway's configuration database. This lesson covers what needs to be registered, what each field means, and how to avoid the most common mistakes.

Two registrations, not one You register a Client for anything that requests tokens (a frontend app, a backend service, a CLI tool). You register an API Resource + Scope for anything that accepts tokens (an API). A new service typically needs both.

Part 1: Registering a Client

A Client is stored in Duende IS's Clients table (managed by CustomConfigurationDbContext). The most important fields:

FieldDescription
ClientId required Unique identifier for this client. Appears as client_id in OAuth requests and in the JWT. Convention: use the service short-name in lowercase (e.g. my-spa, assistmgntapi).
ClientName optional Human-readable name shown on consent screens. Keep it accurate — end users may see it.
AllowedGrantTypes required Which OAuth flows this client may use. Use authorization_code for SPAs/web apps. Use client_credentials for backend services. Never mix both on one client unless you have a specific reason.
ClientSecrets required for conf. clients Hashed secret for confidential clients (backend services using client_credentials). Public clients (SPAs) must have no secret — they use PKCE instead. Store secrets in Azure Key Vault, never in code.
RedirectUris required for code flow Exact (or wildcard) URIs the IdG may redirect to after login. The WildcardRedirectUriValidator supports patterns like https://*.equisoft.com/*.
PostLogoutRedirectUris optional Where to redirect after logout. Same wildcard support.
AllowedScopes required Which API resources and identity scopes this client may request. Must include openid for any OIDC flow. Add API scope names here (e.g. assistmgntapi). If a client requests a scope not in this list, the token request fails.
AllowedCorsOrigins optional Origins allowed to call the Identity Gateway from a browser (for SPAs). Supports wildcard patterns via WildcardCorsPolicyService.
AllowOfflineAccess optional Set to true to enable refresh tokens. The client must also request the offline_access scope.
RequirePkce security Should be true for all public clients (SPAs, mobile). Should also be true for confidential clients as defence-in-depth. Never set to false for new registrations.
AccessTokenLifetime optional Seconds until the access token expires. Default Duende IS: 3600 (1 hour). Keep short for high-security clients; set higher only for long-running processes.
AbsoluteRefreshTokenLifetime Maximum lifetime of a refresh token regardless of use. After this, the user must log in again. Linked to the Redis session in the Equisoft IdG.

Example: Registering a New SPA (Authorization Code + PKCE)

// In your seeding script or EF migration:
new Client
{
    ClientId   = "my-new-spa",
    ClientName = "My New Application",

    AllowedGrantTypes = GrantTypes.Code,     // Authorization Code only
    RequirePkce       = true,                // Always true for public clients
    // No ClientSecrets — public client

    RedirectUris           = { "https://*.my-app.equisoft.com/callback" },
    PostLogoutRedirectUris = { "https://*.my-app.equisoft.com/logout" },
    AllowedCorsOrigins     = { "https://*.my-app.equisoft.com" },

    AllowedScopes = {
        IdentityServerConstants.StandardScopes.OpenId,   // "openid"
        IdentityServerConstants.StandardScopes.Profile,  // "profile"
        "assistmgntapi",                                 // your API audience
    },

    AllowOfflineAccess           = true,    // enables refresh tokens
    AccessTokenLifetime          = 3600,    // 1 hour
    AbsoluteRefreshTokenLifetime = 2592000, // 30 days
}

Example: Registering a Backend Service (Client Credentials)

new Client
{
    ClientId   = "my-backend-service",
    ClientName = "My Backend Service (machine identity)",

    AllowedGrantTypes = GrantTypes.ClientCredentials,
    RequireClientSecret = true,
    ClientSecrets = { new Secret("hashed-secret-from-key-vault".Sha256()) },

    // No redirect URIs — no user interaction
    // No CORS origins — server-side only

    AllowedScopes = { "userapi", "assistmgntapi" },

    // Claims that will appear in the service token:
    Claims = {
        new ClientClaim("service_isService", "true"),   // marks as service account
        new ClientClaim("service_role", "MyService"),   // role for Authorization Service
    }
}
Don't forget service_isService Service-account tokens must have this claim to pass the IsService policy on management endpoints. Without it, calls to DELETE /api/v1/Cache/permissions and similar endpoints will return 403.

Part 2: Registering an API Resource

An API Resource tells the Identity Gateway about a downstream API — what its audience name is, what scopes it exposes, and which claims it needs in tokens.

FieldDescription
Name required The audience name. This must match the audience in the API's appsettings.json. Appears in the token's aud claim. Convention: use the service short-name (e.g. assistmgntapi).
DisplayName optional Human-readable name for admin UIs.
Scopes required The scopes this API exposes. Clients must list these in AllowedScopes to request them. A scope can represent the whole API (assistmgntapi) or a fine-grained operation (assistantManagement.create).
UserClaims optional Additional user claims to include in the access token when this API is the audience. By default only standard claims are included; list extra ones here (e.g. tenantId, identityRole).

Example: Registering a New API

new ApiResource
{
    Name        = "my-new-service-api",   // must match audience in appsettings
    DisplayName = "My New Service",

    Scopes = { "my-new-service-api" },    // at minimum, one scope = the whole API

    UserClaims = {
        "tenantId",      // Equisoft custom claims to include in access tokens
        "identityRole",
        "tenant",
        "externalobjectidentifier"
    }
}

// Also register the scope itself:
new ApiScope
{
    Name        = "my-new-service-api",
    DisplayName = "Access to My New Service"
}
UserClaims on the ApiResource If you omit tenantId from UserClaims, it will not appear in tokens issued for your API's audience — even though EquisoftTokenResponseGenerator computes it. Always include the Equisoft custom claims your service needs.

Part 3: The Registration Checklist

Use this checklist when onboarding a new service end-to-end. Check each item off as you go.

  • API Resource registered with correct Name (audience) and UserClaims
  • API Scope registered with the same name as the API Resource
  • Client registered for the frontend/service that will call this API
  • AllowedScopes on the client includes the new API scope
  • For SPAs: RequirePkce = true, no client secret, CORS origins set
  • For services: client secret stored in Key Vault, service_isService claim added
  • API's appsettings.json has auth.jwt.authority (IdG URL) and correct audience
  • Token validated end-to-end in dev: call the API with a real token and confirm 200
  • Authorization Service: new role/policy/permission rows added if the service needs fine-grained authz
  • Cache invalidation: confirm that DELETE /api/v1/Cache/permissions works if permissions are managed at runtime

Common Mistakes

SymptomLikely causeFix
invalid_scope error at token endpoint The scope requested by the client is not in AllowedScopes Add the scope to the client's AllowedScopes in the IdG DB
API returns 401 even with a valid token Token aud does not match the API's configured audience Ensure ApiResource.Name in IdG matches auth.jwt.audience in the API's appsettings
tenantId claim missing from token tenantId not listed in ApiResource.UserClaims Add "tenantId" to the ApiResource's UserClaims list
CORS preflight fails on the token endpoint The SPA's origin not in AllowedCorsOrigins Add the origin (with wildcard if needed) to the client's CORS origins
Service account gets 403 on management endpoints service_isService claim missing from client definition Add new ClientClaim("service_isService", "true") to the client's Claims
Redirect URI mismatch after login Exact URI doesn't match — includes trailing slash or query string Check the redirect URI configured in IdG against the exact value the client sends
