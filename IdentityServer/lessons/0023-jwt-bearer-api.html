JWT Bearer API — Protecting an ASP.NET Core API

~25 minutes Module 4 start Prerequisite: Module 3

So far every lesson has been about the issuer side — Duende IS producing tokens. This lesson is about the resource server side: an ASP.NET Core API that receives a Bearer token on every request, validates it, and serves or refuses based on what it finds.

Duende IS (what you built)
  • Issues access tokens
  • Signs them with a private key
  • Publishes the public key at /.well-known/jwks
Your API (this lesson)
  • Receives Authorization: Bearer <token>
  • Fetches the public key from Duende IS
  • Validates the signature, expiry, issuer, audience
  • Reads claims to make authorization decisions

What "JWT Bearer" Means at Runtime

Client → GET /api/calendar Authorization: Bearer eyJhbGci…
Every request carries the access token in the Authorization header
API middleware → GET https://is.example.com/.well-known/openid-configuration
On first request, middleware fetches the discovery doc (then caches it)
API middleware → Fetch JWKS → verify signature → check exp, iss, aud
All validation is local — no round-trip to IS per request after the first
API middleware → Populate HttpContext.User with claims from the token
Your controller reads claims from User — no database call for identity

The public key is fetched once on startup (or first request) and cached. Subsequent requests validate entirely locally — this is why JWT Bearer scales well across many API replicas with no shared session store.

Two Lines to Protect an API

Add one NuGet package and configure two options. That is the entire setup.

Terminal — add the package
dotnet add package Microsoft.AspNetCore.Authentication.JwtBearer
Program.cs — JWT Bearer setup
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        // Where to find the discovery document + JWKS
        options.Authority = "https://localhost:5001";

        // The 'aud' claim the token must carry
        options.TokenValidationParameters.ValidAudience
            = "https://localhost:5001/resources";
    });

builder.Services.AddAuthorization();

// …

var app = builder.Build();

// Order matters: Authentication before Authorization
app.UseAuthentication();
app.UseAuthorization();

Authority
The base URL of your Duende IS. The middleware appends /.well-known/openid-configuration to discover endpoints and the JWKS URI automatically.

ValidAudience
The aud claim the token must contain. Duende IS sets aud to the ApiResource name (if configured) or the ApiScope name. Mismatches cause 401.

Middleware order
UseAuthentication() must come before UseAuthorization(). Swapping them silently breaks everything — a classic gotcha.

What the middleware validates automatically

  • 1
    alg / signature — verified against the JWKS public key from Duende IS
  • 2
    exp — token must not be expired
  • 3
    iss — issuer must match options.Authority
  • 4
    aud — audience must match ValidAudience
  • 5
    nbf — "not before" time must have passed

All five checks happen before your controller code runs. A failed check returns 401 Unauthorized automatically — you write no validation logic.

Protecting Endpoints

Apply [Authorize] to require a valid token. Without it, the endpoint is public. Use [AllowAnonymous] to explicitly opt out on an otherwise protected controller.

Controllers/CalendarController.cs
[ApiController]
[Route("api/[controller]")]
[Authorize]                     // all actions require a valid token
public class CalendarController : ControllerBase
{
    [HttpGet]
    public IActionResult GetEvents()
    {
        // Claims are populated from the token — no DB call needed
        var sub      = User.FindFirstValue("sub");
        var tenantId = User.FindFirstValue("tenant_id");

        return Ok(new { sub, tenantId });
    }

    [HttpGet("public")]
    [AllowAnonymous]            // this one action needs no token
    public IActionResult PublicInfo() => Ok("no auth required");
}

Minimal API (no controllers)

Program.cs — Minimal API equivalent
app.MapGet("/api/calendar", (HttpContext ctx) =>
{
    var sub = ctx.User.FindFirstValue("sub");
    return Results.Ok(new { sub });
})
.RequireAuthorization();    // equivalent to [Authorize]

Requiring a Specific Scope

[Authorize] alone only checks for a valid token. To require a specific scope claim (calendar.read), add a policy:

Program.cs — scope policy
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("CalendarRead", policy =>
        policy.RequireClaim("scope", "calendar.read"));
});
Controller — applying the policy
[Authorize(Policy = "CalendarRead")]
public IActionResult GetEvents() { … }
Scope claim format Duende IS may emit scopes as a space-separated string ("scope": "calendar.read openid") or as an array depending on configuration and token type. The HasScope() extension from Duende.IdentityServer.AccessTokenValidation handles both formats. Alternatively, the Microsoft.Identity.Web package provides scope-policy helpers.

Debugging 401 Responses

The middleware swallows validation errors silently by default. Turn on detailed logging to see exactly what failed:

appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore.Authentication": "Debug"
    }
  }
}

Common causes of 401:

  • Wrong Authority URL — scheme, host, or port mismatch
  • Missing or wrong audience — check the token's aud claim in jwt.ms
  • Middleware order — UseAuthentication after UseAuthorization
  • Expired token — clock skew between services (default tolerance: 5 minutes)
  • Token sent as cookie, not Bearer header — JWT Bearer only reads the Authorization header

Knowledge Check

1. On the first request after API startup, how does the JWT Bearer middleware get the public key to verify the token signature?

2. You add [Authorize] to a controller. A request arrives with a valid, unexpired token — but the aud claim does not match ValidAudience. What does the API return?

3. What is the difference between a 401 and a 403 response from an API protected with JWT Bearer?

4. You call app.UseAuthorization() before app.UseAuthentication() by mistake. What happens?
