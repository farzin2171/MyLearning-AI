The Equisoft Identity Ecosystem

~25 minutes Conceptual + interactive Prerequisites: OAuth 2.0 & OIDC basics

Before writing a single line of authentication code, you need a mental map of the three services that handle identity and authorization across the Equisoft platform. This lesson gives you that map and explains why the responsibilities are split the way they are.

The Three Services at a Glance

Applications.IdentityGateway

Identity Gateway (IdG)

Issues tokens. Knows who you are. Authenticates users via local accounts, Azure AD, Azure AD B2C, SAML providers, and a custom JWT Bearer assertion grant.

Services.Authorization

Authorization Service

Decides what you can do. Translates roles into a flat list of access keys using a policy hierarchy stored in its own database.

Services.AssistantManagement

Assistant Management

A downstream API that uses identity. Shows the standard pattern: accept JWT tokens from IdG, enforce scope-based policies, call other services with service-account tokens.

Key principle Authentication (who are you?) lives in the Identity Gateway. Authorization (what can you do?) lives in the Authorization Service. Downstream APIs trust both via standard JWT validation.

The browser initiates the Authorization Code flow with PKCE. The Identity Gateway shows a login page or redirects to an external provider (Azure AD, SAML IdP, etc.). After successful login, IdG issues an authorization code.

Step 2 — Token issuance: The code exchange

Frontend app
→
IdG /connect/token
→
Access Token + ID Token + Refresh Token

The app exchanges the code for tokens. This is where Equisoft's customisation kicks in: the EquisoftTokenResponseGenerator enriches the access token with two Equisoft-specific claims before returning it:

  • tenantId — looked up from the Tenant Management service via the tenant key in the user's claims
  • identityRole — looked up from the User service (e.g. "Admin", "Advisor")

The API receives the token in the Authorization: Bearer … header. It validates the JWT signature using the Identity Gateway's public keys (fetched via discovery), then enforces scope-based policies using ASP.NET Core's authorization middleware.

Step 4 — Permission check: Asking the Authorization Service

AssistantManagement API
→
GET /api/v1/Authorize?key=…
Authorization Service
→
Redis cache
+ SQL DB

For fine-grained permission checks, the API calls the Authorization Service. It passes the user's token (forwarded as Authorization: Bearer …) and asks "does this user have access key shared.eapp.Lock?" The Authorization Service resolves roles → policies → permissions and returns a binary Allow/Deny.

Step 5 — Service-to-service: Machine identity

AssistantManagement
→
client_credentials grant
IdG /connect/token
→
Service access token
→
User Service

When AssistantManagement needs to call the User Service on its own behalf (not on behalf of a user), it uses the Client Credentials grant. It requests a service-account token from IdG using its client_id and client_secret, then uses that token to call the downstream API.

Why Is Authorization a Separate Service?

New developers often ask: "why not just put roles in the JWT and check them in the API?" There are three reasons for the separation:

  1. Tokens are point-in-time snapshots. A JWT is typically valid for minutes to hours. If a user's role changes, the old token keeps the old role. The Authorization Service queries a live database (with a short Redis TTL), so permission changes take effect almost immediately.
  2. Roles alone are too coarse. The authorization model uses a policy hierarchy: a role maps to a policy, a policy contains permissions, and permissions are organised into a tree of access keys (e.g., shared.eapp.Lock). This cannot fit cleanly into a JWT without bloating it.
  3. External role sources. Permissions can come from Azure AD groups, a custom API, or OIPA computations. The Authorization Service abstracts over all these external role providers so individual APIs do not have to know about them.
Tip — for new devs On your first day, ask your team lead: which external role provider is used in your tenant? This determines where group/role assignments are managed (Azure AD vs. internal DB vs. custom API).

The Identity Gateway Is Not a Simple OIDC Provider

The Identity Gateway is built on Duende IdentityServer v7.3.2 but with significant customisations layered on top. Key things that are non-standard:

FeatureStandard Duende ISEquisoft Customisation
Token enrichment Claims from profile service only EquisoftTokenResponseGenerator adds tenantId and identityRole by calling external services at issuance time
Redirect URIs Exact match WildcardRedirectUriValidator supports wildcard patterns (e.g. https://*.equisoft.com/*)
CORS origins Exact match WildcardCorsPolicyService supports SQL LIKE patterns
Grant types Standard (code, client_credentials, refresh) + custom urn:ietf:params:oauth:grant-type:jwt-bearer for federated authentication
Session management Cookie-based Refresh tokens linked to external IdP sessions in Redis; session validated on every refresh
User provisioning Manual Auto-provisioning on first external login via UserStore.ProvisionUserAsync()
