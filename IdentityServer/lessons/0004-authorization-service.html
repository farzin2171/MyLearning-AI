The Authorization Service: From Token to Permission

~35 minutes Code-heavy + interactive Prerequisites: Lesson 3

You now know what a token contains. This lesson covers what happens after a downstream API asks "but can this user actually do X?" — a question the token alone cannot answer. The Authorization Service bridges that gap.

The Core Question the Authorization Service Answers

Every authorization check in the ecosystem boils down to one HTTP call: 

GET /api/v1/Authorize?key=shared.eapp.Lock
Authorization: Bearer <user's access token>

200 OK         → the user has this permission
403 Forbidden  → the user does not

That is the entire public API surface for permission checking. The complexity lives inside the service — in how it maps a token to a set of access keys. That mapping is what this lesson explains.

The Permission Model: A Three-Layer Hierarchy

The Authorization Service's data model has three entities: Roles, Policies, and Permissions. They form a tree.

Role: "Advisor" (identityRole claim from JWT)
→ Policy: "shared" (top-level policy)
→ Policy: "eapp" (child policy)
→ Permission: "Lock" → access key: shared.eapp.Lock
→ Permission: "Create" → access key: shared.eapp.Create
→ Policy: "reports" (sibling child policy)
→ Permission: "View" → access key: shared.reports.View
→ Policy: "assistants" (top-level policy)
→ Permission: "Manage" → access key: assistants.Manage

The separator is always . (dot). An access key is the full path from root policy down to the leaf permission or role, joined by dots. The stored procedure spGetUserAccessKeys computes this path using a recursive SQL CTE.

The key insight A user does not have permissions directly — they have a Role. The Role maps to one or more Policies. Policies contain Permissions. The full dot-path from root to leaf is the Access Key. Checking a permission means checking whether a specific access key is in the user's computed set.

How the Authorization Service Resolves a User's Permissions

When a token arrives at GET /api/v1/Authorize?key=…, the service runs through these steps internally:

  1. Extract identity from the JWT.
    The DIT Identity middleware populates IIdentityContext. For user tokens: TenantId, ResourceOwnerId (sub), ExternalPrincipalId (the external IdP user ID). For service tokens: IdentityType.Service, service_role claim.
  2. Check the Redis cache.
    Cache key: authz.keys.{userId}. TTL: 300 seconds (5 minutes). If the user's access keys are cached, return immediately — no DB query needed. This is why permission changes can take up to 5 minutes to propagate.
  3. Resolve external roles via providers.
    The service queries the active External Role Providers (configured per tenant in the DB) to find the user's roles. The first provider that returns results wins. Provider types: ClaimBased, MicrosoftGraph, CustomApi, OipaExposedComputation, Assistant. Results are a list of role name strings (e.g. ["Advisor", "SeniorAdvisor"]).
  4. Map external roles to internal roles.
    The ExternalRole table maps provider role names to internal Role entities. A single external role can map to multiple internal roles, or directly to permissions (bypassing roles entirely).
  5. Compute access keys via SQL stored procedure.
    spGetUserAccessKeys(@TenantId, @Roles) uses a recursive CTE to walk the policy tree for each role and returns the flat list of access keys. This is the most expensive step — it's why the cache matters.
  6. Store in cache and answer the query.
    The access key set is stored in Redis. The requested key is checked against the set. Binary answer: in the set → 200 OK, not in the set → 403 Forbidden.

The Stored Procedure: The Heart of Permission Resolution

Understanding spGetUserAccessKeys is essential for debugging permission problems. Here is a simplified version of what it does (the real file is at Services.Authorization\src\Equisoft.AuthorizationService.Data\Scripts\StoredProcedures\spGetUserAccessKeys.sql): 

-- Input: @TenantId (GUID), @Roles (comma-separated role names like 'Advisor,Admin')
-- Output: flat list of access keys

WITH PolicyHierarchy AS (
  -- Anchor: top-level policies (no parent)
  SELECT PolicyId, [Key], [Key] AS FullPath
  FROM   tblPolicy
  WHERE  ParentPolicyId IS NULL AND TenantId = @TenantId AND IsActive = 1

  UNION ALL

  -- Recursive: child policies
  SELECT  p.PolicyId, p.[Key], ph.FullPath + '.' + p.[Key]
  FROM    tblPolicy p
  JOIN    PolicyHierarchy ph ON p.ParentPolicyId = ph.PolicyId
  WHERE   p.IsActive = 1
)

-- Combine: path + permission key = access key
SELECT DISTINCT ph.FullPath + '.' + perm.[Key] AS AccessKey
FROM   PolicyHierarchy ph
JOIN   tblPolicyPermissionMapping ppm ON ppm.PolicyId = ph.PolicyId
JOIN   tblPermission perm             ON perm.PermissionId = ppm.PermissionId
JOIN   tblPolicyRoleMapping prm       ON prm.PolicyId = ph.PolicyId
JOIN   tblRole r                      ON r.RoleId = prm.RoleId
WHERE  r.[Key] IN (SELECT value FROM STRING_SPLIT(@Roles, ','))
  AND  r.IsActive = 1
  AND  perm.IsActive = 1
Debugging tip If a user unexpectedly lacks a permission, run this stored procedure directly in SSMS against the Authorization Service database with the user's tenantId and their role name(s) from the token's identityRole claim. If the expected access key is missing from the results, the role-to-policy mapping is incomplete.

The Evaluate Endpoint: Getting All Keys at Once

The Authorize endpoint answers one key at a time. The Evaluate endpoint returns the full access-key set — useful for caching permissions client-side or populating a UI: 

-- Only accessible to service accounts (service_isService: true)
GET /api/v1/Evaluate
Authorization: Bearer <service account token>

Response: {
  "accessKeys": [
    "shared.eapp.Lock",
    "shared.eapp.Create",
    "shared.reports.View",
    "assistants.Manage"
  ]
}

-- Check if a specific user can access a product (for licensing checks):
GET /api/v1/Evaluate/{userId}?productKey=equisoft-connect
Authorization: Bearer <service account token>

Response: true / false
Important: Evaluate is service-account-only GET /api/v1/Evaluate requires a token with service_isService: true. If you call it with a user token, you get 403 Forbidden. This is by design — users should not be able to enumerate all their own permissions.

External Role Providers: Where Roles Actually Come From

The identityRole claim in the JWT is a shortcut — it's the role that the Identity Gateway knew about at token issuance time. The Authorization Service does not rely solely on it. Instead, it queries External Role Providers configured in its own database per tenant.

Provider TypeWhere roles come fromWhen to use
ClaimBased The user's JWT claims (standard role claim) Simplest setup; roles are already in the token
MicrosoftGraph Azure AD group memberships (via Graph API) Tenant manages roles in Azure AD; groups map to internal roles
CustomApi Any HTTP endpoint returning role names Role management lives in a bespoke service
OipaExposedComputation OIPA system computation results Roles are derived from policy admin data in OIPA
Assistant AssistantManagement service Roles derive from assistant relationships in AssistantManagement

Providers are tried in priority order. The first one that returns any roles wins — the others are skipped. If a provider fails (HTTP error, timeout), the service catches the exception, logs it, and tries the next provider.

Key insight: roles are resolved at request time, not at login time This is the core reason for the separate Authorization Service. If a user's Azure AD group membership changes, the next authorization check (after the cache expires) will reflect that change — no re-login required. The JWT's identityRole is just a hint; the Authorization Service always re-derives from the source of truth.

Cache Invalidation: When Permissions Change Immediately

The 5-minute TTL means permission changes take up to 5 minutes to propagate normally. But there is an explicit cache invalidation endpoint for urgent cases: 

-- Clear all cached permissions for a specific user (service account only):
DELETE /api/v1/Cache/permissions/{executedBy}?executedFor={executedFor}
Authorization: Bearer <service account token with service_isService: true>

-- Clear cached permissions related to an assistant:
DELETE /api/v1/Cache/clear/assistants/{assistantId}
Authorization: Bearer <user token>  ← this one accepts user tokens

AssistantManagement calls the second endpoint whenever it modifies assistant relationships — so that permission changes from assistant assignment take effect immediately. See Services.AssistantManagement\src\AssistantManagementService.Data\External\V1\Clients\IAuthorizationCacheClient.cs.

Practical Walkthrough: Tracing a Permission Check

Here is the full trace of what happens when a user tries to lock an eApp and the API checks shared.eapp.Lock: 

// 1. In the API controller (simplified):
var result = await _authorizationClient.AuthorizeAsync(
    key:   "shared.eapp.Lock",
    token: Request.Headers["Authorization"]  // forwarded user token
);
if (!result.IsAuthorized) return Forbid();

// 2. The Authorization Service receives:
// GET /api/v1/Authorize?key=shared.eapp.Lock
// Authorization: Bearer eyJhbGci... (user token)

// 3. Internally:
// → Extract sub = "user-guid", tenantId = "tenant-guid"
// → Check Redis: authz.keys.user-guid — MISS (first request)
// → Query AssistantExternalRolesProvider (priority 1 for this tenant):
//   POST https://assistmgntapi/v2/... → returns ["Advisor"]
// → Map "Advisor" external role → internal Role "Advisor" (row in tblRole)
// → Run spGetUserAccessKeys("tenant-guid", "Advisor")
// → Returns: ["shared.eapp.Lock", "shared.eapp.Create", "shared.reports.View"]
// → Store in Redis: authz.keys.user-guid → [...] TTL 300s
// → Check: "shared.eapp.Lock" IN result set → YES
// → Return 200 OK

Hands-on: Inspect the Permission Model

On your dev environment, connect to the Authorization Service database and run: 

-- 1. See all active policies and their hierarchy:
SELECT p.Key, parent.Key AS ParentPolicy, p.TenantId
FROM   tblPolicy p
LEFT JOIN tblPolicy parent ON parent.PolicyId = p.ParentPolicyId
WHERE  p.IsActive = 1
ORDER BY p.TenantId, parent.Key, p.Key;

-- 2. See all roles and their policy mappings:
SELECT r.Key AS Role, p.Key AS Policy
FROM   tblRole r
JOIN   tblPolicyRoleMapping prm ON prm.RoleId = r.RoleId
JOIN   tblPolicy p              ON p.PolicyId = prm.PolicyId
WHERE  r.IsActive = 1;

-- 3. Manually compute access keys for a role in your tenant:
EXEC spGetUserAccessKeys
  @TenantId = 'your-tenant-guid-here',
  @Roles    = 'Advisor';

Compare the output of query #3 to what you see in a decoded JWT (Lesson 3 exercise). The access keys computed here are exactly what the Authorization Service caches in Redis and checks against.
