Clients & Resources — Config.cs

~25 minutes Concepts → configuration code Prerequisite: Lesson 18

Every OAuth concept you know has a direct C# representation in Duende IS. They all live in a single static class — Config.cs — that you wire up once during startup. This lesson is about reading and writing that file.

The three things Config.cs must declare What scopes exist (your API surface) · What clients exist (who can ask for tokens) · What identity resources exist (what claims go in the ID token)

The Three Configuration Objects

API surface
ApiScope

A named permission your API enforces. e.g. calendar.read, contacts.write. Clients request scopes; APIs validate them.

ID token claims
IdentityResource

Governs which user claims appear in the ID token. openid is mandatory. profile adds name/picture. Custom ones add anything.

Token requesters
Client

A registered application that can ask for tokens: a web app, SPA, mobile app, or background service. Each has an ID, secret (if confidential), and allowed grant types.

There is also ApiResource, which groups related scopes under one logical API with a shared audience (aud) claim. You can skip it for simple setups — scopes alone are enough to get started. The distinction matters when you have multiple microservices and need each one to accept only its own tokens.

Identity Resources — What Claims Go in the ID Token

Identity resources map an OIDC scope to a set of claims. When a client requests the profile scope, Duende IS knows to include name, given_name, family_name, etc. in the ID token.

Config.cs — IdentityResources
public static IEnumerable<IdentityResource> IdentityResources =>
    new IdentityResource[]
    {
        new IdentityResources.OpenId(),   // "sub" claim — mandatory
        new IdentityResources.Profile(),  // name, given_name, family_name, picture, …
        new IdentityResources.Email(),    // email, email_verified

        // Custom identity resource: your own scope → your own claims
        new IdentityResource(
            name:         "tenant",
            displayName:  "Tenant membership",
            userClaims:   new[] { "tenant_id", "tenant_role" }
        )
    };

The built-in shorthand classes (IdentityResources.OpenId() etc.) are just convenience wrappers — they expand to the same new IdentityResource(…) form. A custom identity resource lets you bundle any set of user claims under one scope name that clients must explicitly request.

API Scopes — Your API's Permissions

API scopes are the permissions your APIs enforce. A client that requests calendar.read gets an access token containing that scope. Your API middleware checks the token carries the right scope before serving the request.

Config.cs — ApiScopes
public static IEnumerable<ApiScope> ApiScopes =>
    new ApiScope[]
    {
        new ApiScope("calendar.read",   "Read calendar events"),
        new ApiScope("calendar.write",  "Create and edit calendar events"),
        new ApiScope("contacts.read",   "Read contacts"),
        new ApiScope("contacts.write",  "Create and edit contacts"),
    };
Naming convention Use resource.action — noun first, then verb. It reads naturally in consent screens ("This app wants to: Read calendar events") and is easy to grep in code.

Clients — Who Can Request Tokens

A Client is a registered entry for each application in your system. Two client types dominate real-world usage:

  • Client Credentials — background service, daemon, API-to-API. No user involved.
  • Authorization Code + PKCE — web app, SPA, or mobile app acting on behalf of a user.

Key Client Properties

Property What it controls Example
ClientId Unique identifier for this application "calendar-web"
ClientSecrets Shared secret (confidential clients only). Store hashed. new Secret("secret".Sha256())
AllowedGrantTypes Which OAuth flows this client may use GrantTypes.Code
AllowedScopes Which API scopes this client may request { "calendar.read", "openid" }
RedirectUris Allowed callback URLs after login (auth code flow only) "https://app/callback"
RequirePkce Enforce PKCE (always true for public clients) true
AllowOfflineAccess Permit refresh tokens (requires offline_access scope) true
RequireClientSecret Set to false for public clients (SPAs, mobile) false

Side-by-Side: Two Client Patterns

Client Credentials — machine-to-machine
new Client
{
    ClientId   = "reports-service",
    ClientName = "Reports Microservice",

    AllowedGrantTypes =
        GrantTypes.ClientCredentials,

    ClientSecrets =
    {
        new Secret(
          "s3cr3t!".Sha256())
    },

    AllowedScopes =
    {
        "calendar.read",
        "contacts.read"
    }
}
Auth Code + PKCE — user-facing web app
new Client
{
    ClientId   = "calendar-web",
    ClientName = "Calendar Web App",

    AllowedGrantTypes =
        GrantTypes.Code,
    RequirePkce = true,
    RequireClientSecret = false,

    RedirectUris =
    {
        "https://app/callback"
    },
    PostLogoutRedirectUris =
    {
        "https://app/signout"
    },
    AllowedScopes =
    {
        IdentityServerConstants
          .StandardScopes.OpenId,
        IdentityServerConstants
          .StandardScopes.Profile,
        "calendar.read",
        "calendar.write"
    },
    AllowOfflineAccess = true
}
Never store secrets in plain text in production. The .Sha256() extension hashes the secret before storage. In production, pull secrets from environment variables or a secrets manager and hash them on the fly.

Wiring Config.cs into Program.cs

All three collections plug into the AddIdentityServer() builder. You saw this skeleton in Lesson 18 — now you know what each method receives:

Program.cs — connecting Config.cs
builder.Services.AddIdentityServer()
    .AddInMemoryIdentityResources(Config.IdentityResources)
    .AddInMemoryApiScopes(Config.ApiScopes)
    .AddInMemoryApiResources(Config.ApiResources)   // optional
    .AddInMemoryClients(Config.Clients)
    .AddDeveloperSigningCredential();               // dev only — replace in prod

AddDeveloperSigningCredential() generates a throwaway RSA key on every startup. It is intentionally unsuitable for production — tokens signed on one run become invalid on the next. In production you provide a real certificate via AddSigningCredential(cert).

The Full Config.cs — Putting It Together

Config.cs — complete example
using Duende.IdentityServer.Models;

public static class Config
{
    public static IEnumerable<IdentityResource> IdentityResources =>
        new IdentityResource[]
        {
            new IdentityResources.OpenId(),
            new IdentityResources.Profile(),
            new IdentityResources.Email(),
        };

    public static IEnumerable<ApiScope> ApiScopes =>
        new ApiScope[]
        {
            new ApiScope("calendar.read",  "Read calendar events"),
            new ApiScope("calendar.write", "Create and edit calendar events"),
        };

    public static IEnumerable<Client> Clients =>
        new Client[]
        {
            // Machine-to-machine: reports service reads calendar data
            new Client
            {
                ClientId          = "reports-service",
                AllowedGrantTypes = GrantTypes.ClientCredentials,
                ClientSecrets     = { new Secret("s3cr3t!".Sha256()) },
                AllowedScopes     = { "calendar.read" }
            },

            // Interactive: user-facing web application
            new Client
            {
                ClientId              = "calendar-web",
                AllowedGrantTypes     = GrantTypes.Code,
                RequirePkce           = true,
                RequireClientSecret   = false,
                RedirectUris          = { "https://localhost:5002/signin-oidc" },
                PostLogoutRedirectUris = { "https://localhost:5002/signout-callback-oidc" },
                AllowOfflineAccess    = true,
                AllowedScopes         =
                {
                    IdentityServerConstants.StandardScopes.OpenId,
                    IdentityServerConstants.StandardScopes.Profile,
                    "calendar.read",
                    "calendar.write"
                }
            }
        };
}

Knowledge Check

1. A nightly reporting service needs to call your calendar API with no user present. Which grant type should its client use?

2. You add calendar.read to ApiScopes in Config.cs but forget to add it to the client's AllowedScopes. What happens when the client requests a token with that scope?

3. What is the difference between an ApiScope and an IdentityResource?

4. Why is RequirePkce = true set on the web-app client but not the machine-to-machine client?

Quick Reference

I want to… I configure…
Define a permission my API enforces ApiScope in ApiScopes
Add user claims to the ID token via a scope IdentityResource in IdentityResources
Register a background service that calls APIs Client with GrantTypes.ClientCredentials
Register a web app that logs users in Client with GrantTypes.Code + RequirePkce = true
Allow refresh tokens (stay logged in) AllowOfflineAccess = true + offline_access in AllowedScopes
Group related scopes under one API audience ApiResource in ApiResources
