UserInfo, Discovery & OIDC in Practice

~25 minutes Conceptual + interactive Prerequisite: Lesson 16

Lesson 16 established the core OIDC contract: scope=openid → ID token in the response. This lesson completes the picture with the three remaining practical pieces: the UserInfo endpoint (getting fresh claims on demand), the discovery document (auto-configuring clients), and RP-Initiated Logout (ending sessions properly).

The UserInfo Endpoint

The UserInfo endpoint is a protected REST endpoint the Authorization Server exposes. The client calls it — with an access token as Bearer auth — to fetch the current claims for the authenticated user. It returns JSON (or optionally a signed JWT).

Request — present the access token:
GET /connect/userinfo HTTP/1.1
Host: auth.example.com
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...

Response — user's current claims as JSON:
HTTP/1.1 200 OK
Content-Type: application/json

{
  "sub": "usr-42",
  "name": "Alice Example",
  "given_name": "Alice",
  "family_name": "Example",
  "email": "alice@example.com",
  "email_verified": true,
  "picture": "https://cdn.example.com/avatars/usr-42.jpg"
}

The claims returned depend on which scopes were granted in the original authorization request. If the client asked for openid profile email, it gets sub, profile fields, and email fields.

ID Token vs. UserInfo — when to use each

SituationUse ID tokenUse UserInfo endpoint
At login — establish who the user is ✓ Available immediately in the token response —
Large user profiles (100+ custom claims) — ✓ Keeps token size small; fetch on demand
Claims that change frequently (name, avatar) — ID token is a snapshot at issuance ✓ Always returns current data
Showing the user's profile page — May be stale ✓ Fetch fresh data on each page load
Making authorization decisions — Neither token is designed for authz — Use the access token + scopes instead
In practice with ASP.NET Core AddOpenIdConnect() has a GetClaimsFromUserInfoEndpoint = true option that automatically calls the UserInfo endpoint after login and merges the claims into the identity. This is the recommended approach — let the middleware handle it rather than calling UserInfo manually.

The Discovery Document

Every OIDC-compliant Authorization Server publishes a machine-readable metadata document at a well-known URL:

GET https://auth.example.com/.well-known/openid-configuration

A client that knows only the Authorization Server's base URL can use this document to auto-configure all its endpoints. This is how ASP.NET Core's AddOpenIdConnect() works — you set Authority = "https://auth.example.com" and the middleware fetches the discovery document on startup.

Key fields in the discovery document

"issuer"

The AS's canonical URL. Must exactly match iss in all tokens.

"authorization_endpoint"

Where to send auth requests (Step 1 of the Auth Code flow).

"token_endpoint"

Where to exchange codes for tokens (back-channel POST).

"jwks_uri"

Where to fetch public keys for token signature verification.

"userinfo_endpoint"

Where to fetch user claims using an access token.

"end_session_endpoint"

Where to redirect users for RP-Initiated Logout.

"scopes_supported"

List of scopes this AS recognizes: ["openid","profile","email",…]

"claims_supported"

List of claims this AS can return: ["sub","name","email",…]

Duende IdentityServer generates this document automatically. You can view the live discovery document for any OIDC server by appending /.well-known/openid-configuration to its base URL — for example, Google's is at accounts.google.com/.well-known/openid-configuration .

RP-Initiated Logout

When a user clicks "Sign out," most developers clear the application's session cookie and redirect to the home page. This is incomplete.

The partial logout problem Clearing your app's cookie only ends the session in your app. The user's session at the Authorization Server (the IdP session cookie) is still active. If they visit another OIDC app sharing the same AS, they are silently signed back in — no login prompt. To users this looks like "logout is broken."

RP-Initiated Logout terminates the IdP session as well. The flow:

1

User clicks Sign Out. The client clears its own session cookie and local tokens.

2

Client redirects the browser to the AS's end_session_endpoint:

GET /connect/endsession?
  id_token_hint=eyJhbGciOiJSUzI1NiJ9... (the ID token from login)
  &post_logout_redirect_uri=https://app.example.com/signed-out
3

The AS ends the IdP session. If front-channel logout is configured, it fires invisible <iframe> requests to all other registered OIDC clients, telling them to end their sessions too (federated single sign-out).

4

AS redirects the browser to post_logout_redirect_uri — your "you've been signed out" page.

id_token_hint is key The id_token_hint tells the AS which user to sign out. Without it, the AS may show a confirmation screen asking "are you sure?" (it doesn't know whose session to end). Store the raw ID token string until logout completes, then discard it.

The Complete OIDC Flow — All Together

Combining Lessons 16 and 17, here is the full OIDC Authorization Code flow: 

1. CLIENT → Browser redirect →
   GET /connect/authorize
     ?response_type=code
     &client_id=my-app
     &redirect_uri=https://app.example.com/callback
     &scope=openid profile email
     &state=abc
     &nonce=xyz
     &code_challenge=...&code_challenge_method=S256

2. User authenticates at AS, approves scopes

3. AS → Browser redirect →
   GET /callback?code=SplxlO&state=abc

4. CLIENT server → AS (back-channel) →
   POST /connect/token
     grant_type=authorization_code
     &code=SplxlO&code_verifier=...

5. AS responds:
   {
     "access_token":  "...",   // → send to APIs
     "id_token":      "...",   // → validate nonce, read sub/name/email
     "refresh_token": "...",
     "expires_in":    3600
   }

6. CLIENT (optional) →
   GET /connect/userinfo
     Authorization: Bearer <access_token>
   ← { "sub": "usr-42", "email": "alice@example.com", ... }

7. LOGOUT: Client → Browser redirect →
   GET /connect/endsession
     ?id_token_hint=<id_token>
     &post_logout_redirect_uri=https://app.example.com/signed-out
