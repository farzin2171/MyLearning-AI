OpenID Connect — Adding Identity to OAuth

~25 minutes Conceptual + interactive Prerequisite: Lessons 12–15

Module 1 gave you the full OAuth 2.0 toolkit: grant types, tokens, flows. Now a counterintuitive question: can you use an OAuth access token to find out who a user is?

The answer is no — and understanding exactly why leads directly to OpenID Connect.

The Authentication Gap in OAuth 2.0

OAuth 2.0 was designed for authorization — delegating access to resources. It was explicitly not designed for authentication — proving who someone is. This distinction matters in practice.

Consider what an access token actually proves:

What an access token proves "The user who owns this account granted this client permission to access these scopes at this time."

It does NOT prove: "The person making this request is the user."

Three concrete problems arise when developers try to use OAuth alone for authentication:

  1. Access tokens may be opaque. The Authorization Server can issue a random string as an access token (for efficiency). The client gets a token it literally cannot decode to extract user identity. OAuth says nothing about what's inside an access token.
  2. Audience mismatch. An access token's aud targets the Resource Server (API). If you send the token to your own app to "prove identity," that's a misuse — your app is not the intended audience.
  3. No standard for identity claims. OAuth doesn't define which claims identify the user. Two Authorization Servers solving the same problem would put the user ID in different places under different names — there's no interoperability.

OpenID Connect (OIDC) solves all three problems with one clean addition: a standardized identity layer built directly on top of OAuth 2.0.

What OpenID Connect Adds

The one-sentence definition OpenID Connect = OAuth 2.0 + scope=openid + an ID token in the token response.

That's really it. OIDC doesn't replace OAuth — it extends it. The Authorization Code flow is identical to what you learned in Lesson 13. The only additions are:

  • A new required scope: openid
  • A new token in the response: the ID token
  • An optional UserInfo endpoint for fetching additional claims
  • A standard discovery document at /.well-known/openid-configuration

The token response with OIDC

Before OIDC (standard OAuth token response):

{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "8xLOxBtZp8"
}

With OIDC (add scope=openid to the authorization request):

{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "8xLOxBtZp8",
  "id_token": "eyJhbGciOiJSUzI1NiJ9..." // ← OIDC addition
}

The ID Token

The ID token is a JWT whose sole purpose is to convey the user's identity to the client application. It answers: "Who just logged in?"

Decoded ID token payload:

{
  "iss":       "https://auth.example.com",
  "sub":       "usr-42",
  "aud":       "my-web-app",        // ← the CLIENT, not an API
  "exp":       1717170000,
  "iat":       1717166400,
  "auth_time": 1717166390,          // when the user actually authenticated
  "nonce":     "n-0S6_WzA2Mj",     // replay-attack prevention
  "name":      "Alice Example",
  "email":     "alice@example.com"
}

Notice the aud: it contains the client ID, not a Resource Server URL. The ID token is for the client to read — it is never sent to APIs.

sub is the stable user identifier The sub (subject) claim is the durable, unique identifier for this user at this Authorization Server. It never changes even if the user updates their name or email. Use sub as the primary key when storing user records locally — not email or name.

OIDC Scopes — Requesting Identity Claims

OIDC defines standard scopes that control which identity claims appear in the ID token and at the UserInfo endpoint. These are defined in the OIDC Core spec §5.4.

openid required Returns: sub
The one mandatory OIDC scope. Without it you're making a plain OAuth request, not an OIDC request.
profile Returns: name, given_name, family_name, middle_name, nickname, picture, website, gender, birthdate, zoneinfo, locale, updated_at
The user's profile information. Most apps use this to show the user's name.
email Returns: email, email_verified
email_verified is important — don't trust an email address for access control unless it's verified.
phone Returns: phone_number, phone_number_verified
Phone number in E.164 format.
address Returns: address (structured)
Postal address. Rarely used in web apps; more common in financial/government contexts.
offline_access Returns: (enables refresh_token)
Not an identity scope, but defined by OIDC. Requests a refresh token so the client can access resources when the user is offline.

ID Token vs. Access Token — Side by Side

Many developers confuse these. They look the same (both JWTs) but serve completely different purposes.

Property ID Token Access Token
Purpose Prove who the user is Authorize access to a resource
Audience (aud) The client (client_id) The Resource Server (API URL)
Who reads it The client application The Resource Server (API)
Sent to APIs? Never — keep it client-side Yes — in the Bearer header
Key claims sub, name, email, nonce scope, aud, client_id
Lifetime Short (used once at login time) Short (15 min – 1 hr; refreshed)
Defined by OpenID Connect Core OAuth 2.0 / RFC 9068
The most common mistake Sending the ID token to an API. The API expects an access token with aud=api-url. The ID token's aud is the client_id — the API will (correctly) reject it.

The nonce — Replay Attack Prevention

Add nonce to the OIDC authorization request alongside state: 

GET /connect/authorize?
  ...
  &scope=openid profile email
  &state=csrf-protection-value
  &nonce=anti-replay-value

The Authorization Server binds the nonce to the ID token. When the client receives the ID token, it verifies the nonce matches what it generated. This prevents an attacker from capturing an ID token from one session and replaying it in another.

  • state = CSRF protection for the redirect
  • nonce = replay protection for the ID token
