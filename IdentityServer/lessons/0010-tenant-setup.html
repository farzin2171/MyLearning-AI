New Tenant Setup: IDG, Services & Data

Lesson 10 of 10 ~30 min Multi-tenancy · IDG Config · Service Wiring

You now understand how authentication works for existing tenants. This lesson answers the question: what must you configure when a brand-new tenant is provisioned? There are three layers — IDG config, application config, and database seeding — and missing any one of them causes cascading failures across services.

1. The Mental Model: Tenant as a JWT Claim

A "tenant" in this ecosystem is fundamentally a short string key (e.g., ACME). Everything else — database rows, hostname mappings, identity providers — serves one purpose: getting that key into a JWT and using it to isolate data at runtime.

How "tenant" gets into a token EquisoftTokenResponseGenerator.AddTenantIdToPayloadAsync() in the Identity Gateway reads the tenant claim from either the user's Subject (set at login via the identity provider's EcosystemTenant property) or from the client's configured claims (for service accounts). It then resolves the key to a GUID via a tenant API call and injects both tenant (key) and tenantId (GUID) into the access token.

2. How Services Resolve the Current Tenant

Different services use different resolution strategies — all of them ultimately read from the same place.

Apply MVC (browser)

MultiTenantMiddleware

Maps hostname to tenant key via Multitenancy.TenantHostnames in appsettings. E.g. acme.app.equisoft.com → ACME.

Apply / API services (JWT)

TenantMiddleware / TenantFilter

Reads the tenant claim from the validated JWT Bearer token. Used for service-to-service calls.

Services.Localization (header)

TenantIdentificationFilter

Reads X-Tenant-Name header (or legacy X-Client-Key). Returns 401 if the header is absent and the JWT has no tenant claim.

3. Data Isolation: EF Core Query Filters

Applications.Apply — src/Equisoft.Apply.Data/Database/ApplicationDbContext.cs — OnModelCreating()

Every entity that belongs to a tenant has an EF Core global query filter applied automatically: 

// Applied to Form, User, Language, CorsOrigin, SignatureSettings, etc.
modelBuilder.Entity<Form>()
    .HasQueryFilter(x => x.TenantId == _tenantContext.Tenant.Id);

This means you never write WHERE TenantId = … in application queries — it is injected invisibly. If ITenantContext is not set for the request, these queries will fail. This is why tenant resolution must happen before any DB call.

New entity checklist Whenever you add a new tenant-scoped entity: add a TenantId FK, register the query filter in OnModelCreating, and seed a row for every existing tenant in the migration. Forgetting any one of these three causes data leakage or empty results.

7. Knowledge Check

Quiz — New Tenant Setup

Where in the IDG codebase is the tenant claim injected into an access token?

✓ Correct. The custom token response generator reads the tenant key (from the user Subject or client claims), resolves it to a GUID via a tenant API call, then injects both tenant and tenantId claims into the token payload.

How does Apply's database ensure one tenant cannot see another tenant's forms?

✓ Correct. ApplicationDbContext.OnModelCreating() registers a HasQueryFilter on every tenant-scoped entity. EF Core appends the predicate automatically to every LINQ query.

A new tenant's browser users log in successfully but every API call between services returns 401. What is most likely missing?

✓ Correct. Service-to-service calls use client credentials tokens from {service}.NEWTENANT clients. If these aren't registered in the IDG, token requests fail with 401 and no service can authenticate to call another on behalf of this tenant.

Which property in identityProviders.json links an Azure B2C identity provider to a specific tenant key?

✓ Correct. EcosystemTenant: "ACME" is what the IDG reads to know which tenant this identity provider belongs to. At login, the IDG uses this to inject the correct tenant claim into the user's token.
