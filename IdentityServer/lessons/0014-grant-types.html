Grant Types — Choosing the Right Flow

~25 minutes Conceptual + interactive Prerequisite: Lesson 13

Not every OAuth scenario involves a browser. A background service calling another service has no user to redirect. A smart TV has no keyboard for typing a password. A long-running job needs to keep working after an access token expires. Each situation has its own grant type — the OAuth term for "the method a client uses to obtain a token."

This lesson maps every important grant type to the scenario it was designed for, so you can always reach for the right one.

Confidential vs. Public Clients

Before grant types make sense, you need one foundational distinction. RFC 6749 §2.1 splits all OAuth clients into two categories based on a single question: can the client keep a secret?

Client typeWhere it runsCan keep a client_secret?Examples
Confidential Server side only Yes — secret never leaves the server ASP.NET Core web app, microservice, background worker
Public On user's device No — code is visible to anyone with the device SPA (React/Angular), mobile app, CLI tool

This matters because some grant types require a client_secret. Public clients must use PKCE instead. Confidential clients use both — PKCE defends against code interception, while the client_secret authenticates the client itself to the token endpoint.

The Grant Types

Authorization Code + PKCE
Use this

Interactive login for any app where a user is present. Covered in depth in Lesson 13.

Client Credentials
Use this

Machine-to-machine. No user present. The client authenticates as itself to call another service.

Refresh Token
Use this

Not a standalone flow — used after Auth Code to silently renew an expired access token without re-authenticating the user.

Device Authorization
Use this

For devices with no browser or limited input (CLI tools, smart TVs). User approves on a secondary device.

Implicit
Deprecated

Returned tokens in the URL. Replaced by Auth Code + PKCE. Never use for new clients.

Resource Owner Password Credentials (ROPC)
Deprecated

App collects the user's raw password. Violates the OAuth contract. Never use for new clients.

Client Credentials Grant

The simplest grant type. Used when a service needs to act as itself — not on behalf of any human user. There is no authorization redirect, no consent screen, no code exchange. The client proves its identity with client_id + client_secret (or a signed JWT assertion) and receives an access token directly.

Service A ── POST /token (client_id + client_secret) ──▶ Auth Server ── access_token ──▶ Service A ── Bearer token ──▶ Service B API

The token request:

POST /connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=invoice-processor
&client_secret=s3cr3t
&scope=payments:read payments:write

The response — notice there is no refresh_token and no id_token:

{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type":   "Bearer",
  "expires_in":   3600
}
Why no refresh token? Confidential service clients can re-authenticate at any time with their credentials. There's no UX cost — no human is waiting. When the token expires, just fetch a new one. Issuing a refresh token to a service adds risk without benefit.

Refresh Token Grant

Access tokens are intentionally short-lived (typically 15 minutes to 1 hour). When one expires, the app has two options:

  1. Ask the user to log in again — poor UX for a routine expiry.
  2. Use the refresh token to silently get a new access token.
Client ── POST /token (refresh_token) ──▶ Auth Server ── new access_token ──▶ Client 
POST /connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=8xLOxBtZp8
&client_id=my-web-app

The Authorization Server validates the refresh token (it is long-lived but still finite and revocable), then returns a fresh access token. Refresh tokens should be stored securely — they are high-value, long-lived credentials.

Refresh token rotation Modern Authorization Servers (including Duende IS) implement refresh token rotation: each use of a refresh token invalidates it and returns a new one. If an old refresh token is ever presented again, the server detects a theft and revokes the entire family. This is why you always store the latest refresh token, not the original.

Device Authorization Grant

Some devices cannot open a browser — a CLI tool, a smart TV, or an IoT device. The Device Authorization Grant (RFC 8628) handles this by splitting the interaction across two devices.

CLI tool ── POST /connect/deviceauthorization ──▶ Auth Server
← receives: device_code, user_code="WDJB-MJHT", verification_uri="https://example.com/activate"
CLI displays: "Visit https://example.com/activate and enter code: WDJB-MJHT"
User's phone ── enters code + logs in ──▶ Auth Server
CLI tool ── polls POST /connect/token (device_code) ──▶ Auth Server ── access_token when approved ──▶ CLI tool

You've seen this flow if you've ever authenticated the GitHub CLI or logged into YouTube on a TV.

Flows to Avoid

Implicit Flow

Returned tokens in the URL fragment (#access_token=…). Superseded by Authorization Code + PKCE for all SPA and mobile use cases. Both the OAuth Security BCP and the OAuth 2.1 draft prohibit it. Duende IS still supports it for legacy systems but disables it for new clients by default.

Resource Owner Password Credentials (ROPC)

The client collects the user's username and password and POSTs them to the token endpoint. This completely defeats the purpose of OAuth: the user hands their credentials to an untrusted third party (the client). It precludes MFA, federated login, and phishing resistance. Even for "internal tools," avoid it — use Authorization Code + PKCE with the Authorization Server as the login page.

Decision Matrix

ScenarioGrant typeWhy
User logs into a web app authorization_code + PKCE User present, browser available, tokens stay out of URLs
Service A calls Service B (no user) client_credentials Machine identity, no user context needed
CLI tool, smart TV, IoT device urn:ietf:params:oauth:grant-type:device_code No browser on device; user approves elsewhere
Access token expired, user already authenticated refresh_token Silent renewal without re-prompting the user
SPA (React, Angular, Vue) authorization_code + PKCE Public client — PKCE replaces client_secret
Mobile app (iOS, Android) authorization_code + PKCE Public client — open system browser, PKCE for security
Legacy internal admin tool asks for password Refactor to Auth Code + PKCE ROPC is deprecated; no MFA, no federation, high phishing risk
