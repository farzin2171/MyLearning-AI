Client Credentials — Service-to-Service Token Acquisition

~25 minutes Implementation patterns Prerequisite: Lesson 23

Lesson 23 showed how your API validates tokens. This lesson is the other side: how a background service, worker, or microservice obtains a token to call that API — and how to do it correctly without hammering Duende IS on every HTTP request.

Client Credentials in one sentence The service posts its own client_id and client_secret to the token endpoint, receives an access token, caches it, and attaches it as a Bearer header on every outbound API call — until the token expires.

The Raw HTTP Exchange

Before looking at library helpers, see what is actually happening at the wire level. This is a standard OAuth 2.0 Client Credentials token request:

POST /connect/token → Duende IS
POST /connect/token HTTP/1.1
Host: localhost:5001
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=reports-service
&client_secret=s3cr3t!
&scope=calendar.read
200 OK response
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9…",
  "expires_in":   3600,
  "token_type":   "Bearer",
  "scope":        "calendar.read"
}

expires_in is in seconds — here 3600 = 1 hour. A well-behaved client caches the token and does not request a new one until approximately 60 seconds before it expires.

Two Implementation Approaches

Manual — IdentityModel
  • Low-level control
  • Good for scripts, tests, CLIs
  • You manage caching and renewal
  • NuGet: IdentityModel
Automatic — Duende.AccessTokenManagement
  • Caching and renewal built in
  • Integrates with IHttpClientFactory
  • Correct for production services
  • NuGet: Duende.AccessTokenManagement

Approach 1 — Manual with IdentityModel

IdentityModel provides extension methods for HttpClient that handle the form-encoded token request and parse the response.

One-off token request (IdentityModel)
using IdentityModel.Client;

var httpClient = new HttpClient();

// 1. Discover the token endpoint from the discovery document
var discovery = await httpClient.GetDiscoveryDocumentAsync(
    "https://localhost:5001");

if (discovery.IsError)
    throw new Exception(discovery.Error);

// 2. Request a token
var tokenResponse = await httpClient.RequestClientCredentialsTokenAsync(
    new ClientCredentialsTokenRequest
    {
        Address      = discovery.TokenEndpoint,
        ClientId     = "reports-service",
        ClientSecret = "s3cr3t!",
        Scope        = "calendar.read"
    });

if (tokenResponse.IsError)
    throw new Exception(tokenResponse.Error);

// 3. Use it
httpClient.SetBearerToken(tokenResponse.AccessToken);
var response = await httpClient.GetAsync(
    "https://localhost:6001/api/calendar");

This works but has a critical flaw: it calls /connect/token every time. A busy service making hundreds of requests per second would overwhelm Duende IS. You must add caching.

Approach 2 — Automatic with Duende.AccessTokenManagement

Duende.AccessTokenManagement integrates token acquisition and caching directly into IHttpClientFactory. You declare a named HTTP client that always carries a valid token — the library handles fetching, caching, and renewal transparently.

Program.cs — setup
// 1. Register the token management service
builder.Services.AddClientCredentialsTokenManagement()
    .AddClient("calendar-api", client =>
    {
        client.TokenEndpoint =
            "https://localhost:5001/connect/token";
        client.ClientId     = "reports-service";
        client.ClientSecret = "s3cr3t!";
        client.Scope        = "calendar.read";
    });

// 2. Register a named HttpClient bound to that client config
builder.Services.AddClientCredentialsHttpClient(
    "CalendarClient",
    configureClient: client =>
    {
        client.BaseAddress =
            new Uri("https://localhost:6001");
    },
    tokenClientName: "calendar-api");

AddClientCredentialsTokenManagement
Registers an in-memory token cache (or Redis if configured). Tokens are cached by client config and refreshed ~60s before expiry.

AddClientCredentialsHttpClient
Creates a named HttpClient whose DelegatingHandler automatically fetches a token and attaches the Bearer header before every outbound request.

ReportsService.cs — using the managed client
public class ReportsService
{
    private readonly HttpClient _client;

    // Inject the named client — already has Bearer header management baked in
    public ReportsService(IHttpClientFactory factory)
    {
        _client = factory.CreateClient("CalendarClient");
    }

    public async Task<CalendarData> GetCalendarDataAsync(string tenantId)
    {
        // No token management here — the library handles it
        var response = await _client.GetAsync($"/api/calendar?tenant={tenantId}");
        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<CalendarData>();
    }
}
What the library does for you On the first call it requests a token, caches it with a TTL based on expires_in, and attaches it as the Bearer header. On subsequent calls it returns the cached token. When ~60 seconds remain, the next call silently requests a fresh token in the background. You write zero caching code.

Token Lifetime and Caching Strategy

t = 0s
Token requested. expires_in = 3600. Cached until t = 3540s (60s buffer).
t = 1s – 3539s
All requests use the cached token. Zero calls to Duende IS.
t = 3540s
Cache considers the token "expiring soon." Next request triggers a background renewal.
t = 3600s
Token expires. Any request that didn't catch the renewal gets a fresh token on demand.

The 60-second buffer exists because clocks between servers are never perfectly synchronized (clock skew). A token that expires at t=3600 on Duende IS may already be rejected by an API whose clock is 30 seconds ahead. The buffer prevents this edge case.

Knowledge Check

1. A reports service calls /connect/token on every HTTP request to your calendar API. What is the primary problem with this?

2. Duende.AccessTokenManagement caches a token with expires_in = 3600. When does it first attempt to fetch a new token?

3. Which grant type does a background service use when there is no user involved in the request?

4. Using AddClientCredentialsHttpClient, where in your service code do you attach the Authorization: Bearer header?
