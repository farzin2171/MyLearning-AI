OAuth Flows in the Ecosystem

~30 minutes Conceptual + code walkthrough Prerequisites: Lesson 1

Equisoft uses three OAuth grant types. Each serves a distinct scenario. Choosing the wrong one is a common new-developer mistake — this lesson gives you the decision rule and shows you exactly where each flow is implemented in the codebase.

The Three Flows at a Glance

Grant Type Who initiates? User interaction? When to use
Authorization Code + PKCE Frontend app / browser Yes — login page Any user-facing application (SPA, MVC web app, mobile app)
Client Credentials Backend service No Service-to-service calls (no user involved)
JWT Bearer Assertion
(urn:ietf:params:oauth:grant-type:jwt-bearer)		 Trusted service No (user was authenticated elsewhere) Federated / SSO scenarios where an external system has already authenticated the user
Decision rule Is a human logging in right now? → Authorization Code + PKCE.
Is a backend service calling another service with no user involved? → Client Credentials.
Has an external system already authenticated the user and you need to exchange their identity for an Equisoft token? → JWT Bearer Assertion.

Flow 1: Authorization Code + PKCE (User Login)

This is the main flow for user-facing applications. The Identity Gateway acts as the Authorization Server. PKCE (Proof Key for Code Exchange) prevents code interception attacks — it is mandatory for all public clients (SPAs, mobile apps) and strongly recommended for confidential clients.

Step-by-step

1. App generates a random `code_verifier` and computes `code_challenge = SHA256(code_verifier)`

2. Browser navigates to:
   GET https://identity.equisoft.com/connect/authorize
     ?response_type=code
     &client_id=my-app
     &redirect_uri=https://app.equisoft.com/callback
     &scope=openid profile assistmgntapi
     &state=abc123
     &code_challenge=<hash>
     &code_challenge_method=S256

3. User logs in. IdG redirects back:
   GET https://app.equisoft.com/callback?code=AUTH_CODE&state=abc123

4. App exchanges the code (server-side):
   POST https://identity.equisoft.com/connect/token
     grant_type=authorization_code
     &code=AUTH_CODE
     &redirect_uri=https://app.equisoft.com/callback
     &client_id=my-app
     &code_verifier=<original_verifier>

5. IdG returns:
   {
     "access_token": "eyJhbGci...",   ← JWT with tenantId, identityRole, scopes
     "id_token":     "eyJhbGci...",   ← OIDC: who the user is
     "refresh_token": "...",          ← stored in Redis, linked to external session
     "token_type": "Bearer",
     "expires_in": 3600
   }
Where to find this in the codebase The client is registered in the Identity Gateway's SQL database (Clients table, managed via the Duende IS Configuration Store). The ClientId, AllowedGrantTypes, RedirectUris, and AllowedScopes are all stored there. See: IdentityGateway/Data/Contexts/CustomConfigurationDbContext.cs

What makes Equisoft's token different

After step 4, before returning the token, EquisoftTokenResponseGenerator fires and calls two external services: 

// Simplified from EquisoftTokenResponseGenerator.cs
private async Task AddTenantIdToPayloadAsync(JwtPayload payload, string tenantKey)
{
    var tenantId = await _tenantClient.GetTenantIdByKeyAsync(tenantKey);
    payload.AddClaim(new Claim("tenantId", tenantId));
}

private async Task AddIdentityRoleToPayloadAsync(JwtPayload payload, string externalUserId)
{
    var role = await _userClient.GetUserRoleAsync(externalUserId);
    payload.AddClaim(new Claim("identityRole", role));
}

This means your token arrives pre-enriched. You do not need to call the Tenant or User services yourself to find out the current user's tenant or role — it is already in the token.

Flow 2: Client Credentials (Service-to-Service)

When a backend service needs to call another service and no user is involved, it uses Client Credentials. The service authenticates with its own identity — a client_id and client_secret — and receives a token that represents the service, not any user.

Step-by-step

POST https://identity.equisoft.com/connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=assistmgntapi
&client_secret=<secret-from-appsettings>
&scope=userapi

Response:
{
  "access_token": "eyJhbGci...",   ← no sub claim — this is a service token
  "token_type": "Bearer",
  "expires_in": 3600
}

Where this is configured in AssistantManagement

// appsettings.json — AssistantManagementService
"ServiceAccounts": {
  "TokenEndpoint": "https://api.dev.equisoft.com/identity/connect/token",
  "PrincipalPrefix": "assistmgntapi",
  "TenantSecrets": {
    "ACME": "the-client-secret-for-this-tenant"
  }
}

The DIT framework's ConfigureTokenClient (called in Startup.cs) wires up the token endpoint and caches service tokens so they are reused until near-expiry.

Important: Service tokens vs. user tokens A service token has no sub claim (no user subject). A downstream API receiving a service token will see IdentityType.Service from IIdentityContext, not IdentityType.User. Make sure your authorization logic handles both. See: AuthorizeServiceAccountFilter in both services — it rejects user tokens on management endpoints.

How the Authorization Service uses service tokens

The Authorization Service's management endpoints (roles, policies, permissions) are protected by the IsService policy, which requires the claim service_isService: true. Only a service account token contains this claim: 

// Startup.cs — Services.Authorization
services.AddAuthorization(options =>
{
    options.AddPolicy("IsService", policy =>
        policy.RequireClaim("service_isService", "true"));
});

Flow 3: JWT Bearer Assertion (Federated SSO)

This is the most unusual grant type and is unique to Equisoft's federation requirements. It is defined in RFC 7523.

When is it used?

When an external, trusted system has already authenticated the user and needs to exchange that user's identity for an Equisoft access token — without showing the user a login page again. The external system asserts the user's identity by presenting a signed JWT as its credential.

Step-by-step

1. External system creates and signs a JWT (assertion):
   {
     "iss": "trusted-external-system",     ← the external system's identity
     "sub": "user@example.com",            ← the user being asserted
     "aud": "https://identity.equisoft.com",
     "exp": 1700000000,
     "iat": 1699999000
   }
   Signed with the external system's private key (registered with IdG as a client certificate)

2. External system calls:
   POST https://identity.equisoft.com/connect/token
     grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
     &client_id=the-federated-client
     &assertion=<signed-jwt>

3. IdG's custom validator fires:
   JwtBearerAuthorizeAssertionExtensionGrantValidator
   → Validates JWT signature against the registered client certificate
   → Provisions the user locally if first login (UserStore.ProvisionUserAsync)
   → Returns a standard Equisoft access token for that user

Where this lives in the codebase

// IdentityGateway/IdentityServer/Validation/
// JwtBearerAuthorizeAssertionExtensionGrantValidator.cs

public class JwtBearerAuthorizeAssertionExtensionGrantValidator
    : IExtensionGrantValidator
{
    public string GrantType =>
        "urn:ietf:params:oauth:grant-type:jwt-bearer";

    public async Task ValidateAsync(ExtensionGrantValidationContext context)
    {

        // 2. Extract subject (user identity)

}
Key insight: this is how SAML-to-OIDC bridging works When a SAML IdP authenticates a user, the Identity Gateway's SAML handler converts the SAML assertion into a local session. The JWT Bearer grant is used when a trusted service — not a browser — needs to bridge an external identity into an Equisoft token programmatically.

Refresh Tokens and Session Linking

The Identity Gateway's refresh token handling is more sophisticated than standard Duende IS. When a token is issued (Flow 1), the refresh token is stored in Redis alongside the external IdP's session information: 

// Stored in Redis under the refresh token key:
UserExternalSessionInformation {
    PrincipalId:           "ext-user-abc123",
    SessionExpirationUtc:  2024-12-01T10:00:00Z,  // external IdP session expiry
    LocalRefreshToken:     "refresh_abc..."
}

On every token refresh, EquisoftTokenRequestValidator checks whether the external IdP session has expired. If it has, the refresh is rejected — the user must log in again. This ensures that if a user's Azure AD session expires, their Equisoft tokens also expire, even if the Equisoft refresh token has not.
