Module 6 Bridge — WorkSpace → Equisoft

~25 minutes Module 6 orientation Prerequisite: Modules 1–5 complete

You built the WorkSpace system: Identity Server, two APIs, a web client, a background service. The Equisoft platform is the same architecture with three differences in name, one difference in the authorization layer, and one additional abstraction over claims. This lesson maps the two systems onto each other so Lessons 1–11 become immediately actionable.

Standard OAuth + one Equisoft layer on top Everything in Modules 1–5 applies to Equisoft unchanged. The one thing Equisoft adds is the Authorization Service — an HTTP decision service that translates roles into flat permission strings called access keys. That is the new concept in Module 6.

The New Thing: Services.Authorization and Access Keys

In the WorkSpace system, your Calendar API made authorization decisions locally using IAuthorizationRequirement handlers. Equisoft externalizes this into a dedicated service. Instead of a local handler, your API makes an HTTP call:

Client
sends Bearer token
→
Your API
validates JWT
→
Services.Authorization
POST /api/v1/Authorize?key=…
→
200 / 403
permit / deny

An access key is a dot-separated string that names a specific permission, e.g. shared.eapp.Lock. The Authorization Service:

  1. Reads the JWT's identityRole and tenantId claims
  2. Looks up the role in its policy hierarchy
  3. Checks whether that role has the requested access key
  4. Returns 200 (permitted) or 403 (denied)

What this replaces vs. what it adds to

WorkSpace — local policy handler
// In CalendarController
var result = await _authz.AuthorizeAsync(
    User, resource,
    new EventOwnerRequirement());

if (!result.Succeeded) return Forbid();
Equisoft — Authorization Service call
// In AssistantManagement controller
var permitted = await _authSvc.AuthorizeAsync(
    accessToken: token,
    key: "shared.eapp.Lock");

if (!permitted) return Forbid();

The JWT Bearer validation step (AddJwtBearer, [Authorize]) is identical in both systems — it comes first. The Authorization Service is an additional layer that runs after JWT validation for fine-grained business rules.

IIdentityContext — Equisoft's Claim Abstraction

Where WorkSpace uses User.FindFirstValue("tenant_id") directly, Equisoft services inject IIdentityContext — an interface that wraps the claim-reading logic and is easier to mock in tests.

WorkSpace pattern vs. Equisoft pattern
// WorkSpace: direct claim access
var tenantId = User.FindFirstValue("tenant_id");
var role     = User.FindFirstValue("role");

// Equisoft: IIdentityContext abstraction
public class MyService
{
    private readonly IIdentityContext _identity;

    public MyService(IIdentityContext identity) => _identity = identity;

    public void DoWork()
    {
        var tenantId = _identity.TenantId;        // wraps "tenantId" claim
        var role     = _identity.IdentityRole;     // wraps "identityRole" claim
        var userId   = _identity.UserId;           // wraps "sub" claim
    }
}

The underlying OAuth/OIDC mechanics are unchanged — IIdentityContext is simply a named wrapper around HttpContext.User.FindFirstValue() with Equisoft's claim names hardcoded.
