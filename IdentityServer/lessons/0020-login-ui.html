The Login UI — Razor Pages & Auth Code Issuance

~25 minutes Architecture + code walkthrough Prerequisite: Lesson 19

You have a running Duende IS instance and a registered client. A user hits your web app, which redirects to /connect/authorize. Now what? Duende IS validates the request — then hands control to your login page. This lesson explains that handoff and what your Razor Pages must do.

Duende IS ships no built-in UI This is intentional. Your login page may need your brand, your MFA flow, your employee directory. Duende IS handles the protocol; you own the UX. The QuickStart UI is a scaffold you can copy into your project to get started immediately.

The Full Login Sequence

Browser → GET /connect/authorize?client_id=…&redirect_uri=…&code_challenge=…
Client redirects the browser to Duende IS with the OIDC request
Duende IS → 302 → /Account/Login?returnUrl=…
IS validates params, sees no active session, redirects to your login page
Browser → POST /Account/Login (username + password)
User fills in credentials; your Razor Page handles the POST
Login Page → HttpContext.SignInAsync(principal)
You validate credentials, build a ClaimsPrincipal, create the auth cookie
Login Page → 302 → returnUrl (back to /connect/authorize)
Redirect back — now the session exists, IS issues the auth code
Duende IS → 302 → redirect_uri?code=abc123&state=…
Auth code delivered to the client's callback URL
Client → POST /connect/token (code + code_verifier)
Client exchanges code for access + ID token (back-channel, browser not involved)

Your code only needs to implement steps 3–5. Everything else — validating the OIDC request, generating the auth code, calling the token endpoint — is Duende IS.

The Three Pages You Must Provide

Pages/Account/Login
Collect username & password. Validate credentials. Call SignInAsync(). Redirect back to returnUrl.
Pages/Account/Logout
Call SignOutAsync(). Use the interaction service to find the post-logout redirect URI and notify the client the session ended.
Pages/Consent
Show the user which scopes the client wants. Only needed if RequireConsent = true on the client (typically for third-party apps).

Internal line-of-business apps typically set RequireConsent = false — the user trusts the app because it's owned by the same org. Public OAuth apps (think "Login with GitHub") always show a consent page.

The Login PageModel in Detail

Two Duende IS services do the heavy lifting in your PageModel:

  • IIdentityServerInteractionService — reads the OIDC request embedded in the returnUrl and records the login result.
  • IEventService — emits structured audit events (successful logins, failures). Optional but strongly recommended.
Pages/Account/Login.cshtml.cs — OnPostAsync (simplified)
public class LoginModel : PageModel
{
    private readonly IIdentityServerInteractionService _interaction;
    private readonly IEventService _events;

    // Injected from your own user store
    private readonly IUserValidator _users;

    public LoginModel(
        IIdentityServerInteractionService interaction,
        IEventService events,
        IUserValidator users)
    {
        _interaction = interaction;
        _events      = events;
        _users       = users;
    }

    [BindProperty] public InputModel Input { get; set; }

    public async Task<IActionResult> OnPostAsync()
    {
        // 1. Retrieve the OIDC context for this login attempt
        var context = await _interaction
            .GetAuthorizationContextAsync(Input.ReturnUrl);

        if (!ModelState.IsValid) return Page();

        // 2. Validate credentials against YOUR user store
        var user = await _users.ValidateAsync(
                       Input.Username, Input.Password);

        if (user == null)
        {
            // Audit event: failed login
            await _events.RaiseAsync(
                new UserLoginFailureEvent(Input.Username,
                    "invalid credentials", clientId: context?.Client.ClientId));

            ModelState.AddModelError("", "Invalid username or password");
            return Page();
        }

        // 3. Build the ClaimsPrincipal for the session
        var isuser = new IdentityServerUser(user.SubjectId)
        {
            DisplayName = user.DisplayName
        };

        // 4. Issue the authentication cookie
        await HttpContext.SignInAsync(isuser);

        // 5. Audit: success
        await _events.RaiseAsync(
            new UserLoginSuccessEvent(user.Username, user.SubjectId,
                user.DisplayName, clientId: context?.Client.ClientId));

        // 6. Redirect back — IS will now issue the auth code
        return Redirect(Input.ReturnUrl ?? "~/");
    }
}
You own the user store Duende IS does not manage users. You provide the validation logic — querying a database, calling an LDAP directory, delegating to an external IdP. This is why Lesson 21 (External Providers) exists as its own topic.

What IdentityServerUser Is

IdentityServerUser is a thin wrapper that sets a fixed AuthenticationScheme ("idsrv") and structures the claims Duende IS needs to issue tokens. The only required field is SubjectId — the stable, unique identifier for this user (the sub claim in every token). 

var isuser = new IdentityServerUser(user.SubjectId)
{
    DisplayName        = user.DisplayName,
    AdditionalClaims   = new[]
    {
        new Claim("tenant_id", user.TenantId),
        new Claim("role",      user.Role)
    }
};
SubjectId — becomes sub in every token. Use a stable DB key, not a username.

DisplayName — shown on consent pages.

AdditionalClaims — extra claims baked into the session. They flow into tokens only if the relevant IdentityResource or ApiScope user-claim list includes them (covered in Lesson 22 — Claims Enrichment).

Getting the QuickStart UI

Rather than writing the Razor Pages from scratch, scaffold the QuickStart UI into your project using the Duende CLI:

Terminal — scaffold QuickStart UI
# Install the template tool once
dotnet tool install -g Duende.IdentityServer.Templates

# Inside your IS project directory
duende new ui

# This adds:
# Pages/Account/Login.cshtml + .cs
# Pages/Account/Logout.cshtml + .cs
# Pages/Consent.cshtml + .cs
# Pages/Diagnostics.cshtml (dev helper)
# Services/TestUserStore.cs (replace with your real store)

The QuickStart UI also includes a TestUserStore with hard-coded users — perfect for development, easy to swap out for your real user repository in production.
