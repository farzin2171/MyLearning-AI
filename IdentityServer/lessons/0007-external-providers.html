Configuring External Identity Providers

~35 minutes Practical + code walkthrough Source: dit-base-config-common + Applications.IdentityGateway

The Identity Gateway can delegate authentication to external providers — Azure AD B2C, corporate Azure AD, any OpenID Connect provider (including Okta), and SAML 2.0 IdPs. This lesson explains how each provider is configured, how the JSON in dit-base-config-common maps to the C# classes in the Identity Gateway, and what happens after the user comes back from the external provider.

The Configuration Source: dit-base-config-common

Provider configurations live in the shared config repository at: 

dit-base-config-common\src\identitygateway\
├── {Dev|QA|Demo|Template}\
│   └── identityProviders.json   ← one per environment
└── clients.json, apiResources.json, ...  (shared across environments)

identityProviders.json is loaded by the Identity Gateway at startup. It maps into the ExternalProviders configuration section via a file path set with the environment variable EQSVC_ExternalProvidersConfigurationFilePath.

Two loading modes Providers can be loaded from file (AddExternalProvidersFromFile — for static environments) or from the database (AddExternalProvidersFromDatabase — when DynamicIdentityProviderEnabled: true in appsettings). This lesson focuses on the file-based approach used in dev/QA.

The JSON Structure: Three Required Fields + Properties

Every provider entry has the same top-level shape, regardless of type: 

// dit-base-config-common/src/identitygateway/Dev/identityProviders.json
{
  "IdentityProviders": [
    {
      "Scheme":      "ACME_AzureB2C",               // ① Unique scheme name
      "DisplayName": "Equisoft IE Dev - Azure AD B2C", // ② Shown on login page
      "Type":        "azureadb2c",                  // ③ Which provider class to use
      "Properties": {                               // ④ Type-specific configuration
        "EcosystemTenant": "ACME",
        ...
      }
    }
  ]
}

The Scheme naming convention

{TenantKey}_{ProviderType} → e.g. ACME_AzureB2C

The Scheme is the identifier the Identity Gateway uses internally to route authentication challenges. It appears as the scheme parameter in /External/Challenge?scheme=ACME_AzureB2C. Keep it unique across all providers in a deployment. The Scheme also appears in audit logs and the user's external session store.

The four provider types

Type valueUse caseUnderlying protocol
"azureadb2c"Consumer-facing login with self-service sign-up, password resetOIDC (Azure B2C-specific)
"azuread"Corporate employee SSO (single Azure AD tenant)OIDC (Microsoft Entra)
"openidconnect"Any generic OIDC IdP — Okta, Auth0, Ping, etc.Standard OIDC
"saml"Legacy enterprise IdPsSAML 2.0
"guest"Unauthenticated guest accessCustom built-in

Provider Configuration by Type

Azure AD B2C (type: "azureadb2c")

Used for consumer-facing tenants. B2C is Microsoft's customer identity platform — it supports self-service sign-up, password reset, and profile editing through "User Flows" (policies). Each tenant that uses B2C needs its own provider entry (different EcosystemTenant), even if they share the same B2C app registration.

Real example from Dev:

{
  "Scheme":      "ACME_AzureB2C",
  "DisplayName": "Equisoft IE Dev - Azure AD B2C",
  "Type":        "azureadb2c",
  "Properties": {
    "EcosystemTenant": "ACME",                       // Equisoft tenant key
    "ClientId":        "27418294-0e76-4fb9-8c8e-153b180ce02d", // B2C App Registration ID
    "Tenant":          "equisoftiedev.onmicrosoft.com",          // B2C tenant domain
    "SignUpSignInPolicyId": "B2C_1_SignUpSignIn",    // User flow (required)
    "PasswordResetPolicyId": "B2C_1_PasswordReset",  // User flow (optional)
    "ClientSecret":    "5F|vF?5XI1M#1j&NKMO/S(96",  // App secret
    "AppUri":          "https://acme.app.dev.equisoft.com/portal" // For password reset redirect
  }
}

Properties reference:

PropertyRequired?Description
EcosystemTenantreqEquisoft tenant key (e.g. ACME). The IdG uses this to add the tenant claim after login.
ClientIdreqAzure app registration GUID from the B2C tenant.
TenantreqB2C domain: {name}.onmicrosoft.com
SignUpSignInPolicyIdreqThe user flow name for login, e.g. B2C_1_SignUpSignIn
PasswordResetPolicyIdoptUser flow for password reset. If set, B2C's "forgot password" button works.
EditProfilePolicyIdoptUser flow for profile editing.
ClientSecretoptRequired for server-side token exchange. Store in Key Vault in production.
AppUrioptThe app's root URL — used as the redirect after password reset.
PriorityoptInteger sort order on the login page (lower = higher up).
FederatedConfiguration.EnabledoptSet to true if this B2C federates to another IdP internally.
Multi-tenant pattern In the Dev config, five tenants (ACME, ACMEDEV, ACMEAUTO, ACME-COSMOS, ACMETEST) each have their own azureadb2c entry — same B2C app registration (ClientId), different EcosystemTenant and AppUri. This is the standard pattern when multiple Equisoft tenants share a single Azure AD B2C directory.

What the IdG computes from this config:

// AzureAdB2CAuthenticationOptions — computed properties:
Authority = "https://equisoftiedev.b2clogin.com/equisoftiedev.onmicrosoft.com/B2C_1_SignUpSignIn/v2.0"
// (built from Tenant + SignUpSignInPolicyId)

AzureAdB2CInstance = "https://equisoftiedev.b2clogin.com/"
// (Tenant domain with "b2clogin.com" replacing "onmicrosoft.com")

Claim transformation on login:

The B2C extension maps emails (array) → email (string) and strips extension_ prefixes from B2C custom attributes. Standard OIDC claims then flow through ExternalController.Callback().

Azure AD / Microsoft Entra (type: "azuread")

Used for corporate employee SSO. The user logs in with their organisational Microsoft account. Requires an App Registration in the client's Azure AD tenant (not a B2C tenant).

Real example from Dev:

{
  "Scheme":      "ACME_AzureAd",
  "DisplayName": "Equisoft - Azure AD",
  "Type":        "azuread",
  "Properties": {
    "EcosystemTenant": "ACME",
    "ClientId":  "d9e948ae-1dbf-46a0-a547-f12ed1fd34e5", // App reg in Equisoft's Azure AD
    "Tenant":    "equisoft.onmicrosoft.com",             // Corporate Azure AD tenant domain
    "ClientSecret": "RDHX8_ZUJZJpK.=t-cj14mA3y5SOpN*W"
  }
}

Properties reference:

PropertyRequired?Description
EcosystemTenantreqEquisoft tenant key.
ClientIdreqApp registration GUID in the corporate Azure AD.
TenantreqCorporate Azure AD domain or tenant GUID. Must match the registered app.
ClientSecretreqApp secret from Azure portal.
AzureAdInstanceoptDefaults to https://login.microsoftonline.com. Override for sovereign clouds.
ScopesoptSpace-separated. offline_access is added automatically for refresh tokens.
CallbackPathoptMust be unique per provider. Defaults to /signin-oidc — override if you have multiple providers.
Authority computed automatically: https://login.microsoftonline.com/equisoft.onmicrosoft.com (from AzureAdInstance + Tenant)

OpenID Connect / Okta (type: "openidconnect")

This is the generic OIDC provider. Use it for Okta, Auth0, Ping Identity, or any IdP that supports standard OpenID Connect. You provide the Authority URL and the IdG fetches the discovery document automatically.

Example — Okta configuration:

{
  "Scheme":      "ACME_Okta",
  "DisplayName": "Okta SSO",
  "Type":        "openidconnect",
  "Properties": {
    "EcosystemTenant": "ACME",

    // ① The Okta authorization server URL (your Okta org + auth server path):
    "Authority":     "https://your-org.okta.com/oauth2/default",

    // ② App registration values from Okta admin:
    "ClientId":     "0oa1b2c3d4e5f6g7h8i9",
    "ClientSecret": "the-client-secret-from-okta",

    // ③ Scopes to request (openid profile email are the minimum):
    "Scopes": "openid profile email",

    // ④ Callback path — must be unique (register this in Okta as redirect URI):
    "CallbackPath": "/signin-okta",

    // ⑤ Fetch all user attributes from UserInfo endpoint:
    "GetClaimsFromUserInfoEndpoint": "true",

    // ⑥ Remove Okta-internal claims that could conflict with Equisoft's claim space:
    "ClaimsToRemove": ["ver", "at_hash", "nonce", "idp", "amr"]
  }
}

Properties reference:

PropertyRequired?Description
AuthorityreqThe OIDC issuer URL. For Okta: https://{org}.okta.com/oauth2/default. The IdG appends /.well-known/openid-configuration to discover endpoints.
ClientIdreqClient ID from the OIDC provider's app registration.
ClientSecretreqClient secret from the provider.
EcosystemTenantreqEquisoft tenant key.
ScopesoptSpace-separated. Include at least openid profile email. Add provider-specific scopes as needed.
CallbackPathoptMust be unique and registered as a redirect URI with the provider.
GetClaimsFromUserInfoEndpointoptDefault: true. Fetches the full user profile from /userinfo after token validation.
ClaimsToRemoveoptArray of claim types to strip before they enter Equisoft's claim pipeline. Use to remove provider-specific noise.
ApplyUrlDecodeoptDefault: false. Set to true if the IdP URL-encodes the OAuth state parameter (some Okta configurations do this).
Watch out for claim collisions Some OIDC providers (Okta included) return claims that start with client_ or contain service_. The Identity Gateway will reject the login with an exception if these reach the claim pipeline. Always populate ClaimsToRemove to strip them. Source: ExternalController.cs — the forbidden-claim check at line 164.

Registering the callback URI in Okta:

The redirect URI you register in the Okta app must exactly match:

https://{identity-gateway-host}/External/Callback

And the callback path in the provider config must match what you register as the Sign-in redirect URI in the Okta app configuration. The actual ASP.NET Core callback route is built as {CallbackPath} on the IdG host.

SAML 2.0 (type: "saml")

For enterprise IdPs that do not support OIDC (older on-premise identity systems, legacy insurance platforms). The Identity Gateway acts as the SAML Service Provider (SP). Uses ComponentSpace SAML2 library.

Real example from Dev:

{
  "Scheme":      "ACME_SAML2",
  "DisplayName": "UMS - SAML 2.0",
  "Type":        "saml",
  "Properties": {
    "EcosystemTenant": "ACME",
    "PartnerIdentityProviderConfiguration": {
      "Name":                      "urn:insuranceElements:IdentityProvider", // IdP Entity ID
      "SignAuthnRequest":           true,      // Sign the AuthnRequest
      "WantSamlResponseSigned":     true,      // Require signed Response
      "WantAssertionSigned":        false,
      "WantAssertionEncrypted":     false,
      "SingleSignOnServiceUrl":     "http://06.qa.equisoft.com/IdentityProvider/Public/Web/SAML/signin",
      "SingleLogoutServiceUrl":     "http://06.qa.equisoft.com/IdentityProvider/Public/Web/SAML/signout",
      "SingleSignOnServiceBinding": "urn:oasis:names:tc:SAML:2.0:bindings:HTTP-Redirect",
      "PartnerCertificates": [
        { "String": "LS0tLS1CRU..." }   // base64-encoded IdP signing cert
      ]
    },
    "Extensions": {
      "environmentName": "EIS_DEV"  // Custom XML extension added to AuthnRequest
    }
  }
}

Key fields:

FieldDescription
NameThe IdP's SAML Entity ID URI. Get this from the IdP's metadata XML.
SignAuthnRequestIf true, IdG signs requests with its SP certificate. The IdP must have IdG's public cert.
WantSamlResponseSignedRequire the IdP to sign its response. Strongly recommended.
SingleSignOnServiceUrlThe IdP's SSO endpoint URL (from IdP metadata).
PartnerCertificatesBase64-encoded PEM certificates used by the IdP for signing. Paste the cert string here.
ExtensionsCustom key-value pairs added as XML extensions to the AuthnRequest (IdP-specific custom data).

Guest (type: "guest")

A special built-in provider for unauthenticated guest access. Users authenticated through this provider get the isGuest: true claim and have their identityRole forced to "Guest" by the token generator (no User Service call needed).

{
  "Scheme":      "ACME_Guest",
  "DisplayName": "Guest",
  "Type":        "guest",
  "Properties": {
    "EcosystemTenant": "ACME",
    "Roles": "Guest"    // Comma-separated role names assigned to guest users
  }
}

What Happens After the User Authenticates: The Callback Flow

Every provider type — whether B2C, Azure AD, Okta, or SAML — ends up at the same callback handler: ExternalController.Callback(). Understanding this is key to diagnosing login failures.

1
External auth completes.
The provider redirects to https://idg.host/External/Callback. The external principal is temporarily stored in a cookie keyed by the provider scheme.
2
Claim extraction and normalisation.
FindUserFromExternalProviderAsync() extracts the unique user identifier: first tries sub, falls back to nameidentifier. Applies claim type remapping: oid → externalobjectidentifier.
3
Tenant claim injection.
TenantAccessor.GetAsync(providerScheme) looks up the EcosystemTenant from the provider's config (e.g. ACME) and adds it as the tenant claim.
4
User lookup or auto-provisioning.
UserStore.FindByExternalProviderAsync(scheme, userId) looks up the user in the local UserDb. If not found, UserStore.ProvisionUserAsync() creates a new user record. No manual user creation is needed for first-time logins.
5
Session storage.
External session data (OIDC refresh token or SAML session expiry) is stored in Redis keyed by session ID. This is later checked on every token refresh.
6
IdentityServer cookie issued.
The user is now authenticated in the Identity Gateway. The Authorization Code flow continues — the IdG redirects back to the application with the auth code.

Claim Mapping Reference

Each provider transforms claims slightly differently. Here is what arrives in Equisoft's claim pipeline after each provider's OnTokenValidated event:

ProviderExternal claim → Equisoft claimNotes
Azure B2C emails[0] → email
oid → externalobjectidentifier
extension_X → X		 B2C returns email as an array; the extension strips the prefix from custom attributes
Azure AD oid → externalobjectidentifier Duplicate name claims removed; standard OIDC claims pass through
OpenID Connect (Okta) sub → externalobjectidentifier (if no oid) Claims in ClaimsToRemove are stripped; everything else passes through
SAML 2.0 nameidentifier → used as external user ID ComponentSpace handles assertion parsing; SAML attributes become claims
All providers Provider scheme → tenant (via TenantAccessor)
oid / objectidentifier → externalobjectidentifier		 Applied in ExternalController after provider-specific processing
Forbidden claims The Identity Gateway throws if any claim coming from an external provider starts with client_ or service_, unless the claim is in the explicit allow-list (tenantId and identityRole). This protects against spoofing. Always use ClaimsToRemove on OpenID Connect providers to strip any offending claims. Source: ExternalController.cs

How Clients Are Restricted to Specific Providers

In clients_common.json, each client can list which provider schemes it allows via IdPRestrictions. When a user tries to log in with a different provider, the IdG rejects the authorization request before showing a login page: 

// dit-base-config-common/src/identitygateway/Dev/clients_common.json
{
  "ClientId": "my-spa-app",
  "IdPRestrictions": [
    "ACME_AzureB2C",    // only these schemes are allowed for this client
    "ACME_AzureAd"
  ]
}

The Properties map on a client can also store a per-tenant default provider mapping, used when the application needs to auto-select the right scheme for a given tenant: 

"Properties": {
  "ACME":       "ACME_AzureB2C",
  "ACMEDEV":    "ACMEDEV_AzureB2C",
  "ACMETEST":   "ACMETEST_AzureB2C"
}

Hands-on: Add a New Provider

Follow these steps to add a new OpenID Connect (Okta-style) provider for a new tenant:

  1. Open dit-base-config-common\src\identitygateway\Dev\identityProviders.json
  2. Add a new entry following the openidconnect pattern above
  3. Set EcosystemTenant to the new tenant's key (e.g. NEWCO)
  4. Set Scheme to NEWCO_Okta
  5. Set Authority to the Okta org URL + /oauth2/default
  6. Set CallbackPath to a unique path e.g. /signin-newco-okta
  7. Register that callback path as a redirect URI in the Okta app
  8. Add NEWCO_Okta to the relevant client's IdPRestrictions in clients_common.json
  9. Restart the Identity Gateway and test the login flow
