Policy-Based Authorization

~25 minutes Fine-grained enforcement Prerequisite: Lesson 23

[Authorize] alone only asks "is there a valid token?" Real applications need more: "does this token allow this specific action?" ASP.NET Core's authorization framework answers that question through policies — named rules evaluated against the current user's claims.

Three tiers, one framework Every authorization check in ASP.NET Core — from a simple [Authorize(Roles="Admin")] to a multi-step business rule — goes through the same pipeline: requirements → handlers → Succeed / Fail.

The Three Tiers

Simple
Built-in policy builder methods: RequireClaim, RequireRole, RequireAssertion. Zero boilerplate — good for scope checks, role checks.
Custom requirement
Implement IAuthorizationRequirement + AuthorizationHandler<T>. Reusable, testable, can inject services. Good for multi-claim rules and business logic.
Imperative
Inject IAuthorizationService and call AuthorizeAsync anywhere — inside a method body, a service class, or a Razor Page. Good when you need to check authorization on a specific resource object.

Tier 1 — Simple Policies

Program.cs — simple policy declarations
builder.Services.AddAuthorization(options =>
{
    // Must have the calendar.read scope claim
    options.AddPolicy("CanReadCalendar", policy =>
        policy.RequireClaim("scope", "calendar.read"));

    // Must have any of these role claim values
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireRole("admin", "superadmin"));

    // Arbitrary lambda — full claim access
    options.AddPolicy("ActiveTenant", policy =>
        policy.RequireAssertion(ctx =>
            ctx.User.HasClaim("tenant_status", "active")));

    // Combine rules — all must pass
    options.AddPolicy("TenantCalendarRead", policy =>
    {
        policy.RequireClaim("scope", "calendar.read");
        policy.RequireAssertion(ctx =>
            ctx.User.HasClaim(c => c.Type == "tenant_id"));
    });
});
Controller — applying policies
[ApiController]
[Route("api/[controller]")]
[Authorize(Policy = "CanReadCalendar")]   // applies to whole controller
public class CalendarController : ControllerBase
{
    [HttpGet]
    public IActionResult GetEvents() { … }

    [HttpDelete("{id}")]
    [Authorize(Policy = "AdminOnly")]      // stricter policy on one action
    public IActionResult DeleteEvent(int id) { … }
}

Tier 2 — Custom Requirements and Handlers

For reusable, service-injected logic, implement the two-class pattern: a requirement (data) and a handler (logic).

Step 1 — The Requirement (data only)
// A requirement is a data class — no logic.
// It parameterises the rule.
public class MinimumTenantTierRequirement
    : IAuthorizationRequirement
{
    public string MinimumTier { get; }

    public MinimumTenantTierRequirement(string tier)
    {
        MinimumTier = tier;
    }
}
Step 2 — The Handler (logic + DI)
public class TenantTierHandler
    : AuthorizationHandler<
        MinimumTenantTierRequirement>
{
    private readonly ITenantService _tenants;

    public TenantTierHandler(ITenantService t)
        => _tenants = t;

    protected override async Task
        HandleRequirementAsync(
          AuthorizationHandlerContext ctx,
          MinimumTenantTierRequirement req)
    {
        var tenantId = ctx.User
            .FindFirstValue("tenant_id");

        var tier = await _tenants
            .GetTierAsync(tenantId);

        if (tier >= req.MinimumTier)
            ctx.Succeed(req);
        // Do not call ctx.Fail() unless you
        // want to block other handlers too
    }
}
Program.cs — register and use
// Register the handler in DI
builder.Services.AddSingleton<
    IAuthorizationHandler, TenantTierHandler>();

// Use the requirement in a policy
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("ProfessionalTier", policy =>
        policy.AddRequirements(
            new MinimumTenantTierRequirement("professional")));
});

The handler decision pipeline

All handlers
run in parallel
→
Any handler
calls ctx.Fail()?
→ yes →
403 Forbidden
placeholder
→
All requirements
have a Succeed?
→ yes →
Access granted
→ no →
403 Forbidden

A handler that neither calls Succeed nor Fail is neutral — it leaves the decision to other handlers. Call Fail only when you want to hard-block access regardless of what other handlers decide.

Tier 3 — Imperative Authorization

When the authorization decision depends on the specific resource being accessed (not just the user's claims), inject IAuthorizationService and check inside the action body:

Imperative check inside an action
[ApiController]
[Route("api/[controller]")]
[Authorize]   // still require a valid token
public class CalendarController : ControllerBase
{
    private readonly IAuthorizationService _authz;
    private readonly ICalendarRepo _repo;

    public CalendarController(
        IAuthorizationService authz,
        ICalendarRepo repo)
    {
        _authz = authz;
        _repo  = repo;
    }

    [HttpDelete("{id}")]
    public async Task<IActionResult> DeleteEvent(int id)
    {
        var calEvent = await _repo.GetAsync(id);
        if (calEvent == null)
            return NotFound();

        // Check: does THIS user own THIS event?
        var result = await _authz.AuthorizeAsync(
            User, calEvent, "EventOwner");

        if (!result.Succeeded)
            return Forbid();

        await _repo.DeleteAsync(id);
        return NoContent();
    }
}

Why imperative?
You cannot know at attribute-registration time which specific event a user is deleting. You need the loaded resource object to check ownership. Imperative authorization passes the resource as the second argument to AuthorizeAsync.

Resource-based handler
The "EventOwner" policy's handler receives both the ClaimsPrincipal (the user) and the resource (the CalendarEvent object). It can compare ctx.User.FindFirstValue("sub") against resource.OwnerId.

Knowledge Check

1. A handler calls neither ctx.Succeed(req) nor ctx.Fail(req). What effect does this have on the authorization decision?

2. An action is decorated with two policies: [Authorize(Policy = "CanRead")] at the controller and [Authorize(Policy = "AdminOnly")] at the action. A user satisfies CanRead but not AdminOnly. What is the response?

3. When should you use imperative authorization (IAuthorizationService.AuthorizeAsync) instead of [Authorize(Policy = "…")]?

4. What is the role of the IAuthorizationRequirement class in the custom handler pattern?
