Duende IdentityServer — Project Setup & Architecture

~30 minutes Architecture + code walkthrough Prerequisite: Modules 1–2

Modules 1 and 2 taught you the concepts: delegation, authorization code flows, ID tokens, discovery documents. Now those concepts become configuration code. Duende IdentityServer is an OpenID Connect Provider and OAuth 2.0 Authorization Server implemented as ASP.NET Core middleware. Everything in the spec — token endpoints, JWKS, discovery document, consent UI — is generated by the library once you configure it.

Module 3 approach These lessons show you the code you'll write and explain why each piece exists. You're building your mental model now. Module 5 (the Capstone) is where you build and run the full system.

Duende IS is its own deployable process. It is not embedded inside your API or web app — it runs as an independent service that everything else trusts. Your APIs validate tokens by fetching the JWKS from Duende IS. Your web apps redirect to Duende IS for login.

Project Setup

A Duende IS host is just a standard ASP.NET Core web project with the NuGet package added:

dotnet new web -n IdentityServer
cd IdentityServer
dotnet add package Duende.IdentityServer

For a full quickstart with pre-built UI (login/logout/consent pages), Duende provides a template: 

dotnet new install Duende.IdentityServer.Templates
dotnet new isinmem -n IdentityServer

The Three Configuration Objects

Before writing Program.cs, you need to understand the three objects that define everything your Authorization Server knows:

Maps to OIDC scopes
IdentityResource

Defines an OIDC scope and the claims it adds to the ID token. You must register openid at minimum.

Maps to OAuth scopes
ApiScope

Defines a scope for API authorization. The scope claim in the access token is populated from these.

Maps to OAuth clients
Client

A registered application. Controls grant types, redirect URIs, client secrets, and which scopes it may request.

The mapping you already know IdentityResource = OIDC scopes from Lesson 16 (openid, profile, email) → appear in the ID token. ApiScope = OAuth scopes from Lesson 12 (calendar:read, payments.write) → appear in the access token.

Config.cs — the complete minimal configuration

using Duende.IdentityServer.Models;

public static class Config
{
    // ── OIDC identity scopes ──────────────────────────────────────────
    public static IEnumerable<IdentityResource> IdentityResources =>
        new IdentityResource[]
        {
            new IdentityResources.OpenId(),    // "openid" scope → sub claim
            new IdentityResources.Profile(),   // "profile" scope → name, given_name, etc.
            new IdentityResources.Email(),     // "email" scope → email, email_verified
        };

    // ── API scopes (appear in access token "scope" claim) ────────────
    public static IEnumerable<ApiScope> ApiScopes =>
        new ApiScope[]
        {
            new ApiScope("weather.read",  "Read weather forecasts"),
            new ApiScope("weather.write", "Submit weather reports"),
        };

    // ── Registered clients ───────────────────────────────────────────
    public static IEnumerable<Client> Clients =>
        new Client[]
        {
            // Interactive web app — Authorization Code + PKCE
            new Client
            {
                ClientId     = "webapp",
                ClientSecrets = { new Secret("secret".Sha256()) },
                AllowedGrantTypes = GrantTypes.Code,
                RequirePkce  = true,

                RedirectUris           = { "https://localhost:5001/signin-oidc" },
                PostLogoutRedirectUris = { "https://localhost:5001/signout-callback-oidc" },
                FrontChannelLogoutUri  =   "https://localhost:5001/signout-oidc",

                AllowedScopes = { "openid", "profile", "email", "weather.read" }
            },

            // Background service — Client Credentials
            new Client
            {
                ClientId      = "weatherservice",
                ClientSecrets = { new Secret("servicesecret".Sha256()) },
                AllowedGrantTypes = GrantTypes.ClientCredentials,
                AllowedScopes = { "weather.read", "weather.write" }
            }
        };
}

Notice how the two clients reflect exactly the lessons on grant types: the web app uses GrantTypes.Code (Authorization Code + PKCE, Lesson 13); the background service uses GrantTypes.ClientCredentials (Lesson 14).

Program.cs — Wiring It Together

var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddIdentityServer()
    .AddInMemoryIdentityResources(Config.IdentityResources)
    .AddInMemoryApiScopes(Config.ApiScopes)
    .AddInMemoryClients(Config.Clients)
    .AddDeveloperSigningCredential();  // dev-only: generates a temp RSA key pair

var app = builder.Build();

app.UseIdentityServer();  // activates all OIDC/OAuth endpoints

app.Run();
MethodWhat it adds
AddInMemoryIdentityResourcesLoads IdentityResources from your config list (dev/learning use)
AddInMemoryApiScopesLoads ApiScopes from your config list
AddInMemoryClientsLoads Clients from your config list
AddDeveloperSigningCredentialGenerates a temporary RSA key pair for signing tokens — never use in production
UseIdentityServer()Registers middleware that handles /connect/authorize, /connect/token, /connect/jwks, etc.
In-memory stores are for development only AddInMemory* stores configuration in RAM. The data is lost on restart, and you cannot change it without redeploying. For production, use Entity Framework Core stores (AddConfigurationStore, AddOperationalStore).

Verifying It Works: The Discovery Document

Once the server is running, the fastest sanity check is the discovery document (Lesson 17). Navigate to: 

https://localhost:5000/.well-known/openid-configuration

You should see JSON listing your configured endpoints:

{
  "issuer":                "https://localhost:5000",
  "authorization_endpoint":"https://localhost:5000/connect/authorize",
  "token_endpoint":        "https://localhost:5000/connect/token",
  "userinfo_endpoint":     "https://localhost:5000/connect/userinfo",
  "jwks_uri":              "https://localhost:5000/.well-known/jwks",
  "end_session_endpoint":  "https://localhost:5000/connect/endsession",
  "scopes_supported":      ["openid","profile","email","weather.read","weather.write"],
  ...
}

If this loads, your Authorization Server is live and will accept OAuth/OIDC requests. The scopes_supported list shows your IdentityResources and ApiScopes registered correctly.
