External Providers — Federating Google & Azure AD

~25 minutes Architecture + code walkthrough Prerequisite: Lesson 20

Your login page currently validates users against a local store. An external provider lets a user prove their identity through a third-party — Google, Microsoft Entra ID, GitHub — so your IS can issue tokens without ever holding a password. This is the pattern behind every "Log in with Google" button.

Key insight: Duende IS becomes the relying party From the external provider's perspective, your Duende IS instance is just another OAuth client. From your clients' perspective, nothing changes — they still get tokens from your IS. You translate between the two.

The Full Federation Flow

Browser → GET /Account/Login?returnUrl=…
User arrives at your IS login page
Browser → GET /ExternalLogin?provider=Google&returnUrl=…
User clicks "Log in with Google" — your page issues an OAuth Challenge
Google → User authenticates at accounts.google.com
Browser redirected to Google's consent/login page
Google → 302 → /ExternalLogin/Callback?code=…&state=…
Google sends the auth code back to your IS callback URL
IS Callback → AuthenticateAsync("Google") → external claims
Your callback page exchanges the code and reads Google's claims
IS Callback → Map claims → find/create local user → SignInAsync(isuser)
You translate Google's identity into your IdentityServerUser
IS Callback → Redirect → returnUrl → auth code issued
Same ending as a local login — IS issues your tokens

Steps 3–4 happen entirely between the user's browser and Google — your IS never sees the password. Steps 5–7 are entirely your code.

Step 1 — Register the External Provider in Program.cs

External providers plug in as ASP.NET Core authentication handlers alongside Duende IS. Each provider has its own NuGet package.

Program.cs — Google
builder.Services
    .AddAuthentication()
    .AddGoogle("Google", options =>
    {
        options.SignInScheme =
            IdentityServerConstants
            .ExternalCookieAuthenticationScheme;

        // From Google Cloud Console
        options.ClientId     = "…";
        options.ClientSecret = "…";
    });
Program.cs — Azure AD / Entra ID
builder.Services
    .AddAuthentication()
    .AddOpenIdConnect("aad", "Azure AD", options =>
    {
        options.SignInScheme =
            IdentityServerConstants
            .ExternalCookieAuthenticationScheme;

        options.Authority  = "https://login.microsoftonline.com/{tenantId}/v2.0";
        options.ClientId   = "…";
        options.ClientSecret = "…";
        options.ResponseType = "code";
        options.Scope.Add("openid");
        options.Scope.Add("profile");
        options.Scope.Add("email");
    });
SignInScheme must be ExternalCookieAuthenticationScheme This tells ASP.NET Core to store the external identity in a temporary cookie (not the main IS session cookie) until your callback page processes it. Forgetting this is the most common setup mistake.

The Login Page Button

Each registered provider appears as a button on your login page. The button issues a challenge — an HTTP 302 to the provider's authorize endpoint:

Login.cshtml.cs — OnGetExternalLogin
public IActionResult OnGetExternalLogin(string provider, string returnUrl)
{
    // Where the provider should send the auth code callback
    var callbackUrl = Url.Page("/ExternalLogin",
        pageHandler: "Callback",
        values: new { returnUrl });

    var props = new AuthenticationProperties
    {
        RedirectUri = callbackUrl
    };

    // Issues the redirect to Google / Azure AD / etc.
    return Challenge(props, provider);
}

Step 2 — Handle the Callback

When Google redirects back, your ExternalLogin Callback page runs. It has two jobs: read the external claims, then build your own IdentityServerUser.

Pages/ExternalLogin.cshtml.cs — OnGetCallbackAsync
public async Task<IActionResult> OnGetCallbackAsync(
    string returnUrl = null)
{
    // 1. Exchange the external auth code → external claims
    var result = await HttpContext.AuthenticateAsync(
        IdentityServerConstants
            .ExternalCookieAuthenticationScheme);

    if (result?.Succeeded != true)
        throw new Exception("External auth failed");

    // 2. Extract claims from the external identity
    var externalUser   = result.Principal;
    var providerUserId = externalUser.FindFirstValue(ClaimTypes.NameIdentifier)
                         ?? externalUser.FindFirstValue("sub");
    var provider       = result.Properties.Items["scheme"];   // "Google"
    var email          = externalUser.FindFirstValue(ClaimTypes.Email);
    var name           = externalUser.FindFirstValue(ClaimTypes.Name);

    // 3. Find or create a local user linked to this external identity
    //    (your own DB logic — match by email or provider+userId pair)
    var localUser = await _users
        .FindByExternalProviderAsync(provider, providerUserId)
        ?? await _users.AutoProvisionUserAsync(provider, providerUserId, email, name);

    // 4. Build the IdentityServerUser for the session
    var isuser = new IdentityServerUser(localUser.SubjectId)
    {
        DisplayName        = localUser.DisplayName,
        IdentityProvider   = provider,           // records "Google" in the session
        AdditionalClaims   = localUser.Claims
    };

    // 5. Sign in and delete the temporary external cookie
    await HttpContext.SignInAsync(isuser);
    await HttpContext.SignOutAsync(
        IdentityServerConstants.ExternalCookieAuthenticationScheme);

    return Redirect(returnUrl ?? "~/");
}

Step 1
AuthenticateAsync on the external scheme reads the temporary cookie Google wrote, decodes the provider's tokens, and surfaces a ClaimsPrincipal.

Step 2
External providers use different claim types. Google uses ClaimTypes.NameIdentifier for the user ID; Azure AD may use "sub". Always check both.

Step 3
Your IS never stores Google passwords — it stores a link: "local user #42 = Google user abc123". Auto-provisioning creates the local record on first login.

Step 5
Deleting the external cookie is important — it prevents re-use of the temporary credential after the session is established.

Claims Mapping — External → Internal

Each provider sends different claim names. You must translate them into the stable, internal claims your tokens use.

Google sends Your internal claim Notes
ClaimTypes.NameIdentifier → sub (via SubjectId) Your local user ID, not Google's
ClaimTypes.Email → email Direct pass-through if verified
ClaimTypes.Name → name Or split into given_name / family_name
(not sent) → tenant_id You add this from your own DB at step 3
Never use Google's user ID as your sub If you change providers later, your user IDs break everywhere. Always generate a stable local ID (a database primary key or GUID) and record the external ID as a separate link table entry.
