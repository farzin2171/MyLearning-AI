Claims Enrichment — IProfileService

~25 minutes Module 3 finale Prerequisite: Lesson 21

After login you have a sub claim — the user's stable ID. Your APIs need more: tenant ID, roles, permissions, subscription tier. These cannot go in the authentication cookie at login time because they may change between sessions, differ per client, or come from multiple data sources. IProfileService is where Duende IS asks you to supply them, on demand, each time a token is issued.

Module 3 capstone concept Every piece of Module 3 leads here. You register clients (L19), handle login (L20), federate external users (L21) — and then enrich the tokens those users receive with the business claims that make authorization decisions possible.

The scope configuration (step 2) acts as a filter — it declares which claim types can appear. Your IProfileService (step 3) supplies the actual values. Both must cooperate.

Which Token Gets Which Claims

ID Token — identity claims
  • Declared via IdentityResource
  • Consumer: the client app (not your API)
  • Typical: sub, name, email
  • Keep it small — clients may display these
Access Token — authorization claims
  • Declared via ApiScope or ApiResource
  • Consumer: your API (validates and enforces)
  • Typical: scope, tenant_id, role
  • Can be larger — the API needs context
Don't put authorization claims in the ID token The ID token is for the client to learn who the user is. The access token is for APIs to decide what the user may do. Mixing them inflates the ID token and leaks authorization logic to the front end.

Wiring IProfileService

Implement the interface and register it. Duende IS calls it every time it needs to populate an ID token or access token.

Services/MyProfileService.cs
public class MyProfileService : IProfileService
{
    private readonly IUserRepository _users;

    public MyProfileService(IUserRepository users)
    {
        _users = users;
    }

    // Called when a token is about to be issued
    public async Task GetProfileDataAsync(
        ProfileDataRequestContext context)
    {
        // Who is the user?
        var sub = context.Subject.GetSubjectId();

        // Load their data from your store
        var user = await _users.GetBySubjectIdAsync(sub);

        // What claim types did the scope config declare?
        // Only include what was requested — don't add everything.
        var claims = new List<Claim>();

        if (context.RequestedClaimTypes.Contains("tenant_id"))
            claims.Add(new Claim("tenant_id", user.TenantId));

        if (context.RequestedClaimTypes.Contains("role"))
            claims.AddRange(
                user.Roles.Select(r => new Claim("role", r)));

        if (context.RequestedClaimTypes.Contains("email"))
            claims.Add(new Claim("email", user.Email));

        // Add them to the token
        context.IssuedClaims.AddRange(claims);
    }

    // Called to check if the user is still allowed to get tokens
    public async Task IsActiveAsync(IsActiveContext context)
    {
        var sub  = context.Subject.GetSubjectId();
        var user = await _users.GetBySubjectIdAsync(sub);

        // Return false to silently block token issuance
        // (e.g. account locked, suspended, deleted)
        context.IsActive = user?.IsActive ?? false;
    }
}

GetProfileDataAsync
Called for both ID tokens and access tokens. The context.RequestedClaimTypes set tells you exactly which claims the client's requested scopes have declared — filter to that set.

IsActiveAsync
Called before every token issuance and before returning tokens from the UserInfo endpoint. Return false to block a locked or deleted account from receiving new tokens.

Only emit requested claims
Adding claims not in RequestedClaimTypes wastes bytes and may expose data the client didn't ask for. Filter strictly.

Registration

Program.cs
builder.Services.AddIdentityServer()
    .AddInMemoryIdentityResources(Config.IdentityResources)
    .AddInMemoryApiScopes(Config.ApiScopes)
    .AddInMemoryClients(Config.Clients)
    .AddProfileService<MyProfileService>()   // ← add this
    .AddDeveloperSigningCredential();

Connecting Scopes to Claims — The Full Picture

Your Config.cs must declare which claim types a scope surfaces. IProfileService then supplies the values. Both sides must agree on the claim name.

Config.cs — declaring claim types on scopes
// Access token claims — declared on ApiScope
new ApiScope(
    name:       "calendar.read",
    displayName: "Read calendar events",
    userClaims:  new[] { "tenant_id", "role" }   // ← these get added to the access token
),

// ID token claims — declared on IdentityResource
new IdentityResource(
    name:       "tenant",
    displayName: "Tenant membership",
    userClaims:  new[] { "tenant_id", "tenant_role" }   // ← these get added to the ID token
)

When a client requests scope=calendar.read, Duende IS sees that scope needs tenant_id and role, sets those in RequestedClaimTypes, and calls your GetProfileDataAsync. Your service loads them from the database and adds them to IssuedClaims.

Context-Aware Claims — Per Client

context.Client tells you which registered client is requesting the token. Use this to add different claims for different applications:

Per-client claims in GetProfileDataAsync
// Only add highly-privileged claims for the internal admin tool
if (context.Client.ClientId == "admin-dashboard"
    && context.RequestedClaimTypes.Contains("permission"))
{
    var permissions = await _users.GetPermissionsAsync(sub);
    claims.AddRange(
        permissions.Select(p => new Claim("permission", p)));
}

Quick Reference

I want to…I do…
Add a custom claim to the access token Add it to ApiScope.userClaims in Config.cs AND emit it in GetProfileDataAsync
Add a custom claim to the ID token Add it to IdentityResource.userClaims in Config.cs AND emit it in GetProfileDataAsync
Block a suspended account from getting tokens Return context.IsActive = false in IsActiveAsync
Add different claims per client Check context.Client.ClientId inside GetProfileDataAsync
Register the service .AddProfileService<MyProfileService>() in AddIdentityServer()
