The Authorization Code Flow + PKCE

~25 minutes Conceptual + interactive Prerequisite: Lesson 12

You know why OAuth 2.0 exists. Now: how does a web app actually obtain a token? The Authorization Code Flow — with PKCE — is the answer for any app that runs in a browser or on a device. It is the most important OAuth flow in practice and the default for every new OAuth client you will build.

Why Not Just Return the Token Directly?

A reasonable first instinct: after the user logs in on the Authorization Server, redirect them back to the app with the access token in the URL. This is the old Implicit Flow, and it was deprecated for good reason.

The implicit flow problem Tokens in redirect URLs leak. Browser history records them. Server logs capture them. The HTTP Referer header can expose them to third-party scripts on the page. A token in a URL is a token at risk.

The Authorization Code Flow solves this by using a two-step process:

  1. Step 1 (front-channel, visible to browser): the Authorization Server redirects back with a short-lived authorization code.
  2. Step 2 (back-channel, server-to-server): the app's server exchanges the code for a token. The token never appears in a URL.

The Flow — Step by Step

Four parties are active. Keep Lesson 12's actor definitions in mind as you read.

1

User clicks "Login" — the Authorization Request

Client → browser redirect → Authorization Server

The client sends the browser to the Authorization Server's /authorize endpoint with these query parameters:

GET /connect/authorize?
  response_type=code
  &client_id=my-web-app
  &redirect_uri=https://app.example.com/callback
  &scope=openid%20profile%20email
  &state=a3fwQRthY7
  &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
  &code_challenge_method=S256
response_type=codeTells the AS: "give me back a code, not a token"
client_idIdentifies which registered client is asking
redirect_uriWhere to send the user after login (must match a pre-registered URI)
scopeThe permissions being requested (space-separated)
stateRandom value to prevent CSRF (explained below)
code_challengePKCE commitment (explained below)
code_challenge_method=S256SHA-256 was used to derive the challenge
2

User authenticates and consents

User → logs in and approves scopes on → Authorization Server

The Authorization Server shows a login page (or delegates to an external IdP). After successful authentication, it shows a consent screen listing the requested scopes. The user approves.

3

AS redirects back with the authorization code

Authorization Server → browser redirect → Client 
GET /callback?
  code=SplxlOBeZQQYbYS6WxSbIA
  &state=a3fwQRthY7

The code is short-lived (typically 60 seconds), single-use, and meaningless without the PKCE verifier. Even if an attacker captures this URL, they cannot use it. The state is echoed back so the client can verify it matches what it sent.

4

Client exchanges the code for tokens (back-channel)

Client server → HTTPS POST → Authorization Server 
POST /connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https://app.example.com/callback
&client_id=my-web-app
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk

This is a direct server-to-server call — the browser is not involved. The Authorization Server verifies the code_verifier against the code_challenge committed in Step 1 and, if valid, responds with tokens: 

{
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type":   "Bearer",
  "expires_in":   3600,
  "refresh_token": "8xLOxBtZp8",
  "id_token":     "eyJhbGciOiJSUzI1NiJ9..."
}
5

Client calls the Resource Server

Client → HTTPS request → Resource Server (API) 
GET /api/user/profile
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...

The access token travels in the Authorization: Bearer … header. The Resource Server validates it (signature, expiry, audience, scopes) and serves the response.

Key insight — why the code? The authorization code exists solely to protect the access token from appearing in a browser URL. It is a short-lived, one-time-use proxy that is worthless without the back-channel POST and PKCE verification. This is the entire reason for Step 3 and Step 4 being separate.

PKCE — Proof Key for Code Exchange

PKCE (RFC 7636, pronounced "pixy") solves the authorization code interception attack. Consider: if a malicious app on the same device intercepts the authorization code before the legitimate app receives it, and the only check is a client_secret, the attacker cannot easily exchange it (they lack the secret). But SPAs and mobile apps cannot keep a client_secret secret — the secret is shipped with the app code.

PKCE replaces the static client_secret check with a per-request dynamic proof:

Step 1 — Client generates a random secret
code_verifier = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"
↓ one-way hash (SHA-256 + BASE64URL)
Step 2 — Client computes a commitment (safe to transmit publicly)
code_challenge = "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM"
↓ sent in the authorization URL (Step 1 of the flow)
Step 3 — Client sends the verifier in the token exchange
code_verifier = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"
↓ Authorization Server verifies: SHA256(verifier) == challenge?
Step 4 — AS confirms the same client started and finished the flow
✓ Match → issue tokens   ✗ Mismatch → reject request

An attacker who intercepts the authorization code in the redirect URL never saw the code_verifier (it was only sent in the back-channel POST). They cannot compute the verifier from the challenge because SHA-256 is one-way. The code is useless to them.

PKCE is now required for all new clients The OAuth 2.1 draft mandates PKCE for all authorization code flows — including confidential clients (those with a client_secret). Duende IdentityServer enforces this by default for new clients.

The state Parameter — CSRF Protection

Before the user starts the login flow, the client generates a random state value and stores it (in a cookie or session). It sends state in the authorization request. When the browser returns with the authorization code, the client checks that state matches what it stored.

Without state, an attacker could trick a user into completing a login that the attacker initiated (CSRF on the redirect endpoint). A matching state proves the redirect originated from this user's own login attempt.
