Token Anatomy & Equisoft Custom Claims

~25 minutes Code-heavy Prerequisites: Lesson 2

Every request in the Equisoft ecosystem is authenticated by a JWT. Knowing exactly what is inside that token — and where each claim comes from — is essential for writing correct authorization code, debugging access issues, and onboarding new clients. This lesson dissects the token anatomy.

Three Token Types: Quick Recap

TokenAudienceWhat it provesUsed for
Access Token API resources (e.g. assistmgntapi) You are authorised to call this API Sent as Authorization: Bearer … to APIs
ID Token The client app itself Who the user is (identity) Used by the frontend; never sent to APIs
Refresh Token Identity Gateway only You may request new access tokens Stored securely server-side; never in browser local storage
Common mistake Do not send the ID Token to an API. APIs validate Access Tokens. The ID Token is for the application to read user information (name, email); it is not an authorization credential.

Anatomy of an Equisoft Access Token

Here is a representative decoded access token issued by the Identity Gateway. Each section is annotated with where the claim comes from.

Header

{
  "alg": "RS256",     // Signing algorithm — RSA with SHA-256
  "kid": "abc123",    // Key ID — maps to the signing key in IdG's JWKS endpoint
  "typ": "at+JWT"     // Token type — "at" marks this as an Access Token (OIDC spec)
}

Payload — Standard Claims

{
  // ── Standard JWT claims (RFC 7519) ────────────────────────────
  "iss": "https://identity.equisoft.com",   // Issuer — the Identity Gateway URL
  "aud": ["assistmgntapi", "authorizationapi"], // Audiences — which APIs accept this token
  "sub": "d4e5f6a7-...",                    // Subject — the user's local ID in IdG's UserDb
  "iat": 1700000000,                         // Issued at (Unix timestamp)
  "exp": 1700003600,                         // Expires at (iat + token lifetime)
  "jti": "unique-token-id",                  // JWT ID — unique per token

  // ── Standard OIDC / OAuth claims ─────────────────────────────
  "client_id": "my-spa-app",                // Which OAuth client requested this token
  "scope": "openid profile assistmgntapi",  // Granted OAuth scopes

Payload — Equisoft Custom Claims

  // ── Added by EquisoftTokenResponseGenerator ──────────────────
  "tenantId": "00000000-aaaa-bbbb-cccc-000000000001", // Tenant GUID — from TenantClient
  "identityRole": "Advisor",               // User's role — from UserClient (can be null)

  // ── From user record (EquisoftProfileService) ─────────────────
  "tenant": "ACME",                        // Tenant key (short name) — from UserProvider
  "externalobjectidentifier": "aad-oid-…", // External IdP user ID — from Azure AD "oid" claim
  "idp": "aad",                            // Which external IdP authenticated the user

  // ── Standard OpenID Connect identity claims ───────────────────
  "name":  "Jane Doe",
  "email": "jane@acme.com",
  "given_name": "Jane",
  "family_name": "Doe"
}
Key insight: two sources of custom claims Claims added by EquisoftTokenResponseGenerator (tenantId, identityRole) require outbound HTTP calls at token issuance time. Claims loaded by EquisoftProfileService come from the Identity Gateway's own database. If a Tenant or User service is down, token issuance degrades gracefully — those claims are omitted rather than failing the whole request.

Custom Claims Reference

ClaimTypeSourceWhat it is
tenantId GUID string EquisoftTokenResponseGenerator → TenantClient The SQL primary key of the user's tenant. Used for data isolation in all services.
tenant Short string EquisoftProfileService → UserProvider table Human-readable tenant key (e.g. ACME). Used in service account secrets config and log correlation.
identityRole String EquisoftTokenResponseGenerator → UserClient The user's platform role (e.g. Advisor, Admin, Guest). The Authorization Service uses this as the starting point for permission resolution.
externalobjectidentifier String EquisoftProfileService — mapped from IdP oid claim The user's ID in the external identity provider (Azure AD object ID). Used to call the User Service and resolve roles.
isGuest "true" / absent Guest authentication handler Present and true for guest users. EquisoftTokenResponseGenerator short-circuits role lookup and sets identityRole = "Guest" for these users.
service_isService "true" / absent Client Credentials grant only Only present on service-account tokens. The DIT framework reads this to set IdentityType.Service. The Authorization Service's IsService policy requires it.
service_role String Client Credentials grant The service's role name. Used by the Authorization Service for role-based permission lookup for service accounts.

How Claims Are Used in Downstream Services

In AssistantManagement

The DIT Identity middleware (app.UseIdentityPrincipal()) parses the JWT and populates IIdentityContext. Controllers access it via dependency injection: 

// AssistantRelationsController.cs
public class AssistantRelationsController : ControllerBase
{
    private readonly IIdentityContext _identityContext;

    public async Task<IActionResult> GetRelations()
    {
        var tenantId    = _identityContext.TenantId;           // from "tenantId" claim
        var userId      = _identityContext.ResourceOwnerId;    // from "sub" claim
        var externalId  = _identityContext.ExternalPrincipalId; // from "externalobjectidentifier"
        var isOnBehalf  = _identityContext.IsOnBehalfOf;       // true if delegated context

        // All queries are filtered by tenantId for data isolation
        var relations = await _service.GetRelationsAsync(tenantId, userId);
        return Ok(relations);
    }
}

In the Authorization Service

The Authorization Service similarly reads IIdentityContext and also uses claims directly for permission resolution:

Validating a Token (as a New API Service)

When you build a new API service, here is the minimal setup to validate Equisoft tokens correctly (following the pattern in AssistantManagement's Startup.cs): 

// Startup.cs (or Program.cs in minimal API)
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme, builder =>
{
    builder.AddJwt();   // DIT helper — wraps Microsoft.AspNetCore.Authentication.JwtBearer
    builder.AddBasicAuth(); // Optional — for health check endpoints
});

// appsettings.json
{
  "auth": {
    "jwt": {
      "authority": "https://api.dev.equisoft.com/identity",  // IdG discovery endpoint
      "audience":  "my-new-service-api",                      // This service's audience name
      "issuer":    ""
    }
  }
}

The JWT middleware automatically fetches the Identity Gateway's public keys from {authority}/.well-known/openid-configuration and caches them. Token signatures are validated against those keys on every request.

You must register your service's audience with the Identity Gateway The audience value in your appsettings must match an API Resource defined in the Identity Gateway's configuration database. Contact your team lead to add the new audience, or run an EF Core migration to seed it. See Lesson 5 for details on client/resource registration.

Interactive Exercise: Token Decoder

Decode a real token from your development environment and find these claims: tenantId, identityRole, tenant, sub, client_id.

  1. Log into a dev application that uses the Identity Gateway.
  2. Open browser DevTools → Network → find a request with Authorization: Bearer … header.
  3. Copy the token value and paste it into jwt.io.
  4. Confirm you can locate all five claims listed above.
  5. Check the exp claim — when does this token expire?
If you do not see tenantId This means either the Tenant service was unavailable during token issuance, or the client app's scope did not request the tenantId claim. Check the client's configured scopes in the Identity Gateway database.
