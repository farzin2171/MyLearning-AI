Duende IS — Claims Pipeline

Quick reference Module 3 — Building Duende IS

The Four-Step Pipeline

1. Login
sub baked into session → 2. Scope config
declares claim types → 3. IProfileService
your code — supplies values → 4. Signed token
claims in JWT payload

Where Each Claim Type Lives

ID Token — identity claims
Declared in: IdentityResource.userClaims
Consumer: client app (never your API)
Standard: sub, name, email, picture
new IdentityResource( "tenant", new[] { "tenant_id" })
Access Token — authorization claims
Declared in: ApiScope.userClaims
Consumer: your protected API
Standard: scope, client_id, sub
new ApiScope( "calendar.read", new[] { "tenant_id", "role" })

IProfileService Interface

MethodWhen calledWhat to do
GetProfileDataAsync Every token issuance; UserInfo endpoint Add claims to context.IssuedClaims — filter by context.RequestedClaimTypes
IsActiveAsync Before every token issuance; UserInfo endpoint Set context.IsActive = false to block a locked / deleted account

Minimal implementation

public async Task GetProfileDataAsync(ProfileDataRequestContext context) { var sub = context.Subject.GetSubjectId(); var user = await _users.GetBySubjectIdAsync(sub); // Only emit what was requested — filter strictly if (context.RequestedClaimTypes.Contains("tenant_id")) context.IssuedClaims.Add(new Claim("tenant_id", user.TenantId)); if (context.RequestedClaimTypes.Contains("role")) context.IssuedClaims.AddRange( user.Roles.Select(r => new Claim("role", r))); } public async Task IsActiveAsync(IsActiveContext context) { var sub = context.Subject.GetSubjectId(); var user = await _users.GetBySubjectIdAsync(sub); context.IsActive = user?.IsActive ?? false; }

Registration

.AddProfileService<MyProfileService>() // inside AddIdentityServer()

Useful Context Properties

PropertyTypeUse for
context.Subject ClaimsPrincipal Read .GetSubjectId() to get the user's sub
context.RequestedClaimTypes IEnumerable<string> Filter: only add claims in this set
context.IssuedClaims List<Claim> Add your claims here
context.Client Client Branch on .ClientId for per-client claim sets
context.Caller string Distinguish: "ClaimsProviderAccessToken" vs "UserInfoEndpoint"

Common Pitfalls

  • Claim type not declared in scope config → never appears in RequestedClaimTypes → silently absent from token.
  • Adding claims without filtering → leaks data to clients that didn't request it.
  • Auth claims in the ID token → inflates client payload; authorization logic bleeds into the front end.
  • Using Google's user ID as sub → breaks if you ever change providers.
  • Forgetting IsActiveAsync → locked accounts can still get tokens from an active session until it expires.
