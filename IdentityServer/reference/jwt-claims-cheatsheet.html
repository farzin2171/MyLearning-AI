JWT Claims — Quick Reference

RFC 7519 · RFC 7517 · RFC 9068 Print-friendly

Structure

BASE64URL(header).BASE64URL(payload).BASE64URL(signature)

Not encrypted — payload is readable by anyone. Signature is tamper-proof but not confidential.

Header Claims

ClaimNameNotes
algAlgorithmRS256 preferred (asymmetric). Avoid none. Be explicit in validation code.
typTypeJWT standard; at+JWT for OAuth access tokens (RFC 9068).
kidKey IDIdentifies which JWKS key to use. Refresh JWKS cache on unknown kid.

Registered Payload Claims RFC 7519 §4.1

ClaimNameNotes
issIssuer validate Authorization Server URL. Must match expected AS exactly.
subSubject Principal the token represents. User ID for interactive logins; client_id for Client Credentials.
audAudience validate Intended recipient(s). Resource Server MUST reject if its identifier is not in aud. Skipping this enables the confused deputy attack.
expExpiration validate Unix timestamp. Reject token at or after this time. Allow ≤60s clock skew.
iatIssued At optional Unix timestamp of issuance. Useful for max-age checks.
nbfNot Before optional Reject token before this Unix timestamp. Validate if present.
jtiJWT ID optional Unique token identifier. Can be checked against a revocation denylist.

Common OAuth / OIDC Claims

ClaimNameNotes
scopeScopeSpace-separated approved OAuth scopes. Check before serving each endpoint.
client_idClient IDThe client that requested this token.
azpAuthorized PartySame as client_id in some AS implementations (e.g. Keycloak).

amrAuth MethodsHow the user authenticated — e.g. ["pwd"], ["mfa"].
acrAuth ContextAuthentication assurance level. Can require MFA with policy.

Signing Algorithms

AlgorithmTypeUse
RS256RSA + SHA-256Preferred. Asymmetric: AS signs with private key; RSes verify with public key from JWKS. Enables key rotation.
RS384 / RS512RSA + SHA-384/512Stronger hash. Use RS256 unless you have a specific requirement.
ES256ECDSA + SHA-256Smaller signatures than RSA. Duende IS supports this.
HS256HMAC + SHA-256Symmetric: shared secret. Avoid for OAuth — requires distributing the secret to all Resource Servers.
none—Never accept. No signature. Always explicitly reject in validation code.

Resource Server Validation Checklist

  • 1Signature — verify using public key from JWKS matching the token's kid
  • 2iss matches the expected Authorization Server URL exactly
  • 3aud contains this Resource Server's identifier
  • 4exp is in the future (with ≤60s clock-skew tolerance)
  • 5nbf is in the past, if the claim is present
  • 6scope includes the scope required by the endpoint being called

JWKS Discovery

GET /.well-known/openid-configuration     → { "jwks_uri": "..." }
GET {jwks_uri}                             → { "keys": [ { "kid": "...", "n": "...", "e": "AQAB" } ] }

Cache JWKS keys. On unknown kid, refresh the cache (AS may have rotated keys). In ASP.NET Core AddJwtBearer() this is handled automatically.
