Lesson 4

The Keep-Alive Flow

Putting Lessons 1–3 together — the flow that uses everything you've learned

After this lesson you can:

  • Explain what keep-alive is for and why it routes differently than notifications
  • Trace a keep-alive call through every layer: controller → cache → hub context → browser
  • Describe what happens to a stale connection when a browser reconnects

What Is Keep-Alive For?

Many enterprise browser applications have idle session timeouts — if the user is inactive for too long, they are logged out. Keep-alive solves a specific problem: a server-side process may know a user is still active (processing a long job, viewing a report) even when the browser looks idle. Keep-alive lets the server send a heartbeat to the browser that says "you're still live — reset your timer."

Unlike notifications, keep-alive signals are targeted at a specific application instance — one browser tab or application. The caller knows which application to keep alive (via its applicationId), not just which user.

The Full Flow — End to End

Here is the complete trace, connecting every layer you have learned.

  1. Browser registers on startup (Lesson 2)

    The browser opens a WebSocket to /hubs/notifications. SignalR assigns a ConnectionId. The browser calls SetApplicationId(appId), which triggers KeepAliveManagementService.LinkAsync.

  2. Connection stored in Redis (Lesson 3)

    LinkAsync calls _notificationCacheManager.SetAsync, which acquires a distributed lock and writes to Redis:

    Key:   appId.ToString()    // e.g. "efa9b2c1-..."
Value: [{ connectionId: "abc123", userId: "user-42", applicationId: "efa9..." }]
TTL:   3600 seconds
  3. Caller POSTs to /api/v1/keepalive

    Some other microservice — or a scheduled job — sends:

    POST /api/v1/keepalive
{
    "userId":        "user-42",
    "applicationId": "efa9b2c1-..."
}

    Both fields are required. The caller knows which user and which application to keep alive.

  4. Controller fetches connection IDs from Redis (Lesson 3)

    KeepAliveController calls GetConnectionIdListAsync(applicationId, userId). This reads the list under key appId.ToString() and filters it by userId, returning only the matching connection IDs.

    // From KeepAliveManagementService.GetConnectionIdListAsync:
return connectionIds?
    .Where(_ => _.UserId.Equals(userId))
    .Select(_ => _.ConnectionId)
    .ToList() ?? Enumerable.Empty<string>();
  5. Hub context pushes to those specific connections (Lesson 2)

    IHubContext sends the event to the exact list of connection IDs:

    await _hubContext.Clients
    .Clients(connectionIds.ToList())
    .SendAsync("keepAliveReceived");
// ↑ No payload — just the event name
  6. Browser fires its keepAliveReceived handler

    The SignalR client fires the registered keepAliveReceived handler. No data arrives — just the event. The front-end resets its session timer (or takes whatever action it defines). The controller returns 204 No Content.

The Single-Connection Guarantee

What happens when a user refreshes their browser tab? The browser reconnects with the same applicationId (the Guid is persistent per application instance, not generated fresh on every load). LinkAsync is called again, and it does this:

// From KeepAliveManagementService.LinkAsync:
var updatedConnections = connections
    .Where(_ => !_.UserId.Equals(userId))  // remove old entry for this user
    .Append(connectionDetails);            // add the new connection

await _notificationCacheManager.SetAsync(key, updatedConnections, ttl);

The old connection record is evicted and replaced with the new one. The cache never accumulates stale entries for refreshed browsers. This is by design — there is exactly one active connection per userId + applicationId pair at any time.

Different tabs get different applicationId values. Each browser tab or application instance generates its own Guid on first load. Tab A has appId-A, Tab B has appId-B. They are tracked independently — a keep-alive for appId-A does not reach Tab B.

How Keep-Alive Differs from Notifications

The two flows share the same connection registration path but diverge completely after that.

Property Notification Keep-Alive
Target parameter userId userId + applicationId
Redis lookup? No Yes
Routing method Clients.User(userId) Clients.Clients(ids)
Reaches All tabs for this user Only this application's tab
Client event notificationReceived keepAliveReceived
Payload { code, subject, body } None
HTTP response 200 OK 204 No Content
