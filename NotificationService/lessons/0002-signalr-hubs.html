Lesson 2

Inside the NotificationHub

The connection lifecycle, hub methods, and the three client-targeting strategies

After this lesson you can:

  • Read and explain every line of NotificationHub.cs
  • Describe the three ways the service targets clients and when it uses each
  • Explain what IHubContext is and why controllers need it instead of Hub

A Correction from Lesson 1

Lesson 1 described the notification delivery as: "the controller looks up the connectionId from Redis." The actual code tells a different story. Here is the entire NotificationsController.Index method:

public async Task<IActionResult> Index([FromBody] NotificationViewModel viewModel)
{
    var sendViewModel = new NotificationSendViewModel
    {
        Body    = viewModel.Body,
        Code    = viewModel.Code,
        Subject = viewModel.Subject
    };

    await _hubContext.Clients.User(viewModel.UserId).SendAsync("notificationReceived", sendViewModel);

    return Ok();
}

No Redis lookup. No cache query. The controller calls Clients.User(userId) and SignalR handles the routing internally. This is the key insight that makes Lesson 2 click: the service uses two completely different routing mechanisms for its two flows.

The Hub Class — What It Gives You

A SignalR hub is a class that inherits from Hub. SignalR creates a new hub instance for each client invocation (not each connection), then discards it. This means hubs are not suitable for holding state — long-lived state lives in injected services.

The Hub base class gives you three things:

  • Context — information about the current connection
  • Clients — ways to push messages to connected clients
  • Groups — methods to add or remove connections from named groups

Everything the NotificationHub does uses exactly these three handles.

The Connection Context

Inside any hub method, Context describes the currently active connection. Two properties matter most in this service:

// A unique string SignalR generates per WebSocket socket.
// Changes every time the browser reconnects.
Context.ConnectionId      // e.g. "abc123xyz..."

// The user ID extracted from the JWT — the NameIdentifier claim.
// SignalR uses this to implement .Clients.User(userId) routing.
Context.UserIdentifier    // e.g. "user-42"

Both are used explicitly in the hub. ConnectionId is passed to KeepAliveManagementService to build the application cache record. UserIdentifier is what powers .User() routing — SignalR maintains an internal map of UserIdentifier → [ConnectionId, ConnectionId, …] using its Redis backplane.

The Connection Lifecycle

Two methods fire automatically — you override them to hook into connection events.

public override async Task OnConnectedAsync()
{
    await Groups.AddToGroupAsync(Context.ConnectionId, "Authenticated SignalR Users");
    await base.OnConnectedAsync();
}

public override async Task OnDisconnectedAsync(Exception? exception)
{
    await Groups.RemoveFromGroupAsync(Context.ConnectionId, "Authenticated SignalR Users");
    await base.OnDisconnectedAsync(exception);
}

This service uses a single group — "Authenticated SignalR Users" — to collect every connected, authenticated socket. The group isn't used for targeted delivery in this service, but it's a natural broadcast target if that need ever arises.

Notice that OnDisconnectedAsync receives an exception parameter. It is null on a clean disconnect (user closes tab) and non-null when the connection dropped unexpectedly (network failure, timeout). The hub doesn't currently differentiate between the two — the group membership cleanup happens either way.

Hub Methods — The Client Calls the Server

Any public method on a hub class becomes callable from the browser. The browser invokes it by name over the WebSocket. Here are the two hub methods this service exposes:

public async Task SetApplicationId(Guid applicationId)
{
    await _keepAliveManagementService.LinkAsync(
        applicationId,
        Context.ConnectionId,
        Context.UserIdentifier);
}

public async Task UnSetApplicationId(Guid applicationId)
{
    await _keepAliveManagementService.UnlinkAsync(
        applicationId,
        Context.ConnectionId,
        Context.UserIdentifier);
}

Both methods are thin wrappers. They extract the values that only the hub can see — ConnectionId and UserIdentifier — and pass them to the service layer. The hub methods themselves contain no business logic.

Note: The browser calls SetApplicationId immediately after connecting. If it fails or is never called, the keep-alive flow breaks — the service has no way to map this connection to an application. Notifications still work (because they use .User() routing, not the app cache), but keep-alive signals will not reach this tab.

The Three Targeting Strategies

This is the heart of the lesson. The service uses three different ways to push messages to clients, each chosen deliberately.

Strategy 1 — Clients.User(userId)  → notifications

// NotificationsController.cs
await _hubContext.Clients.User(viewModel.UserId).SendAsync("notificationReceived", payload);

SignalR internally tracks every ConnectionId associated with a given UserIdentifier. Clients.User() delivers to all of them at once — every tab the user has open, on every service instance, across the Redis backplane. No Redis lookup required in application code.

Strategy 2 — Clients.Clients(ids)  → keep-alive

// KeepAliveController.cs
var connectionIds = await _keepAliveManagementService
    .GetConnectionIdListAsync(viewModel.ApplicationId, viewModel.UserId);

await _hubContext.Clients.Clients(connectionIds.ToList()).SendAsync("keepAliveReceived");

Keep-alive signals must be targeted to a specific application instance — not all of the user's tabs. The controller first fetches the exact ConnectionId list from the application's own Redis cache, then passes them to Clients.Clients(). This is why the application cache exists: to support finer-grained routing than .User() provides.

Strategy 3 — Clients.Group(name)  → available but unused for delivery

// Would broadcast to all authenticated users — not currently used for delivery
await _hubContext.Clients.Group("Authenticated SignalR Users").SendAsync("…", …);

All connected sockets belong to the "Authenticated SignalR Users" group (added in OnConnectedAsync). The group exists but is not currently used to send messages — it's there if a broadcast-to-all-users need arises without a per-user lookup.

→
Notifications use .User(userId) — reach all of the user's tabs, managed by SignalR's backplane.
→
Keep-alive uses .Clients(ids) — reach a specific application instance, after a Redis lookup.

IHubContext — Sending Messages from Outside the Hub

Both controllers use IHubContext<NotificationHub> rather than inheriting from Hub. Why?

A Hub instance only exists during a client invocation — SignalR creates it, calls the method, and destroys it. You cannot hold a Hub as a constructor-injected dependency because it has no meaningful existence outside of that narrow window.

IHubContext<T> is the solution: a singleton service that gives any class access to the Clients surface without needing to be inside a hub. Both controllers inject it:

// NotificationsController
public NotificationsController(IHubContext<NotificationHub> hubContext)
{
    _hubContext = hubContext;
}

// KeepAliveController
public KeepAliveController(
    IHubContext<NotificationHub> hubContext,
    IKeepAliveManagementService keepAliveManagementService)
{
    _hubContext = hubContext;
    _keepAliveManagementService = keepAliveManagementService;
}

The pattern is always the same: inject IHubContext<T>, use .Clients to target, call .SendAsync("methodName", payload).
