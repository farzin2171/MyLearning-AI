Lesson 1

The Two Flows

How a notification travels from an API call to a live browser tab

After this lesson you can:

  • Explain how a browser registers itself to receive notifications
  • Trace a notification from a REST call all the way to a browser event handler
  • Describe what the circuit breaker does when Redis fails

The Big Picture

The Notification Service is a relay. It does not generate notifications — it delivers them. When another microservice wants to tell a user something, it sends a REST request here. This service finds the user's live browser connection and pushes the message in real time.

Two distinct flows make this possible. You must understand both before you can confidently maintain or extend this service.

Flow 1 — Connection Registration

Before any notification can reach a browser, the browser must register itself. This happens when the front-end application loads.

  1. Browser opens a WebSocket

    The browser connects to /hubs/notifications, passing its JWT as a query parameter (?access_token=…). SignalR negotiates the transport — preferring WebSocket — and the connection is held open for the life of the session.

  2. Hub assigns a ConnectionId

    NotificationHub.OnConnectedAsync() fires automatically. SignalR assigns the socket a unique ConnectionId string. The hub adds this connection to the "Authenticated SignalR Users" group.

  3. Browser calls SetApplicationId

    The front-end immediately invokes a hub method, passing an applicationId — a Guid that identifies this specific browser tab or application instance. This step is essential: without it, the connection is open but cannot be associated with an application for keep-alive targeting.

  4. Mapping stored in Redis

    KeepAliveManagementService.LinkAsync() writes the connection record to Redis:

    // Stored under a key derived from userId + applicationId
CachedConnectionIdentifier {
    ConnectionId  = "abc123…",   // SignalR-assigned socket ID
    UserId        = "user-42",
    ApplicationId = Guid("efa9…")
}

    If the same user reconnects with the same applicationId, the old record is replaced. Only one active connection per user per application is ever held — this handles browser refreshes and reconnects cleanly.

Flow 2 — Notification Delivery

Once a connection is registered, any other microservice can push a notification. Here is how a request travels through the system.

  1. Caller POSTs to /api/v1/notifications

    Another service sends a JSON body:

    POST /api/v1/notifications
{
    "userId":  "user-42",          // required
    "code":    "POLICY_APPROVED",  // required — event type identifier
    "subject": "Your policy is approved",  // optional
    "body":    "…"                 // optional
}
  2. Controller looks up the connection

    NotificationsController queries the cache manager by userId. It gets back the CachedConnectionIdentifier, which contains the live ConnectionId — the socket address for that user's current browser session.

  3. Hub context pushes the message

    Using IHubContext<NotificationHub>, the controller calls a method on that specific client:

    await _hubContext.Clients
    .Client(connectionId)
    .SendAsync("notificationReceived", new {
        code    = model.Code,
        subject = model.Subject,
        body    = model.Body
    });

    IHubContext is the key interface that lets ordinary ASP.NET code — controllers, background services — push SignalR messages without being inside a hub class itself.

  4. Browser fires its event handler

    The SignalR client library in the browser fires the registered notificationReceived handler with { code, subject, body }. The front-end reacts — displaying an alert, refreshing a panel, or whatever behaviour the application defines for that code.

What Happens When Redis Goes Down?

Redis serves as both the connection registry and the SignalR backplane. A Redis outage would normally break the service entirely. Two mechanisms working together prevent that.

①
Redis throws an exception — a cache read or write fails.
②
Circuit breaker opens — after 1 failure (configurable via circuitBreaker.exceptionsAllowedBeforeBreaking), the Polly circuit breaker opens. Redis is not called again for 60 seconds. The service logs the break.
③
In-memory cache takes over — all connection lookups fall through to a local IMemoryCache. The service continues to accept and deliver notifications, using whatever connection state it holds locally.
④
Circuit half-opens at 60 seconds — one probe request is sent to Redis. If it succeeds, the circuit closes and normal operation resumes. The service logs the reset.

Important operational caveat: while the circuit is open, connection state lives only in each service instance's local memory. In a multi-replica deployment (e.g. Kubernetes with 3 pods), a browser registered on pod A will not be found by pod B during the outage — and any notification routed to pod B for that user will be silently dropped.
