Lesson 5

Debugging and Operations

The log messages, diagnostic questions, and testing gaps that matter when something breaks

After this lesson you can:

  • Find the four log messages that diagnose most failures in this service
  • Follow a diagnostic tree for "notifications not arriving" and "keep-alive stopped"
  • Explain what the integration tests do and don't verify — and why this matters

The Testing Gap

Before looking at failures, understand what the tests cover. The integration test suite makes HTTP requests and checks status codes:

// NotificationsControllerTests — what's tested:
POST /api/v1/notifications  (unauthorized)         → 401
POST /api/v1/notifications  (missing userId)        → 400
POST /api/v1/notifications  (missing code)          → 400
POST /api/v1/notifications  (valid, authorized)     → 200

// KeepAliveControllerTests — what's tested:
POST /api/v1/keepalive      (unauthorized)          → 401
POST /api/v1/keepalive      (missing userId)        → 400
POST /api/v1/keepalive      (valid, authorized)     → 204

What the tests do NOT verify: that a notification actually reached a browser. A 200 response means the controller ran to completion — it says nothing about whether a user's browser received anything. The same applies to keep-alive: 204 means the controller ran, not that any signal was delivered.

End-to-end delivery requires a live SignalR WebSocket, a running Redis, and a connected browser. None of those exist in the integration test environment. This means delivery failures are invisible to CI. You will only find them from user reports or real-time monitoring.

The Logs You Need

The service emits four structured log messages that cover most failure scenarios. Know these by heart.

  1. "Succeed to set relationship between ApplicationId {x} and ConnectionId {y} for User {z}"

    Emitted by KeepAliveManagementService.LinkAsync. This is your confirmation that the browser called SetApplicationId and the record was written to the cache. If this message never appears for a given user, the browser never registered — the problem is in the front-end, not this service.

  2. "Could not acquire lock"

    Emitted by NotificationServiceCacheManager.SetAsync. This means the distributed lock could not be acquired within 30 seconds — Redis is slow, unavailable, or a previous lock holder crashed without releasing. Critically: this exception counts toward the circuit breaker threshold. One lock failure + one Redis failure = circuit opens.

  3. "Circuit broken"

    Emitted by OnCircuitBreak in the cache manager. Redis is now bypassed. The service falls back to its in-memory cache. Keep-alive delivery degrades to per-instance — connections registered on one pod won't be found by another. This is the most operationally significant log message in the service.

  4. "Circuit reset"

    Emitted by OnCircuitReset. Redis is healthy again and the circuit has closed. Normal cross-instance operation resumes. You should see this 60 seconds after "Circuit broken" if Redis recovered.

Log search note: userId values are logged through .Sanitize(), which escapes control characters, backslashes, and quotes. If a userId contains special characters, its logged form will differ from the raw value. Search by applicationId (a clean Guid) when possible.

Scenario 1 — "Notifications Aren't Arriving"

Notifications use Clients.User(userId) — they don't touch the application Redis cache. This constrains the possible causes.

①
Is the browser connected at all?
Search logs for "Succeed to set relationship" for this user. If absent: the browser never completed SetApplicationId — the front-end never finished its connection setup. The Notification Service is not the problem.
②
Is the JWT valid?
The hub has [Authorize]. If the token is invalid or expired, the WebSocket handshake fails silently from the service's perspective. Check if the browser is receiving 401 responses when trying to connect. Serilog request logging will show this.
③
Does the userId in the POST match the JWT subject?
Clients.User(userId) routes by Context.UserIdentifier — the JWT NameIdentifier claim. If the caller sends a different userId string (e.g. an email instead of an ID, or wrong casing), the message is sent to a user that has no connections. It disappears silently.
④
Is the SignalR Redis backplane healthy?
The backplane is a separate Redis connection from the application cache. If it fails, Clients.User() routing only works for connections on the same service instance. Notifications to users on other instances are lost. This is separate from the circuit breaker — there is no automatic fallback for the backplane.

Scenario 2 — "Keep-Alive Stopped Working"

Keep-alive is more fragile than notifications — it has more moving parts. Work through these in order.

①
Was the connection ever registered?
Search logs for "Succeed to set relationship" with the userId + applicationId. If absent: SetApplicationId was never called or failed silently. No registration = no keep-alive, ever.
②
Has the TTL expired?
The default is 3600 seconds (1 hour). If the user was idle for more than an hour, the Redis record expired and was evicted. The browser must call SetApplicationId again on reconnect. Depending on the front-end implementation, this may not happen automatically.
③
Is the circuit breaker open?
Search for "Circuit broken". If present, the connection may only exist in one instance's memory cache. A keep-alive request routed to a different instance won't find it. Check if "Circuit reset" appeared afterward — if not, Redis may still be unhealthy.
④
Is the applicationId in the request correct?
The keep-alive caller must send the exact same Guid the browser used in SetApplicationId. A mismatch returns 204 (success) but looks up an empty list — the signal is silently dropped. There is no error to look for; you have to verify the Guids match.

Two Things the Health Check Cannot Tell You

The service exposes a /health endpoint that returns 200 when the service is running. This is useful for liveness checks — but it cannot tell you:

  • Whether Redis is reachable. The health check returning 200 is not evidence that Redis is healthy. The circuit could be open and the service running entirely on in-memory cache while /health returns green.
  • Whether notifications are being delivered. A 200 from /health means the process is alive — nothing more. A healthy process with a broken Redis backplane silently drops cross-instance notifications.

For true end-to-end health monitoring, you need a synthetic test: a client that connects via WebSocket, a probe that sends a notification or keep-alive, and a check that the client received the event. This doesn't exist yet in the test suite.

Check Your Understanding

Questions draw from all lessons — retrieval practice from earlier material as well as new content.

1. A valid POST /api/v1/notifications returns 200. What does this confirm?

The browser received notificationReceived
The SignalR Redis backplane is healthy
The controller ran to completion without error
The userId exists in the active connection store

2. You search logs and find no "Succeed to set relationship" for a user. Where is the problem?

The browser — SetApplicationId was never called
Redis — the write was rejected by the lock
The circuit breaker — Redis is currently open
The controller — the request was rejected with 400

3. The circuit breaker opens. Notifications and keep-alive are both in flight. Which is more likely to fail?

Notifications — .User() routing depends on the app cache
Keep-alive — it depends on the app cache for connection IDs
Both equally — both flows require the app cache
Neither — the in-memory fallback handles both flows

4. A keep-alive call returns 204 but the browser receives nothing. Which cause is hardest to detect from logs alone?

The circuit breaker is open — logs show "Circuit broken"
The connection TTL expired — logs show no recent "Succeed to set"
The connection is on another instance — logs show "Using in memory cache"
The applicationId is wrong — returns 204 with no error logged
