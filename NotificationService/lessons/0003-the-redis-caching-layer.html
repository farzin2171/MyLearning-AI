Lesson 3

The Application Cache

What is stored in Redis, how writes are protected, and what happens when Redis fails

After this lesson you can:

  • Describe the exact key/value structure stored in Redis for keep-alive routing
  • Explain why write operations need a distributed lock
  • Trace the code path for a cache read when Redis is unavailable

The Narrative Thread

When a browser calls SetApplicationId(appId) on the hub, execution flows to KeepAliveManagementService.LinkAsync, which calls _notificationCacheManager.SetAsync. This lesson traces exactly what happens inside that cache manager — and what the fallback looks like when Redis is down.

This is the layer that makes keep-alive routing possible. Notifications (Clients.User()) never touch it. Only the keep-alive flow does.

What Is Stored and Why

The cache manager stores one record per applicationId. The value is a list of connection identifiers — because multiple users can share the same applicationId (e.g. a shared workstation, or a scenario where the application concept spans users).

// Key: applicationId as a plain string (Guid.ToString())
Key:   "efa9b2c1-0c14-4a3b-b9f1-ab27e31b4f2c"

// Value: a JSON array
Value: [
  {
    "connectionId":  "abc123...",
    "userId":        "user-42",
    "applicationId": "efa9b2c1-..."
  }
]

// TTL: 3600 seconds (1 hour) — set in appsettings.json
//      persistence:cacheOptions:expiryTimeSpanInSecs

Every time LinkAsync is called for a userId, it:

  1. Reads the current list from cache
  2. Removes any existing entry for that userId (replacing old connections)
  3. Appends the new CachedConnectionIdentifier
  4. Writes the updated list back

This read-modify-write pattern is exactly why writes need a distributed lock — which we'll come to in a moment.

The Interface

IServiceCacheManager is deliberately minimal — just two methods:

Task<T> GetAsync<T>(string key)
Task    SetAsync<T>(string key, T value, TimeSpan timeToLive)

The interface hides whether the underlying store is Redis, memory, or something else. KeepAliveManagementService only knows about this interface — the hybrid strategy is entirely inside the implementation.

Reading: Two Chances to Succeed

Here is the full GetAsync implementation:

public async Task<T> GetAsync<T>(string key) where T : class
{
    // Fast path: circuit already open — skip Redis entirely
    if (_circuitBreakerPolicy.CircuitState == CircuitState.Open)
    {
        return await FallbackToInMemoryCache<T>(key);
    }

    try
    {
        // Try Redis, wrapped in the circuit breaker
        var cacheValue = await _circuitBreakerPolicy.ExecuteAsync(
            async () => await _redisCache.GetStringAsync(key));

        if (cacheValue != null)
            return Deserialize<T>(cacheValue);

        return default;
    }
    catch (Exception)
    {
        // Redis threw something — fall back
        return await FallbackToInMemoryCache<T>(key);
    }
}

There are two routes to the in-memory fallback, not one. The explicit CircuitState.Open check is an early exit: when the circuit is already open, the code never calls ExecuteAsync at all. The catch block handles failures that happen while the circuit is transitioning or closing.

The in-memory read is also guarded:

private async Task<T> FallbackToInMemoryCache<T>(string key) where T : class
{
    if (_cacheOptions.IsInMemoryCacheEnabled)
        return await Task.FromResult(_memoryCache.Get<T>(key));

    return null;
}

If enableMemoryCacheProvider is false in configuration, the fallback returns null — keep-alive lookups return no connections, and the keep-alive signal goes nowhere. The service degrades gracefully rather than throwing.

Writing: Why a Distributed Lock?

Consider what happens without a lock in a two-instance deployment during LinkAsync:

①
Instance A reads the list: [user-1]
②
Instance B reads the same list: [user-1]
③
Instance A writes [user-1, user-42] — adds user-42
④
Instance B writes [user-1, user-99] — adds user-99, overwrites user-42's entry

Instance B's write clobbers Instance A's work. User-42's connection is lost from the cache — their keep-alive signals stop arriving.

The lock prevents this. SetAsync acquires a distributed lock using RedLock.net — a Redis-based locking algorithm. Only one instance can hold the lock at a time:

var lockKey = key + "_lock";  // e.g. "efa9b2c1-..._lock"

using IRedLock cacheLock = await _redisCache.AcquireLockAsync(
    lockKey,
    expiryTimeSpan:  TimeSpan.FromSeconds(30),   // max lock lifetime
    waitTimeSpan:    TimeSpan.FromSeconds(30),   // how long to wait to acquire
    retryTimeSpan:   TimeSpan.FromSeconds(5));   // retry interval

if (!cacheLock.IsAcquired)
    throw new Exception("Could not acquire lock");
    // ↑ This throw counts as a circuit breaker hit —
    //   a lock failure can trigger the fallback path

If the lock cannot be acquired within 30 seconds, the write throws. That exception propagates up to the catch in SetAsync, which falls back to writing the value to the in-memory cache.

Note: The lock is not held during the read phase of LinkAsync — only during the write. This is an optimistic pattern: read without a lock, write with one. It means a rare race is still possible (two instances reading before either writes), but the lock ensures their writes don't interleave.

The Circuit Breaker: Three States

The Polly circuit breaker has three states. It is built once in the constructor and shared across all calls:

_circuitBreakerPolicy = Policy
    .Handle<Exception>()
    .CircuitBreakerAsync(
        exceptionsAllowedBeforeBreaking: 1,   // 1 failure opens the circuit
        durationOfBreak: TimeSpan.FromSeconds(60),
        onBreak:  OnCircuitBreak,   // logs "Circuit broken"
        onReset:  OnCircuitReset);  // logs "Circuit reset"
State Trigger Behaviour
Closed Normal / after successful probe Redis is tried. Exceptions increment the counter.
Open 1 exception in Closed state Redis is bypassed. Explicit check returns fallback immediately. Duration: 60 s.
HalfOpen 60 s after opening One probe request let through. Success → Closed. Failure → back to Open.

After 1 Redis exception the circuit opens immediately (because exceptionsAllowedBeforeBreaking = 1). This is aggressive. A single transient network blip closes the circuit for 60 seconds. The trade-off is that instability is contained fast — but the service relies on its local memory cache for an entire minute during transient Redis hiccups.
