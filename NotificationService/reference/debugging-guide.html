Diagnostic Tree: Notifications Not Arriving

User reports: "I'm not getting notifications"

1. Is the browser connected?
   → Search logs for "Succeed to set relationship" with this userId
   → If missing: browser never called SetApplicationId — front-end issue
   → If present: connection was registered; continue

2. Is the JWT valid?
   → If browser never got a WebSocket connection, check 401 responses
   → Token expired mid-session? SignalR will drop on next reconnect

3. Is the SignalR Redis backplane healthy?
   → Notifications use .User() routing via the backplane
   → If backplane Redis is down: cross-instance routing fails
   → Notifications TO THE SAME INSTANCE still work

4. Is the correct userId being sent by the caller?
   → Compare userId in POST /api/v1/notifications with Context.UserIdentifier
   → These must match exactly (case-sensitive string comparison)

Diagnostic Tree: Keep-Alive Not Arriving

User/system reports: "Keep-alive signals stopped"

1. Was the connection ever registered?
   → Search logs for "Succeed to set relationship" with this userId + applicationId
   → If missing: SetApplicationId was never called — front-end issue

2. Has the circuit breaker opened?
   → Search logs for "Circuit broken"
   → If yes: Redis is/was down; fallback is in-memory (per-instance)

3. Is the connection on the right instance?
   → If circuit opened, the connection may be in instance A's memory
   → Keep-alive request to instance B won't find it

4. Has the TTL expired?
   → Default TTL: 3600 seconds (1 hour)
   → If user was idle for > 1 hour, the record was evicted
   → Browser must call SetApplicationId again (on reconnect or tab refocus)

5. Is the applicationId in the keep-alive request correct?
   → Must match the Guid used in SetApplicationId exactly
   → A Guid mismatch returns 204 but no connection IDs found → silent drop

The Testing Gap

What the tests verifyWhat they do NOT verify
HTTP 200/204 for valid requests That the notification reached the browser
HTTP 400 for missing required fields That the keep-alive reset the session timer
HTTP 401 for unauthorized requests Circuit breaker behaviour under Redis failure
Health check returns 200 Redis connectivity or SignalR backplane health

End-to-end delivery requires a live SignalR connection and Redis. Manual testing or load testing with a real client is needed to verify the full flow.
